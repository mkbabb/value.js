# A.W6 — Blob/aurora idiomatic abstraction (conditional)

**Opens after**: W5 close + glass-ui shipping the metaballs/aurora API extensions (`coordination/Q.md §3`).
**Agents**: 2 implementation lanes (A blob abstraction, B aurora abstraction + config-pane adoption).
**Hard gate**: `useMetaballRenderer.ts` and the demo metaball shader deleted (deletion proof) and the demo blob/atmosphere consume `@mkbabb/glass-ui`; or, if glass-ui declines the API, the wave re-scopes and the named successor is recorded.
**Status**: planned. Conditional — see §"Conditionality".

## Conditionality

`audit/HARDEN-4` found the former combined W5 was un-closeable: it held both the close ceremony and an open cross-repo dependency, and the precept `SPEC §Close` forbids a close ceremony over an open dependency. The augmented schedule splits it — W6 carries the dependency, W7 is the clean close. W6 is not a brittleness window (`A.md §9`): it breaks no tree and suspends no gate. It is a conditional implementation wave.

```yaml
breaking_changes_during_wave: no
suspended_gates: none
cross_repo_dependency: glass-ui metaballs positionSource hook + pointer input
                       + per-blob opacity + color perturbation;
                       Aurora deriveAuroraPalette
fallback: if glass-ui ships the API, W6 proceeds as below;
          if glass-ui declines or defers, W6 re-scopes — the demo-side
          A-key-3 guard already landed at W0, the duplication is documented
          in audit/W6-deferred.md, and the abstraction routes to a named
          successor: a glass-ui successor tranche + value.js tranche B.
```

## Scope

Source: `research/Ae` Part 2, `audit/HARDEN-4`.

### Lane A — blob abstraction (Ae-10)

1. Consume glass-ui's metaballs `positionSource` per-frame hook — `useMetaballs(canvasRef, config, { positionSource })`. The demo keeps `useBlobMood` and `useBlobSatellites` as thin composables that feed blob frames into the hook; these are app-specific affect with no library equivalent.
2. Delete `useMetaballRenderer.ts` and the demo metaball shader. `audit/HARDEN-4` corrected the deletable count to ~200 lines (not 319) — the mood-coupled per-frame color perturbation, the `smin` body silhouette, and the `POS_SCALE` overflow-canvas contract are deeper-coupled than `research/Ae` Ae-10 stated and survive as the demo's `positionSource` callback, not as deleted code. The lane proof doc records exactly what is deleted and what survives.
3. Route GooBlob's pointer affect through glass-ui's pointer input; route the HSV perturbation through glass-ui's `colorPerturbation` config.

### Lane B — aurora abstraction + config-pane adoption (Ae-9, Ae-11, Ae-14)

1. Replace W0's static `AuroraConfig` with a picker-derived palette — call glass-ui's `deriveAuroraPalette(cssColorOpaque, opts)` so the atmosphere tracks the picker color, the demo's original intent.
2. Rewrite the `AuroraPane`/`ConfigSliderPane` slider table against the live `AuroraConfig` fields — `warpAmount`, `softmaxBeta`, `nucleiDrift`, `breathDepth`, `saturation`. This completes the W0 "pane under rework" deferral.
3. Migrate the demo-local `ConfigSliderPane` (built in W4 Lane D) onto glass-ui's existing `./configurator` surface — `ConfiguratorRow` + `useConfiguratorState`. `audit/HARDEN-4` confirmed glass-ui already ships this; the gap was non-adoption. `BlobPane` and `AuroraPane` both consume it.
4. If glass-ui ships `BlobDot`, migrate the 11 `WatercolorDot` consumers onto it and delete the demo-local `WatercolorDot`. If not, `WatercolorDot` stays demo-local with a recorded glass-ui gap.
