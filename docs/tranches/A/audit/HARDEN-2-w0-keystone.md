**Lane**: hardening-2, tranche A.
**Method**: re-read of current source at value.js HEAD `70e61e9` (`w.w2.1-value-js-prebuild`), glass-ui HEAD `95098d1`, keyframes.js HEAD `8d824ee`. The research doc `Aa-runtime-keystone.md` was not trusted; every cited file:line was re-opened.

`Aa §2` claims `import` and `types` "point at deleted files." Half-true: `import: ./dist/keyframes.js` and `types: ./dist/keyframes.d.ts` both point at files absent from the current `dist/`. ### A-key-2 — Aurora integration never migrated — CONFIRMED, cited lines correct, cascade WRONG

`App.vue:321-334` builds `reactive<AuroraConfig>({ colorMode, colors, surfaceMode, surfaceAlpha, blur, speed, blobCount, ... })`. None of those 22 keys exist on the current `AuroraConfig` (`glass-ui/src/components/custom/aurora/presets.ts:68-106`). The current shape requires `palette: OklchStop[]`, `nuclei: AuroraNucleus[]`, `softmaxBeta`, `valueVariance`, `warpAmount`, `medium`, `flow`, and ~25 more. The demo object supplies none of the required keys.

The three sub-breaks in `Aa §3` are each correct against current source:

- **Schema** — `useAurora` → `createAurora` → `setConfig(config)` (called synchronously at `runtime.ts:314`) → `flattenPalette(cfg.palette, MAX_STOPS, ub.palette)` at `runtime.ts:225`. `flattenPalette` reads `Math.min(stops.length, maxStops)` at `color.ts:96`. `cfg.palette` is `undefined`, so `undefined.length` throws `TypeError`. Confirmed.
- **Arity** — `useAurora(canvasRef, configSource, runtimeOptions)` (`useAurora.ts:38-42`). Third arg is `AuroraRuntimeOptions`. The demo passes `cssColorOpaque`, a color ref. Confirmed.
- **Return shape** — `useAurora` returns `UseAuroraReturn` (`useAurora.ts:19-26`): `setCursor`, `clearCursor`, `setCursorRadius`, `renderAt`, `pause`, `resume`. No `config` member. `const { config: auroraConfigResult }` binds `undefined`; `provide("auroraConfig", undefined)` poisons `AuroraPane`. Confirmed.

The cascade mechanism is misdescribed. `Aa §3` and `A.md §1` both state the throw "aborts the rest of the App `mounted` hook" so "the dock, panes, and dropdown wiring never finish." This is wrong on two counts. First, App.vue's `onMounted` (line 341) contains only `loadCustomColorNames()` — there is no dock/pane wiring in it; dock and panes are template-declared and mount through their own lifecycles. Second, `useAurora` does not run in App.vue's `onMounted`. It runs in `useAurora`'s own `onMounted` (`useAurora.ts:52`), a separate registered callback. A throw from one component's `onMounted` callback does not abort another's. The real failure surface is narrower and more severe than the doc claims: Vue's mounted-hook error handling lets the throw propagate, and an uncaught error thrown synchronously during the mount flush will abort the render of the component subtree that owns the `useAurora` call — which is `App.vue` itself, since `useAurora` is called in `App.vue`'s `setup`. The visible result (partial render, bare dock) is consistent with App.vue's own mount being torn down, not with a hook tail being amputated. W0's fix still works regardless; the correction matters because the hard gate (§6) must probe for an App-level crash, not a "hook tail" symptom.

`AuroraPane.vue:13` does `inject<AuroraConfig>("auroraConfig")!` and its `VALUE_JS_DEFAULTS` (lines 15-36) plus all five `SliderDef` tables target the deleted fields. Confirmed dead. The non-null `!` over an injected `undefined` is a second crash site independent of App.vue: any code path that mounts `AuroraPane` and reads `cfg.colorMode` dereferences `undefined`.

### A-key-3 — GooBlob null-canvas render — CONFIRMED, cited line WRONG

`useMetaballRenderer.ts:131` is `gl.blendFunc(gl.ONE, gl.ONE_MINUS_SRC_ALPHA)` inside `initGL` — not a `.width` read, and not a crash. The actual null-canvas hazard is at **lines 178-179**: `const canvas = canvasRef.value!;` followed by `gl.uniform2f(uniforms.uResolution, canvas.width, canvas.height)`. The `!` non-null assertion is the fault. The `render` loop reaches line 178 only past the `if (!gl || !program)` guard at line 167, and `gl` is set only by `initGL`, which `start()` calls with a non-null canvas. The window where `canvasRef.value` is `null` while `gl` is still truthy is a post-mount unmount: a `v-if` removes the canvas, `onUnmounted`/`destroy` is racing the in-flight RAF, and the queued `render` frame dereferences `canvasRef.value!` after the ref cleared. `destroy()` sets `destroyed = true` and cancels `rafId`, and `render` checks `destroyed` at line 152 — so a clean unmount is guarded. The unguarded window is narrow but real: an RAF callback already dequeued and executing when `destroy()` runs is not stopped by `cancelAnimationFrame`. W0 must guard line 178 (`const canvas = canvasRef.value; if (!canvas) return;`), and the spec should correct the line citation from 131 to 178.

## §2 — Missed faults

### MISS-1 — `vue-tsc` is not installed; the W0 gate cannot run

W0's hard gate (`W0.md` items 4 and 5, `A.md §6`) requires `vue-tsc --noEmit` to exit clean and a `typecheck` script wired into `package.json`. value.js's `package.json` has no `vue-tsc` in `devDependencies` and `node_modules/.bin/vue-tsc` does not exist. Only plain `tsc` is present. The demo is `.vue`-heavy; plain `tsc` cannot typecheck SFC `<script setup>` blocks. As written, W0's gate names a tool that does not exist in the tree. W0 Lane C must add `vue-tsc` (glass-ui pins `^2.2.0`) to `devDependencies` before the `typecheck` script can be wired. This is not a deferral — it is a precondition the spec omitted. Without it, the gate "closes green" only because the command silently is unavailable, which is a false close (§6).

A second hazard: the demo `tsconfig.json` sets `exactOptionalPropertyTypes: true` and `noUncheckedIndexedAccess: true`, and `include` is `["src/", "demo/"]`. A first-ever `vue-tsc --noEmit` run over `demo/` will surface pre-existing type errors unrelated to A-key-2 — the demo has never been typechecked (`Aa §7`: "the demo has no typecheck gate"). W0's gate item 4 ("`vue-tsc` exits clean") is not achievable in one wave if the demo carries latent type debt. W0 must either scope the typecheck to the files it touches (a `tsconfig` project that includes only the changed surface) or budget a triage pass. The spec currently assumes a clean baseline that has never been demonstrated.

### MISS-2 — `dist/` is already partially clobbered

`A.md §6` and `Aa §6` correctly identify Q-break-3: `gh-pages` writes `outDir: ./dist/`, the same directory `npm run build` writes the library to. The current `dist/` shows the clobber has already happened — alongside `value.js` and the `.d.ts` tree it carries `postcss-BrHISTov.js` and `standalone-JqHlnZSs.js`, which are demo-build chunk artifacts, not library output. `dist/value.js` itself is still the library build (its first line imports `@mkbabb/parse-that`), so the state is a partial overlay. W0's fix (route `gh-pages` to `dist/gh-pages/`) is correct, but the spec should add a step: `dist/` must be cleared and a clean `npm run build` run, otherwise the W0 gate's "`dist/value.js` + `dist/index.d.ts` present" check (item 3) passes against a polluted directory and the stray chunks ship.

### MISS-3 — the font 403 is real, in scope, and value.js-owned

`demo/@/styles/style.css:3` does `@import "@mkbabb/glass-ui/styles"`, which resolves to glass-ui's `src/styles/index.css`. glass-ui's `typography.css:137` declares `@font-face { ... src: url("../fonts/fira-code/fira-code-latin.woff2") ... }`. That file exists (`glass-ui/src/fonts/fira-code/fira-code-latin.woff2`, 36 KB). A 403 Forbidden — distinct from a 404 — is a Vite dev-server filesystem-access denial, not a missing asset. Vite's `server.fs.allow` defaults to the project workspace root; a `woff2` reached through a symlinked dependency in `node_modules/@mkbabb/glass-ui` (symlink target `../../../glass-ui`) lies outside value.js's root, and Vite's `server.fs` strictness returns 403 for files outside the allow-list. value.js's `vite.config.ts` declares no `server.fs` block. The font 403 is a value.js consumer-resolution fault in exactly the family W0 owns (invariant A2, consumer-resolution integrity), and it is adjacent to the `resolve.conditions` design because both concern how Vite treats symlinked siblings. W0 should add `server.fs.allow` covering the sibling package roots (`../glass-ui`, `../keyframes.js`) or set `server.fs.strict: false` for the demo modes. This belongs in W0, not W1: it is a resolution fault, the demo's primary mono font is unrenderable without it, and a Playwright boot probe that ignores a 403 in the network log is a weaker gate (§6).

### MISS-4 — no stale glass-ui named imports; no other stale reactive schema

For completeness against the task's hunt list: the demo imports ten `@mkbabb/glass-ui` subpaths (`/aurora`, `/dock`, `/tabs`, `/search`, `/glass-carousel`, `/confirm-dialog`, `/controls`, `/dark`, `/forms`, root). No additional stale-schema fault. This negative result should be recorded in W0 so a later wave does not re-hunt it.

## §3 — `resolve.conditions` design review

The W0 design (`W0.md` Lane A item 2) is to add `resolve.conditions` declaring `development` ahead of `import`, "so sibling `@mkbabb/*` packages resolve to `src/` deterministically under the dev server rather than by Vite serve-mode auto-injection."

Both siblings are symlinked into `node_modules/@mkbabb/` (`glass-ui → ../../../glass-ui`, `keyframes.js → ../../../keyframes.js`). Both `package.json` `exports` carry a `development` branch first: glass-ui `"." → development: ./src/index.ts`, keyframes.js `"." → development: ./src/animation/index.ts`, glass-ui `./aurora → development: ./src/aurora.ts`, and so on for every subpath. value.js's own `package.json` carries the same (added in `70e61e9`). The conditional-exports substrate is in place; the design's mechanism is sound for the dev server.

Two traps the W0 spec does not address:

**Trap 1 — `development` is already auto-applied in dev, and harmful in build.** value.js HEAD commit `70e61e9` itself states the `development` condition is "Vite's auto-activated condition in dev mode." Vite injects `development` into `resolve.conditions` for `vite serve` automatically; for `vite build` it injects `production` instead and does NOT inject `development`. So under the default dev server, glass-ui and keyframes.js already resolve to `src/` without any config change — the alias is what blocks it, not a missing condition. Declaring `resolve.conditions: ["development", ...]` in `defaultOptions` (which all four modes spread) would force `development` into the `production` and `gh-pages` builds too. For value.js's own library build that is wrong: `npm run build` must resolve consumers' view of value.js, and forcing `development` would make a value.js build resolve glass-ui/keyframes.js to their `src/` TypeScript rather than their built `dist/`. The `gh-pages` build is more forgiving (it bundles everything), but `hero-lab`'s `optimizeDeps.exclude: ["@mkbabb/keyframes.js"]` plus a forced `development` condition is an untested combination. The correct design is mode-specific: add `resolve.conditions: ["development", ...defaults]` only to the dev-mode and `gh-pages`/`hero-lab` demo-build return objects, never to the `production` library-build object, or omit it from `production` explicitly. W0's spec says "add `resolve.conditions` to `vite.config.ts`" without qualifying which of the four mode branches receives it. That ambiguity is a correctness bug.

**Trap 2 — `defaultOptions` spread ordering.** `vite.config.ts` builds `defaultOptions` with a `resolve.alias` block and spreads `...defaultOptions` into every mode. If W0 adds `resolve.conditions` to `defaultOptions.resolve` alongside `alias`, every mode inherits it — re-introducing Trap 1. If W0 instead adds `conditions` per-mode after the spread, the per-mode `resolve` object must merge rather than replace: writing `resolve: { conditions: [...] }` in a mode branch after `...defaultOptions` would shadow the entire `resolve` block and drop the `@src`/`@styles`/`@components` aliases the demo depends on. The W0 implementer must spread `...defaultOptions.resolve` into the per-mode `resolve`. The spec does not flag this.

The deletion of the `keyframes.js` alias (Lane A item 1) is correct and unambiguous. The `resolve.conditions` addition needs the mode-scoping and merge-ordering corrections above before it is safe.

## §4 — Aurora W0 fix scope review

`DEFAULT_AURORA_CONFIG` (`presets.ts:139-175`) is a fully-populated, valid `AuroraConfig`: 3-stop `palette`, 2-nucleus `nuclei`, every scalar present. A static config seeded from it boots: `createAurora` → `setConfig` → `flattenPalette(palette)` reads `palette.length` (3) without throwing, and every uniform upload finds its key. Confirmed — W0 Lane B's "static valid `AuroraConfig` seeded from `DEFAULT_AURORA_CONFIG`" is sufficient to clear the mount crash.

One correction to W0 Lane B item 1's phrasing: it says "A static valid `palette`/`nuclei` is sufficient." A static `palette` and `nuclei` alone is *not* sufficient — `AuroraConfig` has no optional fields except `AuroraNucleus.elongation`/`angle`, so a partial object fails `verbatimModuleSyntax`-strict typecheck and, at runtime, leaves `softmaxBeta`/`warpMode`/`medium`/`flow` undefined, which `setConfig` uploads as `NaN`/`undefined` uniforms. The fix must spread the whole `DEFAULT_AURORA_CONFIG`, then override fields if desired — `reactive<AuroraConfig>({ ...DEFAULT_AURORA_CONFIG })`. W0 should state "seeded from a full `DEFAULT_AURORA_CONFIG` spread," not "a static valid `palette`/`nuclei`."

The arity and return-shape corrections in W0 Lane B items 2-3 are accurate against `useAurora.ts`. Item 2 ("pass `{}` or `{ onInitError }` as the third argument") is correct — `AuroraRuntimeOptions` defaults to `{}`. Item 3 ("delete the `{ config }` destructure; provide the real reactive config object") is correct: the demo should keep its own `auroraConfig` reactive object and `provide("auroraConfig", auroraConfig)` directly, since `useAurora` neither owns nor returns the config. One addition the spec omits: `App.vue:334` currently provides `auroraConfigResult` (the bound-`undefined` destructure) under key `"auroraConfig"`. After the fix the provided value must be the real `auroraConfig` reactive object, and `AuroraPane.vue:13`'s `inject<AuroraConfig>("auroraConfig")!` then receives a live object — but `AuroraPane`'s entire slider table still targets dead fields. W0 Lane B item 4 already covers this ("rewrite the slider table, or render the pane disabled"). The disabled-pane path is the lower-risk W0 choice; a full rewrite against the 30-field `AuroraConfig` is W5 scope and should not be attempted under a boot-unbreak wave.

## §6 — Hard-gate validity

W0's hard gate is not airtight. It can close green with faults remaining:

1. **The `vue-tsc` gap (MISS-1) is a silent-pass hole.** Gate items 4 ("`vue-tsc --noEmit` exits clean") and the wired `typecheck` script both name a binary not installed. If the implementer runs `npm run typecheck` and the script body is `vue-tsc --noEmit`, npm exits non-zero with "command not found" — loud, not silent. But if the implementer wires the script and the gate operator reads only "typecheck script present in package.json," the gate's intent (the schema desync fails loud) is unmet while the checkbox is ticked. The gate must require an *observed clean run* of `vue-tsc` over the changed surface, with the run output captured to `audit/W0-build-typecheck.md`, and must require `vue-tsc` itself to be a committed `devDependency`.

2. **The Playwright probe does not assert on network failures.** Gate item 2 checks "zero uncaught console errors." A 403 on `fira-code-latin.woff2` (MISS-3) is a failed network request, not necessarily an uncaught console error — a `@font-face` fetch failure logs a warning at most and falls back to the next font in the stack. The probe would pass while the demo's mono font silently fails. The hardened gate must assert zero non-2xx responses for same-origin asset requests, captured from the Playwright network log, not only console errors.

3. **The `dist/` check passes against a polluted directory (MISS-2).** Gate item 3 checks "`dist/value.js` + `dist/index.d.ts` present" and "`dist/gh-pages/` carries the demo build." Both can be true while `dist/` still carries the stray `postcss-*.js`/`standalone-*.js` demo chunks from the prior clobber. The gate must require `dist/` to be cleared before the clean `npm run build`, and must assert `dist/` contains *only* library artifacts (no `index.html`, no demo chunks).

4. **The cascade misdescription weakens the probe target.** Because `Aa §3` frames the failure as an "amputated hook tail," a probe author might look for a half-rendered dock and call it pass-adjacent. The real signal is an App-level mount crash. The gate's probe must assert the full App tree renders — dock, a pane, an opened color-space `Select` — and that the page is not in Vue's error-recovery state. W0.md item 2 already lists those surfaces; the hardening is to state explicitly that *any* of the three faults leaves a detectable App-subtree teardown, and the probe fails on a teardown, not only on a console string.

5. **No deletion proof for the alias.** The gate proves `npm run dev` boots, but a boot can succeed with a warm Vite dep cache even if the alias lingers (`Aa §2`: the `development` mask). The gate must require a cold start — `rm -rf node_modules/.vite` before `npm run dev` — and must assert `vite.config.ts` no longer contains the string `keyframes.js/dist`.

### Hardened W0 hard gate

W0 closes only when ALL of the following are observed and captured to `audit/W0-build-typecheck.md` / `audit/W0-playwright-boot/`:

1. `vue-tsc` is a committed `devDependency`; `npm run typecheck` runs `vue-tsc --noEmit` and exits 0 over the changed surface, with output captured. 3. Playwright probe at 375×667 / 1280×800 / 1440×900: the App tree renders fully — dock with its layers, a pane, an opened color-space `Select` — and the page is not in Vue error-recovery state. Console shows zero uncaught errors AND the network log shows zero non-2xx responses for same-origin assets (catches the `woff2` 403).
4. `dist/` is cleared, then `npm run build` exits clean; `dist/` contains only library artifacts (`value.js`, `value.cjs`, the `.d.ts` tree) — no `index.html`, no demo chunks. `npm run gh-pages` writes to `dist/gh-pages/` and does not touch `dist/value.js`.

6. The W0 close record names the merge-base decision and the commit hash.

A gate that omits items 1's "observed clean run," 2's cold start, 3's network assertion, or 4's `dist/`-clear can close green while A-key-1 lingers under a warm cache, the font 403 ships, or the typecheck never ran. Those four are the airtightness fixes.
