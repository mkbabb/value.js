## §5 The contrast delineation (T-35 — "delineate all")

### §5.1 The root the census could never see

Probed live at the owner URL (light scheme): `--ink-ambient-l` = **0.5103**;
`--accent-live` = **`oklch(0.9711 0.0143 32.6)`**. The picked brick-red
(oklch ≈ 0.51 0.13 32) was walked to L 0.97 / C 0.014 — **near-white with 1.4% chroma**. The
"Lab" trigger the owner photographed DOES wear this ink (`--space-title-ink` = safeAccent at
86% alpha — W4-1's own grammar): the site is compliant; the CONTRACT output is the miss.

Two stacked mechanisms in `certifyAccentInk` (`composables/color/ink.ts`):

1. **The mid-ambient degeneracy**: against a mid-lightness ground (ambient 0.51 — exactly
   where the atmosphere's derived field lives for mid-L picks) NO mid-chroma color clears
   4.5:1 + the 1.25 headroom; the walk must run to an L extreme.
2. **The chroma collapse**: `walkToFloor` steps L while `mapToGamut` clamps C at each step —
   at L 0.97 an H32 red holds C ≈ 0.014 in-gamut. Hue is preserved NOMINALLY; the percept is
   cream. "The current contrast friendly color" stops being the CURRENT color.

**Design ground for the cure** (research, not prescription): (a) certify against the SURFACE
rung, not the page — the trigger sits on the resting PLATE (dark cocoa in the owner's shots,
L ≈ 0.25–0.3), against which a red at L ≈ 0.65–0.70 holds real chroma AND clears the floor;
`useSafeAccentFn(surface)` already exists — the boot writer's ONE page-ambient instance is
the wrong referent for plate-seated text. (b) Walk along the gamut CUSP (max-C at each L —
the library's `getColorSpaceBound`/cusp machinery from the W1-src sampler) instead of
constant-C-then-clamp, so the certified ink lands at the most chromatic point that clears the
floor. (c) The headroom (1.25) compounds the over-walk; it exists for referent tolerance and
dies with P3/P5 published tier lightnesses — worth re-deriving then.

### §6.1 t33-01 — the readout gap + the pathological-number law

**Measured** (owner URL, built bundle): the readout carries `--readout-lines: 2`,
`min-height: 122.4px`; the painted tuple = **61px** (one line). The other 61px is the

retired at cell level ("39–51% of every cell was empty reservation"), surviving at line

**Design ground**: the lock's GOAL is "the containing card rect never moves" — that goal is
satisfiable per-VALUE-CLASS rather than per-space-worst-case: the tuple's line count at the
CURRENT value is derivable from the same static ch table (the live digits are known); the
reservation could ride the live line count while the LOCK constrains only transitions within
a digit-count class… **but** that re-admits card movement at sign/width boundaries — the
honest alternative: the SEAM below the readout absorbs the reserved line (the console rises
into the reservation; the gap becomes designed air, not dead band). This is a W8-class
design bracket; the research names both arms.

**The pathological-number half is a VALUE-DOMAIN law, not a layout law**: the reservation
table already derives worst-case ch from `getColorSpaceBound` — but nothing clamps the LIVE
value into those bounds (a URL/typed `lab(40% 999 47)` inks 999 and blows the derivation's
premise). The owner's words are the spec: "have a maximal value that's possible (dynamically
within that color space's ranges hereof)" — clamp at the model boundary (parse/normalize
already owns the ranges; the readout format then never sees an out-of-domain width), making
the static table's worst case TRUE by construction. Library-adjacent: the clamp belongs at
the demo's model write seam (`useColorPipeline`), not inside the library parser (CSS lab()
legitimately parses out-of-range values — the library must keep parsing them).
