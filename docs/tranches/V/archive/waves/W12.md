**State:** PLANNED · **Meta-tranche:** V.H Hardening · **Depends on:** W11

## Goal

Make every mutable resource clock honest and every published/history/provenance object stable.

## Scope and file bounds

Palette handle/workspace/release repositories and services, ETag/If-Match handling, publish/restore-release/fork provenance paths, transaction tests.

## Triumvirate and worktree

- **Specification:** `PALETTE-DOMAIN.md §3–4`, `PALETTE-FACILITIES.md`, `PALETTE-WIRE-CONTRACT.md`.
- **Plan:** execute W1's target/clock table; database-filter CAS; two-clock publish; separately clocked owner handle, palette policy and release policy; atomic Handle rename across destination/source NamespaceClaims and generation; palette-local immutable release manifests; consume W11's sole ContentObject port for Publish `+Release`, Restore `-Workspace(old)+Workspace(selected)`, seal `-Workspace` and unreported Release purge `-Release`; private lineage-coordinate provenance tombstones plus Report-pinned Release/ContentObject snapshots; atomic active-claim retirement before purged Handle removal; remove hash-as-ID and check-then-write.

## Current RED

## Completion evidence

- Public r18 history remains access-filtered and can disappear after unpublish, withdrawal or trash; owner r50 history remains complete for the same live active-or-trashed lineage. The routes share `ReleaseSummary` coordinates but neither auth mode nor cursor is interchangeable.

- Two writes with one expected clock yield exactly one success and one 412.
- Workspace `displayName`/tags/content and `updatedAt` advance Workspace revision while immutable `createdAt` remains fixed; Handle visibility/lifecycle/active release and exact `trashedAt/purgeAfter` coupling advance generation; nonredundant catalogTier/lineage moderation/reason advance PalettePolicy revision; snapshot moderation/reason advances ReleasePolicy revision; Principal, ColorNamePolicy and TagDefinition each advance only their named clocks. Immutable release fields change only by a new Publish. Runtime lineage construction persists exact `"lineage created"`/creator and Publish persists exact `"release published"`/publisher at revision one; r33/r35 return them until an ordinary mutation replaces both. Cut-created clear policy is the sole absent-pair arm. Admin policy/Principal DTOs return the persisted current rationale/actor/close coordinates rather than dropping them.
- Composite-response clock fixtures mutate each joined authority independently. Workspace detail/list returns Workspace, Handle, PalettePolicy and projected TagDefinition body clocks; Handle detail returns Handle and PalettePolicy clocks; Release detail returns Release, Handle and ReleasePolicy clocks; CurrentSession returns Principal/AuthFence-policy/PrincipalAuthority revisions. Those composite rows emit no HTTP `ETag`; direct one-clock policy, Principal, Grant, Tag, Report and color-name-policy representations alone emit the Wire matrix's strong validator. A mutation of one joined authority changes only its named body coordinate and can never leave a supposedly strong whole-response validator unchanged.
- `PrincipalDetail` contains only Principal-owned clock/state and omits AuthFence `policyEpoch`; rotating that fence changes `CurrentSession` but leaves `PrincipalDetail` bytes and its Principal ETag unchanged.
- Two identical palettes have separate `(paletteId, releaseNo)` histories.
- Restoring a release copies content to the private workspace and atomically moves its one Workspace reference when the digest differs; publishing the restoration acquires one new Release reference, creates a new local release and does not rewrite history.
- Runtime ancestry covers every transition: ordinary Workspace base absent; fork base equal to its unique external edge; edits preserve it; same-lineage restore selects it; Publish maps absent base to no parent/source, local base to `parentReleaseId`, and external base to the matching `sourceReleaseId`, then advances Workspace base to the new Release. Provenance walks local parents nearest-first then at most one external source, with the fork edge as a cross-check rather than a second branch; cycles, mixed parent/source, cross-lineage parent and edge mismatch fail closed.
- Publish rejects either stale workspace or stale handle precondition without partial release/catalog change and atomically creates the immutable release plus initial clear release policy carrying `reason="release published"`, the authenticated publisher and the same transaction time before exposure. Owner handle mutations never use public release ETags.
- Fork/report internal provenance targets a stable release and survives source trash/withdrawal/purge through a private tombstone carrying palette-local lineage coordinates. Report stores `targetPaletteId` plus the immutable Release/ContentObject target snapshot and pins that evidence while retained. Public Q-PROVENANCE rechecks every hop and replaces a hidden/purged ancestor with only `{kind:"unavailable",ordinal}`—no slug, actor, content or correlatable private ID. Vote targets lineage and survives Publish. A fork creates no destination Release: its exact `{slug}` transaction claims a new lineage and copies the selected manifest into a revision-1 private Workspace with `basedOnReleaseId`. `fork_edges.targetPaletteId` is unique forever: many destinations may cite one source Release, but each fork-born destination has exactly one retained origin and a normally created lineage has none. Purge retires a still-active slug claim in the same transaction as Handle tombstone insertion/live removal. Source-visible→fork-visible then source unpublish/trash/withdraw/purge vectors prove durable internal ancestry, reviewable report evidence and public opacity.
- First Publish then second Publish leaves the first immutable Release as public history (`publicHistory=true` is server-set on every Publish, never an input/toggle) while the second is active. Owner history always sees both; public history sees both only while the lineage/policies pass access, unpublish hides both, and individual ReleasePolicy withdrawal hides only its target without byte mutation.
- Rename-away versus concurrent creation/rename toward the old spelling, and two concurrent renames toward one new spelling, serialize on the permanent NamespaceClaim rows: exactly one transaction succeeds, the loser receives `409 palette_slug_claimed`, and neither old nor new spelling can be transferred or orphaned.
- Color-name fixtures persist the Proposal's normalized display spelling and immutable `claimedRevision` beside the Slot's current `claimRevision`. Reject/withdraw then successor-claim leaves the old Proposal ETag byte-stable and reconstructible from Proposal+Policy alone; an old review carrying the successor Slot revision or an old Slot revision/pointer mismatch fails without changing either attempt.
- No `If-Match:*`, hash identity, or pre-read-only concurrency decision remains.
