Exact `st1_` 32-byte session tokens persist only as the frozen SHA-256 domain digest. - The deployment-only DB-direct `admin:bootstrap` command is absent from Hono and authorized by trusted workload identity. The operator supplies an offline-kept canonical OS-CSPRNG `rs1_` secret over stdin; only its HMAC verifier persists and plaintext is cleared. Before its transaction the command mints one canonical UUIDv7 correlation using current Unix milliseconds and OS-CSPRNG random bits and persists it on the singleton BootstrapRecord. Concurrent fresh-schema invocations create exactly one PlatformState/bootstrap tombstone, PublicHandle NamespaceClaim, Principal/AuthFence, Grant containing precisely `{audit:read,name:review,palette:review,principal:read,principal:write,principal:close,grant:read,grant:write,tag:read,tag:write,report:review}` plus `creationReason="initial authority bootstrap"` and `attribution={kind:"bootstrap-workload",bootstrapRecordId:"initial-admin"}`, PrincipalAuthority with eleven unit counts/bits and that grant ID, AuthorityGuard `{epoch:1,eligibleCount:1}` and the Domain table's one `bootstrap` outbox row with absent actor/command, that correlation and exact after projection; ordinary delivery creates exactly one AuditEvent and every later invocation fails. Before that commit `/health` is 503-not-ready and registration/all product routes are `503 platform_uninitialized` with zero writes; bootstrap atomically flips readiness.
- Root eligibility is an active Principal plus its bounded PrincipalAuthority projection containing both `grant:write` and `principal:close`. A grant is a nonempty unique subset of the fixed eleven scopes, at most eleven are active per Principal, and a create that adds no absent scope is `409 grant_redundant`. r43 stores the canonical body reason and exactly one Principal attribution arm; r42/r43/r44 return that immutable creation provenance while revocation adds—but never replaces it with—its own actor/reason. Every grant create/revoke and Principal suspend/reactivate/close updates at most that bounded projection, CASes the shared guard epoch, applies the target eligibility delta to `eligibleCount` and aborts zero with `409 last_root_authority` in one transaction; ordinary authorization inspects no grant history. Concurrent authorized self-removals yield one success and one named 409; a separate cross-target mutual removal yields one success and an ordinary actor-fence 401/403 with zero second effect. A periodic epoch-stable full reconciliation derives expected projections from bounded active grants and records count plus the frozen holder digest outside the command path, and a large-cardinality fixture proves constant per-target reads/writes; no wildcard scope, stale pre-read, zero-holder commit or re-bootstrap path survives.
- Registration, recovery and credential rotation work across two API processes. Rotation advances AuthFence policy, revokes the previous verifier, invalidates every sibling old-policy session, marks the initiating row for exact receipt replay and issues a replacement cookie; public-handle regeneration is absent. Mongo-`$$NOW` fixed windows are exactly registration-new-effect source 5 per 10 minutes, recovery-failed-proof source 10 per 15 minutes and recovery-failed-proof handle 5 per 15 minutes. A matching completed registration receipt is credential-proved before the new-effect budget and bypasses it; a correct recovery proof or authorized completed replay is verified with the same dummy/real work before, and bypasses, both recovery failure budgets. Only a registration receipt miss/mismatched binding consumes the new-effect source row; only a failed recovery proof consumes both failure rows, while a malformed recovery consumes the source-failure row after dummy work. There is no registration-handle row. One transaction plus the `(routeClass,subjectKeyId,subjectHmac,windowStart)` unique index admits exactly N across processes, persists only the versioned HMAC subject, and returns non-enumerating `429 abuse_window_exceeded` plus exact `Retry-After` without a receipt or domain effect. Abuse-key rotation waits until every live fixed window under the old key has closed; TTL lag cannot change a boundary, and no process-local or edge counter survives.
- W10's one operations wrapper is exercised for registration/recovery/logout/rotation/self-lifecycle through both processes. It performs completed-receipt precedence, otherwise inserts the invisible intent and lets only its winner recheck AuthFence before effect; losers restart the whole wrapper. Secret roles are selected after receipt lookup, and every issuance re-reads/reverifies the exact credential under the fence. Same binding replays from the initiating session or command-linked rotation successor only; another valid session gets non-result-bearing 409. Logout/lifecycle/rotation retain only the initiating one-way digest, revocation time/reason and revoking command ID through natural expiry; TTL then purges the session while receipt/audit remain. - Every browser `POST`, `PUT`, `PATCH` and `DELETE` requires the signed tuple containing the profile's one exact Origin (`https://color.babb.dev` production or `https://value.localhost:4173` local), `Sec-Fetch-Site: same-origin`, `Sec-Fetch-Mode: cors`, and `Sec-Fetch-Dest: empty`; every browser `GET` supplies either that exact fetch tuple with absent-or-exact profile Origin or none of those four fields.
