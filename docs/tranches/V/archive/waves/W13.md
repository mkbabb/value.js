# V.W13 — Durable Commands, Votes, Forks, and Counters

**State:** PLANNED · **Meta-tranche:** V.H Hardening · **Depends on:** W10–W12

## Goal

Make retries and social/lifecycle side effects exactly-once across processes and failure boundaries.

## Scope and file bounds

Extensions to W10's existing `api/src/platform/operations/**` command-receipt foundation; idempotency middleware/client contract; palette/social operation integration; vote/fork/report repositories/services; color-name slot/proposal/decision and tag-policy commands; derived stats/reconciliation; failure-injection tests. Principal registration/recovery/logout/rotation/lifecycle receipt integration remains W10-owned.

## Triumvirate and worktree

- **Specification:** `PALETTE-DOMAIN.md §3–4`, `PALETTE-FACILITIES.md`, `PALETTE-OPERATIONS-REPLAY-CONTRACT.md`.
- **Plan:** consume and extend W10's exact intent-first operations wrapper; retain its server-derived namespaces, permanent `requestBindingDigest/effectStatus/resultReference`, receipt-bound session/grant identity, command-linked successor exception and AuthFence recheck without a second middleware path. Integrate palette/social mutations into that one wrapper: lineage votes, the exact fork `{slug}`→NamespaceClaim/private Handle/copied Workspace/clear Policy/unique-destination provenance transaction with `+Workspace(sourceDigest)`, private-safe public fork projection, first release Report with permanent `+Report(targetDigest)` and zero duplicate/replay delta, the unique normalized-name slot with retained proposal attempts, attributed color-name decisions, tag retirement and the singleton catalog dirty clock.
- **Implementation owner:** operations lead plus vote/fork units; one transaction integration owner.

## Current RED

The idempotency map is process-local, handlers begin before reservation, clients mint keys inside calls, vote membership/count and fork/lifecycle counts can diverge, and retries can execute twice.

## Completion evidence

- The same command sent concurrently to two API instances mutates once; the loser waits/re-reads and returns the permanent replay contract's original status, exact canonical success-body bytes and exact `Content-Type`. The tuple carries the exact strong `ETag` bytes only for rows r25, r28, r32, r34, r36, r40, r41, r43, r44, r46, r47 and r49; every other mutation success forbids that field, exactly as `PALETTE-WIRE-CONTRACT.md §5.2` requires. Every success body is ≤65,536 bytes before commit. The route binding, reference-free deterministic-CBOR status/header/body tuple and binary AES-GCM envelope/tag match `PALETTE-OPERATIONS-REPLAY-CONTRACT.md`; later revision/purge cannot reconstruct, enrich or change them, internal effect/tombstone references never serve replay, and old replay keys retire only after the specified state change, writer drain, CAS rewrap and epoch-stable zero proof. `Set-Cookie` alone is regenerated by the narrowly authorized command-linked session CAS and is never stored in the tuple.
- For every palette/social command, the invisible unique-intent winner performs the inherited W10 AuthFence recheck/effect while the loser restarts full receipt precedence after commit. A fence-conflict retry restarts that wrapper; authorization failure aborts the invisible intent. W13 neither forks nor reimplements the platform operation primitive.
- Reusing a command ID with a different non-secret binding inside its server-derived namespace returns `409 idempotency_key_reused`; malformed/noncanonical/non-v7 keys fail before domain work.
- Aborted/crashed transactions leave no visible intent or claim; the encrypted response envelope and permanent effect row never expire, so a command can neither re-execute nor reconstruct its response; registration/recovery/session secrets and bearers never enter a binding digest, receipt, durable browser body or replayable response envelope.
- r32 command binding records both `policyRevision` and immutable `claimedRevision` from its composite ETag; changing only the latter on a completed command is a binding mismatch and returns `409 idempotency_key_reused`.
- W10's registration/recovery/session/lifecycle receipt suite remains green as a consumed platform dependency; W13 adds no session namespace, credential branch, cookie settlement rule or client-visible auth result.
- A caller-supplied stable command ID survives repeated/concurrent HTTP requests; W15 alone owns durable browser outbox preservation across reload.
- Lineage vote `PUT`/`DELETE`, immutable-release fork/report, color-name decide/withdraw, tag create/retire and trash/restore/purge survive injected failure under the Domain's single edge/count law. Vote returns the exact transactional `JsonUInt` edge count; public fork detail/list count is an exact current access-filtered `JsonUInt` join. Every state-changing catalog-affecting winning transaction advances the singleton CatalogBuildState `dirtyRevision` once, while an already-satisfied vote, abort or replay advances it zero times; no per-mutation rank event or rival public counter exists. Checked count aggregation rejects `Number.MAX_SAFE_INTEGER+1` as `platform_consistency_fault` rather than emitting an imprecise result. Fork creates exactly one permanent slug claim, generation-1 active/private Handle, revision-1 copied Workspace, clear Policy with exact `"lineage created"`/forker provenance, source-release edge and Workspace content reference, creates no destination Release, and leaves none of those rows/counters on a conflict/abort. Trash creates one schedulable episode keyed by the literal receipt namespace plus command and copies the command's edge correlation, actor and authority witness; restore cancels that exact episode but retains its terminal row, and a later trash/Principal-close creates a fresh episode rather than reusing or treating history as active. Injecting the same raw UUID under distinct owner/admin Principal namespaces proves two noncolliding episode identities and no command-only/hash alias. First Reports pin their immutable Release/ContentObject target snapshot with one permanent Report reference; duplicate/replay adds zero. Proposals retain attributed evidence and tag retirement performs zero Release writes.
- Every newly completed W13 mutation emits exactly the single event assigned by the Domain construction table; replay, abort and rejection emit zero. Vote PUT/DELETE use the stable `palette/<paletteId>` subject and exact before/after membership projections rather than a vote-edge subject; fork and Report use live UUIDv7 subjects; palette/release/color-name/tag/policy subjects retain ResourceId so cut-created v5 targets remain attributable. r25 Report creation emits one `report-create` event. r49 emits one `report-review` event whether `appliedEffect` is `none`, `palette-policy` or `release-policy`; its closed Report projection binds the applied target and policy before/after revisions, and no second policy event exists. The effect transaction supplies the exact actor, command, edge correlation, projection bytes/digests and authority arrays; no command writes a bare subject ID, infers a type during delivery, hashes a route-local object or serializes a generic string branch.
- Two concurrent fork commands with distinct command IDs targeting one normalized destination slug produce one complete unpublished lineage and one `409 palette_slug_claimed`; the same command through two replicas replays the winning lineage, a retired claim has the same conflict, and the winning response carries the new Handle/Workspace IDs and ETags. The edge's unique key is `targetPaletteId`, so one source can produce many destinations but a destination cannot acquire a second origin.
- Public fork count/list never exposes that private unpublished destination or actor. It appears only after a current Release passes the public predicate, disappears after unpublish/trash/withdrawal, and continues from the last scanned edge without deleting provenance.
- Two concurrent proposals for one normalized name produce one pending Proposal and one `409 color_name_claimed`; approve retains the occupied slot, reject/withdraw releases it atomically, and a later resubmission creates a new Proposal/claim revision without rewriting the old decision.
- Two same-actor Reports against one immutable Release produce one retained case and one `409 report_exists` reference even under different commands; review never releases that identity. Two concurrent TagDefinition creates for one normalized key produce one stable definition and one `409 tag_key_claimed`; retirement never permits reuse and performs zero Release writes.
- W13 produces only the singleton dirty revision in the effect transaction; W14's sole leased CatalogBuildState builder coalesces it into rebuildable historical RankEpochs. Ready epochs are immutable and access/active-release rechecked at read; W13 creates neither an epoch-per-command nor a second builder path.
- Process-local idempotency/rate state and toggle semantics are absent.
