## 5. Card and micro-UI laws

1. A Card houses one bounded object/specimen. A page region, empty column, inner stage or mere padding group does not become a Card by default.
2. A card has one protagonist, one identity line, and at most one persistent action/status region. Additional equal-weight zones require a different `InstrumentChassis` composition.
3. Header‚Üíheadline uses title gap; headline‚Üínext semantic section uses section gap. Renderer, icon or touch footprints may reserve collision space only on the axis where collision exists.
4. A divider is retained only when grouping would be ambiguous without it. Spacing plus material already expressing the same boundary makes the line duplicative.
5. A small icon/mark is either data, status, labeled action, drag affordance, focus/selection register or removed. Decorative controls and operable ornaments without names are forbidden.
6. Add affordance when the surviving action/state is otherwise undiscoverable; do not compensate for an unnecessary action with tooltip proliferation. Subtraction precedes explanation.
7. Visual glyph size, operable target size and layout reservation are separate quantities. Accessibility floors do not require bloated visible chrome.
8. Real rendered relation wins over token intent. Adjacent rungs, measured rects and ink gaps appear in DELTA; token presence alone cannot close a row.
9. A renderer specimen is not an unlabeled button. Picker Blob is inert presentation (`aria-hidden` when redundant with the color value); Copy is a named action in the action region. One action never occupies both the specimen and the action instrument.
10. Readout and editing are separate jobs. The large Picker headline is read-only output; the semantic W21 numeric fields are the only direct channel editors, with commit/cancel/error semantics and no duplicate `contenteditable` path.
11. A display-sized readout is not therefore a document heading or live status. Picker uses one labeled non-live domain output; route H1 owns heading hierarchy and the focused W21 control announces the value it changes.
12. Every Browse/Library palette entity Card uses exactly `{size:"sm", material:"content", tier:"quiet", surface:"opaque", shadow:false, grain:false, specular:"off"}`. Its `C = --spacing(4)` padding ladder is invariant across viewports; only the workspace gutter responds. The Card/article root is a noninteractive container: it owns no activation, focus or selection state, carries no HTML `inert` attribute (palette-Card `[inert]` count `0`) so its live child remains reachable, and has no cartoon/grid/selected variant. Its one native named `<button type="button" aria-pressed="false|true">` child spans specimen/identity and alone owns activation, visible selection and focus; neither root nor seat uses `aria-selected`, `role="option"`, a listbox or composite-selection fiction. `aria-pressed="true"` iff the seat's palette ID equals the selected inspector identity, no more than one seat is true, and no selection means every seat is false plus no inspector. An optional reorder handle is a separate named control. Card/face selected pixels remain unchanged; ordinary pointer, Enter, Space and AT probes must converge on the same pressed seat and inspector identity.
13. Type follows one role matrix: route H1/major argument `text-display`; instrument identity `--type-title`; palette identity `--type-subheading`; section heading `text-heading`; prose `text-prose`; control/label `text-small`; value/code/provenance `text-mono-small` or existing `mono-caption`.
