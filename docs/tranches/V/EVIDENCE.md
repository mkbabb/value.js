# Tranche V — Behavioral and Visual Evidence

## 1. Principle

Evidence observes shipped behavior. Source tokens, Markdown status words, expected-red counters, headless screenshots, and mocked server responses cannot establish a global product claim.

Each wave gets one completion check. The check should be able to fail because the user-visible or domain behavior is wrong, not because a ledger sentence is absent.

Every visual π starts in the live routed in-app Browser: open the exact state, inspect it, and capture the first witness there. Product implementation uses a bounded inspect→change→reinspect loop in that same Browser before the slice widens. Computer Use or headed hardware is used only when native assistive technology, OS/browser chrome, a real device, Safari-only behavior or a GPU surface is inaccessible to the Browser. Standalone/headless Playwright is allowed only for named cross-engine binary automation, exhaustive repetition or synthetic timing the Browser cannot express; its output may supplement a Browser-opened claim but may never author or attest visual gestalt. Literal filenames remain coordinates rather than execution preferences.

## 2. π

π is the minimum decisive real-rendered witness. Open, inspect and capture every visual π in the live routed Browser first; escalate only the Browser-inaccessible part under §1. Use a frame pair for static form, synchronized filmstrip/trace for motion, visual frame plus input/assistive transcript for accessibility, trace/waterfall plus relevant frame for performance, and a renderer-identified temporal sequence for GPU behavior. Capture only the states needed to adjudicate the claim:

- one 1440px desktop and one 390px mobile frame;
- a complete-route 320px frame and actual 400% in-app Browser zoom when the claim is PR-33–35, the closed type matrix, retained palette-Card geometry, or the Lab/Blob crop; actual zoom is not replaced by responsive emulation;
- both schemes only when the claim concerns material/color parity;
- before/after or state-A/state-B in the same browser, seed, viewport, and renderer;
- real GPU when translucency, Aurora, Blob, filter, or frame choreography is claimed.

W32 assembles the small record of record. The expected upper bound is dozens, not thousands, of images. Every cited image is tracked and listed in a manifest; all uncited scratch is deleted before close.

## 7. Orthogonal coverage

The final record is not a route×state×scheme×viewport Cartesian product. It contains one canonical full-route frame for each route, one witness for each shared state/interaction/motion mechanism, and a separate witness only where a feature changes that mechanism. Scheme pairs exist only for material/color claims; viewport pairs only for topology changes; real-GPU sequences only for glass/Aurora/Blob/motion. `W32` maps every omitted combination to the witness that covers it and reruns any assistive journey whose focus, loading, font or renderer boundary changed in W31.
