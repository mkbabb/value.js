## P0 — the pane cannot render at all on the pinned glass-ui (build crash)

### F-1 [P0] glass-ui 3.10.1 `@import` of a non-existent CSS file crashes the whole Vite CSS pipeline → no pane mounts

Live capture is conclusive: **all six** `C5-*.png` are the Vite/postcss error
overlay, not the app. `/tmp/c5-console.txt` shows 5× `500 Internal Server Error`
for `style.css`. The overlay text (read from `shots/C5-01-palettes-view.png`):

```
[plugin:vite:css] [postcss] ENOENT: no such file or directory, open
'.../components/custom/tabs/segmented-tabs.css'
```

Root cause chain (each link verified against the tree, not docs):

1. `demo/@/styles/style.css:23` does `@import "@mkbabb/glass-ui/styles";`
2. that resolves to `node_modules/@mkbabb/glass-ui/dist/styles/index.css`, whose
   **line 148** is `@import "../components/custom/tabs/segmented-tabs.css";`
3. that target **does not exist** — the dir
   `dist/components/custom/tabs/` ships `SegmentedTabs.vue.d.ts` (the BouncyTabs→
   SegmentedTabs rename) but **no `segmented-tabs.css`** (`find … -name segmented-tabs*`
   → nothing). glass-ui's own packaging emitted a dangling `@import`.
4. postcss `@import` resolution is **fatal** — it takes down the entire
   `style.css` module, so the dep-optimizer cache did NOT mask it and **no pane
   renders**. The brief's "may render fine from cache" did not hold here.

This is strictly worse than the "3 typecheck errors" the brief anticipated: it is
a hard, total render outage on the pinned dependency. The two demo *typecheck*
skews are real and adjacent but secondary:

- `ComponentSliders.vue:117` imports `@mkbabb/glass-ui/glass-carousel`; glass-ui
  3.10.1 exports `./carousel` (verified in its `package.json` exports map). Subpath
  renamed.
- `MixSourceSelector.vue:4` and `PaneSegmentedControl.vue:18` import `BouncyTabs`
  from `@mkbabb/glass-ui/tabs`; the component is now `SegmentedTabs`.

**Gestalt fix (tranche N work-order, NOT a band-aid):** the value.js↔glass-ui
publish-spine is out of sync — value.js is pinned to a glass-ui cut whose dist is
internally inconsistent (dangling `@import`) *and* whose public API drifted
(subpath + export renames). N must (a) get glass-ui to ship a self-consistent cut
(the dangling `segmented-tabs.css` `@import` is glass-ui's bug to fix at source),
then (b) migrate the 3 value.js consumers in one sweep: `glass-carousel`→`carousel`,
`BouncyTabs`→`SegmentedTabs`. Until both land, the demo (and therefore the C5 pane)
is dark. This is the single highest-priority blocker for any N visual work on this
pane.

## P1 — design incongruences (grounded in source + working-build pixels)

### F-2 [P1] "My Palettes" rainbow-gradient heading is a **dead class** — never renders the gradient

`PalettesPane.vue:4`:
```html
<span class="capitalize pastel-rainbow-text">My Palettes</span>
```
But `pastel-rainbow-text` is defined **only** inside `PaletteDialogHeader.vue:94`
in a `<style scoped>` block (so it carries a `data-v-*` selector and applies only
within that component), and it appears **nowhere** in glass-ui (grep across the
whole package: zero hits). `PalettesPane.vue` has **no `<style>` block** at all.

Result: the class is inert on this heading; "My Palettes" renders in plain
`foreground` serif, not the intended audacious rainbow pop. The D.W6
`palettes.png` confirms it — "My Palettes" is plain black, flatly contradicting
the same-named treatment that *does* fire in the dialog header. This is exactly
the "colorful audacious pops" the house language wants, silently lost.

**Gestalt fix:** promote `pastel-rainbow-text` (and the `.admin-golden-text`
sibling) into a glass-ui *source* utility (`@utility pastel-rainbow-text`) so it
is a real, reusable design-system token — then both the dialog header and the
PalettesPane heading clip the same gradient. Do not paste a scoped copy into
PalettesPane (that re-creates the drift the facade discipline exists to prevent).

### F-3 [P1] Browse vs My-Palettes header treatments are visually inconsistent

Working pixels (`D.W6/browse.png`): the **Browse** pane header ("Browse" + the
"Discover palettes from the community." sub) and the **My Palettes** header
("My Palettes" + "Save, organize, and share your colors.") are rendered by the
same `PaneHeader.vue` and look near-identical — both plain serif black. Yet the
*dialog* surface (`PaletteDialogHeader.vue`) gives the same content a thick
gradient accent bar (`:24`-tall `linear-gradient`), a WatercolorDot, the rainbow
"Palettes" word, and a `text-display-2 font-black` title. The pane-embedded
headers get **none** of that audacity. The two entry points to the same feature
read as two different design eras.

**Gestalt fix:** unify on one "palette feature header" treatment. Either lift the
dialog header's accent-bar + rainbow word into `PaneHeader` (as an opt-in
`accent`/`gradient` variant) or, at minimum, give the pane headers the rainbow
word so the hierarchy pop is consistent. The φ-typography ladder is already
coherent (F-OK1); the *decoration* is what diverges.

### F-4 [P1] PaletteCard color strip collapses to a flat dark slab for dark-heavy palettes — the card's primary visual is its weakest

`D.W6/browse.png` shows four of five cards whose entire `PaletteColorStrip`
(`h-10`, the card's dominant visual element) is a single near-black bar — because
those palettes are dark-dominant and the strip simply tiles
`backgroundColor: color.css` at `max(100/n, 0.5)%` width
(`PaletteColorStrip.vue:18-22`). With no min-contrast floor, a dark palette's
strip is indistinguishable from the card chrome; the card loses its identity and
the grid reads as a stack of black bars. The house language ("colorful audacious
pops") is undercut by the very element meant to carry color.

**Gestalt fix:** the strip is decorative (`aria-hidden`), so a subtle inner
hairline divider between swatches (`box-shadow: inset 0 0 0 0.5px …`) or a 1px
checker/contrast separator would restore swatch legibility without changing the
palette. A flat single-hue strip should still show its swatch boundaries.

### F-5 [P1] Degraded error surface uses a glass-ui **dock** primitive as a content-area button — wrong component family, weak affordance

`BrowsePane.vue:37-43` renders the no-backend retry as:
```html
<DockIconButton compact class="text-mono-small text-primary" @click="...">
  Tap to retry
</DockIconButton>
```
`DockIconButton` (imported from `@mkbabb/glass-ui/dock`) is a dock-chrome
primitive; using it as an inline content-area retry button is a component-family
mismatch (and couples a content surface to the dock's visual contract). The K
DELTA C-OK3 confirmed the *copy* posture is right ("Failed to load / Tap to
retry", no raw stack — degraded-clean), but the *control* is the wrong primitive.
Compare the My-Palettes pane, which correctly uses the shadcn `Button`
(`PalettesPane.vue:36`) and glass-ui `ConfirmDialog`.

**Gestalt fix:** use the glass-ui `<Button variant="ghost">` (the same primitive
G.W2 Lane F standardised on for `PaletteSlugBar`) for the retry action. Reserve
`DockIconButton` for the dock.

## P2 — polish / smaller incongruences

### F-6 [P2] Empty-state typography is inconsistent across the two empty surfaces

Two different empty-state renderers exist with **different** type treatments:
- `EmptyState.vue:2` → `text-mono-small italic`, `py-4`
- `PaletteCardGrid.vue:13-18` (the one actually used by both panes) →
  `fira-code text-mono-small italic`, `py-8`

Same semantic ("nothing here"), two paddings (`py-4` vs `py-8`) and one adds the
explicit `fira-code` mono face while the other relies on `text-mono-small`'s
family. The standalone `EmptyState.vue` component appears to have **no live
consumer** in the panes (both panes use `PaletteCardGrid`'s inline empty `<p>`),
so it is likely dead. Spacing-rhythm drift on a low-stakes surface.

**Gestalt fix:** delete the unused `EmptyState.vue` (or make `PaletteCardGrid`
consume it) so there is one empty-state recipe with one padding rung.

### F-7 [P2] Saved-card count is shown **twice** in the My-Palettes pane

`PalettesPane.vue` renders the saved count in the header Badge (`:5`) **and**
again as a separate toolbar line "{n} palette(s)" (`:31-34`). Two count chips
within ~200px vertical of each other is redundant hierarchy noise. `D.W6` shows
the header Badge already carries it.

**Gestalt fix:** drop the toolbar count line; keep the header Badge (closer to the
"My Palettes" title where the count belongs) and let the toolbar carry only the
destructive "delete all" affordance.

### F-8 [P2] Vote / heart control has no optimistic-affordance distinction from a disabled-while-degraded state

`PaletteCard.vue:86-97` renders the vote button unconditionally for non-local
palettes; with no backend the click is a no-op (`pm.onVote` will fail silently /
warn). The button gives full hover/active feedback (`active:scale-95`) but the
vote never lands. In degraded mode the heart looks fully live but is inert. Low
severity (degraded is expected) but it is a small honesty gap.

**Gestalt fix:** when `pm.browseError` is set (degraded), the cards are already
empty so this rarely surfaces; if N keeps a degraded read-only browse, gate the
vote button's interactive affordance on backend availability.

### F-9 [P2] Featured-badge golden-shimmer is the only motion on the card; remix/version/vote affordances are static

`PaletteCard.vue:334-356` gives the Featured badge a 4s `golden-text-shimmer`.
That is the house "purposeful animation" idiom — but it is the *only* animated
element on the card. The fork/version/vote micro-indicators (`:50-97`) are flat.
Not a defect, but the motion budget is unevenly spent: a permanent infinite
shimmer on a rare badge, nothing on the frequently-used vote heart. Consider a
one-shot heart-fill spring on vote (the kind of purposeful, event-driven motion
the house language prefers over ambient loops).

## Strengths to GUARD (do not regress in N)

- **F-OK1 — φ-typography ladder is coherent and distinctive.** The card uses
  `text-subheading` (1.272rem = √φ, glass-ui token), `text-mono-small`,
  `text-micro` (11px), `text-caption` in a consistent descent; the Fraunces
  display serif (D.W6 "Lab" italic, the large numeric readout) is genuinely
  non-generic. Grounded: glass-ui `tokens.css` `--type-subheading: 1.272rem; /* √φ */`
  + `PaletteCard.vue:32` + `D.W6/palettes.png`.
- **F-OK2 — degraded posture is console-clean & typed.** `/tmp/c5-console.txt`
  shows zero palette-runtime console noise (the only errors are the F-1 build
  crash). The BrowsePane error block (`:32-44`) renders a typed
  `pm.browseError` string + retry, never a raw stack (K DELTA C-OK3 confirms).
- **F-OK3 — glass surface + cartoon-shadow card language is consistent.**
  `rounded-card border bg-card` + `hover:shadow-card-hover` (the 10px cartoon
  offset, `style.css:79-81`) is applied uniformly across PaletteCard, the panes'
  `Card tier="wash"`, and the skeleton. Cohesive pop-art surface idiom.
- **F-OK4 — the card god-module split (H.W3) holds.** PaletteCard is 388 LoC with
  swatches lifted to `PaletteCardSwatches.vue`; menu, strip, rename, feedback all
  factored. a11y landmarks (`role="article"`/`role="list"`) + aria-labels on every
  icon-only control are present and correct.
