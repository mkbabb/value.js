## 2. K-Era Conformance Suite

The K.W2 conformance suite consists of three files under `test/conformance/`:

### 2.1 `crud.test.ts` (8 tests — K.W2 CRUD-CONTRACT v2)
Binds the wire contract for the palette CRUD shape:
- POST → 201 + FormattedPalette key set (hard-verifies `id` absence; K.W2 hard-removal)
- GET → 200 + ETag header + `voted` key in single-resource read
- GET missing slug → 404 problem+json
- PATCH without If-Match → 428 PreconditionRequired
- PATCH with stale If-Match → 412 PreconditionFailed
- PATCH with matching If-Match → 200 + fresh ETag
- DELETE → `{deleted, deletedAt}`; `/restore` → full FormattedPalette
- publish/unpublish in-place idempotency parity (same document count, `published` tracks `visibility`)

### 2.2 `diff.test.ts` (9 tests — J.W2 /diff + atom-diff algebra)
Binds `J-diff-shape.md §3/§4`:
- Pure algebra: `diffAtoms(A, A)` empty; order-independence of set-hash; diff symmetry; closed op vocabulary
- Wire envelope: exactly `{fromHash, toHash, ops, identical}` (no fromSetHash/toSetHash redundancy per §2.4)
- `fromHash`/`toHash` ARE the atom-set-hashes (not content-hashes)
- Immutable-pair ETag; If-None-Match → 304
- Identical versions → `ops:[], identical:true`
- Unknown `from` → 404
- Divergent branches (inv-J-1 chain guard) → 422

### 2.3 `idempotency.test.ts` (6 tests — K.W2 Idempotency-Key replay)
Exercises `src/middleware/idempotency.ts`:
- Same key → identical 201 body replayed; ONE document created; `Idempotency-Replayed: true` header on replay
- Replay short-circuits even when the second body differs (key is the token)
- Different key → distinct execution, two documents
- Absent key → normal behavior (no coupling)
- Thrown error response NOT cached (adversarial-review regression guard — same key re-executes)
- Key scoped per session (different session same literal key → independent execution)

## 4. Exposure: Health, Docs, OpenAPI

### 4.1 Health Endpoint

The API defines a health check at `GET /` (`src/index.ts:78`):
```typescript
app.get("/", (c) => c.json({ status: "ok", service: "palette-api" }));
```

This serves `{ "status": "ok", "service": "palette-api" }` — a bare status object. There is no `/health` endpoint; `GET /health` returns a 404 problem+json:
```json
{"type":"urn:palette-api:problem:not_found","title":"Not Found","status":404,"instance":"/health"}
```

The `compose.yaml` healthcheck (line 34–38) exercises the `GET /` endpoint via `node -e "fetch('http://localhost:3000/')"` — it uses the root path, not `/health`.

The `scripts/deploy.sh` health-gate (line 44) checks `$API_HEALTH_URL` which defaults to `https://api.color.babb.dev/palettes` (a `GET /palettes` — a data endpoint, not the minimal health check). This is intentional: the gate validates the full stack including MongoDB connectivity, not just process liveness.

### 4.2 Docs / OpenAPI

**There is no `/docs`, `/openapi.json`, or Swagger endpoint.** Live verification:

```
GET https://api.color.babb.dev/docs       → 404 problem+json
GET https://api.color.babb.dev/openapi.json → 404 problem+json
GET https://api.color.babb.dev/health     → 404 problem+json
```

The API has no `@hono/zod-openapi` dependency and no schema-generated docs. The endpoint table in `api/CLAUDE.md` is the only machine-readable(ish) contract documentation. This is a known gap with no tranche N work item tracking it.

### 4.3 Error Fallback

The `app.notFound` handler (index.ts:93–103) returns RFC 7807 problem+json for all unmatched routes — including the absence of `/docs`, `/health`, and `/openapi.json`.
