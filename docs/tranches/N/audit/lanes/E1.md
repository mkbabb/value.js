### (d) useLayerTransition "Local fork" — glass-ui 3.x NOW SHIPS A SUPERIOR PRIMITIVE

This is the most consequential refresh. The demo's fork
(`demo/@/components/custom/dock/composables/useLayerTransition.ts`) self-describes
as a "Local fork … the per-id class + inert helper is a thin convenience" (its
header, `:1-5`). At HEAD that header is **stale**: glass-ui 3.10.1 (the installed
`file:../glass-ui`, `tranche/AY`) ships `useLayerTransition` and **exports it on
the `/dock` subpath** (`node_modules/@mkbabb/glass-ui/src/components/custom/dock/index.ts:44`
`export { useLayerTransition } from "./composables"`; dist d.ts present at
`dist/components/custom/dock/composables/useLayerTransition.d.ts`).

The two have **diverged in opposite directions**:

- **glass-ui 3.10.1** (`src/components/custom/dock/composables/useLayerTransition.ts`):
  a single-scalar `SpringProgress` morph (one analytic ODE → `--dock-morph-t`),
  iOS interruptible-physics + velocity-continuity retarget, FLIP measured once.
  Returns `{ onTransitionEnd, currentLayer, leavingLayer }`.
- **demo fork** (`useLayerTransition.ts`): the OLD CSS-transition FLIP — width-pin
  + `requestAnimationFrame` + a hard-coded `setTimeout(…, 400)` crossfade cleanup
  (`:89-93`). Its only value-add over glass-ui is `layerProps(id)` → `{class, inert}`.

So the demo fork is not "thin convenience around the same engine" — it is a
**whole separate, inferior animation engine** (no spring physics, no velocity
continuity, a magic-number 400ms timer). The single consumer
(`dock/layers/ActionBarLayer.vue:57-58`) uses only `layerProps` + `onTransitionEnd`.

**Elegance fix:** retire the ~120-LoC demo engine; wrap glass-ui's
`useLayerTransition` ({currentLayer, leavingLayer}) in a ~12-LoC `layerProps`
adapter (the class/inert mapping is trivial off the two refs). Net: −108 LoC AND
the dock sub-layer transition inherits glass-ui's spring/iOS correctness for free.
Ledger M.W3.C is now unblocked — the 3.3.0 anchor it was gated on is moot; the
primitive is live in 3.10.1. **This is the second-highest elegance win in the lane
(−108 LoC + a correctness upgrade).**

## PART B — NET-NEW transpositions opened by the hand-off perf wave

### N1 [P1] `lerpArray` (D2 SoA primitive) has ZERO production consumers

`src/math.ts:48` `lerpArray(start, stop, t, out)` was shipped at `c920c9b`
(Tranche-F D2) as "the carrier primitive a numeric-animation SoA substrate
adopts." At HEAD it is referenced **only** by:
- `src/index.ts:175` (re-export)
- `test/tranche-f.test.ts:15,283,299` (its own test)
- `bench/numeric-soa.mjs` (its own benchmark)

**No `src/` or `demo/` site calls it.** It is an orphaned primitive — exported,
tested, benched, but never wired into an interpolation path. The most natural
consumer would be `lerpColorValue`'s hot loop (which already builds
`startN`/`stopN` as `Float64Array` in `buildColorChannelPlan`,
`interpolate.ts:244-245`) — but that loop **cannot** use `lerpArray` as-is
because of (i) the hue channel's `interpolateHue(…)*360` special-case
(`interpolate.ts:126-129`) and (ii) the heterogeneous per-channel destination
write (`cur.value =` vs `setChannel`, `:131-132`). So `lerpArray` is a primitive
built for a substrate that was never built. Either (a) a future numeric
multi-channel path (transforms — see N2) adopts it, or (b) it is honestly an
orphan and the elegance move is to acknowledge it as a measured-but-unconsumed
carrier (the bench even documents it's SLOWER at K=1). **Flagging as a
transposition gap, not a bug:** the SoA idiom did NOT generalize across the
interpolation paths the way the commit message implied — the color path is the
only multi-channel numeric path and it can't use the primitive.

### N2 [P2] `interpolateDecomposed` — AoS scalar lerps + a local `lerp` shadow; the unexploited SoA site

`src/transform/decompose.ts:510-541` does **13 scalar lerps** in AoS tuple style
(translate ×3, scale ×3, skew ×3, perspective ×4) and defines a **local `lerp`
shadow** at `:515` (`const lerp = (v0,v1) => v0+(v1-v0)*t`) that duplicates
`math.ts:lerp` (curried over `t`). This is the *one* genuine multi-channel numeric
interpolation site in `src/` besides color — i.e. the natural home for the D2 SoA
idiom (N1). BUT: it is a **one-shot per matrix-interp**, not a per-frame hot loop,
so the SoA win is marginal (the bench shows `lerpArray` bites only on repeated
calls amortizing the typed-array setup). The honest elegance fix here is small:
**delete the local `lerp` shadow, import the canonical one** (the curry can stay a
2-line wrapper or the call sites pass `t`). The SoA rewrite is NOT worth it for a
one-shot. Elegance-win-per-LoC: low (−1 shadow def), but it removes a
DRY violation the perf wave's `lerp`-canonicalization discipline (D.W3 Lane C L11)
would otherwise flag.

### N3 [P2] `useMixingAnimation` local `lerp` shadow — demo-side DRY violation

`demo/@/components/custom/mix/composables/useMixingAnimation.ts:55`

Trivial (−3 LoC), and it is exactly the kind of shadow the hand-off wave's

### N4 — computed-endpoint cache (C1) idiom: ONE-OFF, correctly scoped (NOT a missed generalization)

`lerpComputedValue`'s `_computedCache` (interpolate.ts:38-67) caches the resolved
`(startN, stopN, unit)` keyed on `(target, epoch)`. I checked whether the same
"cache invariant-resolution keyed on layout epoch" idiom generalizes elsewhere:
it does not — it is specific to computed leaves (`var`/`calc`/`vh`…) that
re-resolve against the live DOM box every frame. The color plan (B3) and numeric
paths have **no** per-frame DOM resolution, so there is nothing to cache. The
two perf idioms (frozen `_colorPlan` SoA for color; `_computedCache` for computed)
are correctly **disjoint** — they cover the two distinct hot paths and don't
overlap. **No transposition here; the cache is appropriately a one-off.** This is
a NEGATIVE finding (the wave did NOT leave a collapsible duplication between the
plan idiom and the cache idiom — they're orthogonal by construction).

### N5 — `prepareInterpVar` `_lerp` predispatch: clean, no collapse available

`interpolate.ts:217-227` predispatches `_lerp` to one of
`lerpComputedValue`/`lerpColorValue`/`lerpNumericValue`. The fallback runtime
dispatch in `lerpValue` (`:191-207`) mirrors it. This is the textbook
predispatch-vs-fallback pair; no collapse without losing the externally-constructed
iv escape hatch. **No finding.**
