## 1. P0 / P1 — the pane does not render (two faces of one glass-ui tabs refactor)

### [P1] MOUNT BLOCKER — broken glass-ui CSS `@import` 500s the whole demo
The demo's `demo/@/styles/style.css:23` does `@import "@mkbabb/glass-ui/styles"`, which resolves (per glass-ui `package.json` exports) to `dist/styles/index.css`. That file at **line 148** does:

```css
@import "../components/custom/tabs/segmented-tabs.css";
```

…and **that file does not exist** anywhere in the glass-ui dist (`find dist -name '*segmented*'` → nothing; the build dropped/mis-named it). Vite's postcss pipeline throws `ENOENT … segmented-tabs.css` → HTTP 500 → `#app` innerHTML = 43 chars, empty `<body>` (my Playwright probe: `APP_MOUNT {"len":43,"bodyText":""}`). This is **not a MIX-pane defect** — it blanks the entire app — but the MIX pane is the lane, and this is what a user sees on the `/#/mix` route. Tagged **P1** (not P0) because it is an *upstream glass-ui dist* bug on a stale lineage (`tranche/AY`/3.10.1; registry is 3.12.0; next cut 3.13.0), not a value.js source bug — value.js merely consumes the broken barrel. Gestalt fix: value.js cannot fix it locally; the dependency must ship a dist whose `index.css` barrel only imports files it actually emits (glass-ui 3.13.0). **Until then the demo is un-runnable from a clean CSS pipeline** regardless of which pane.

### [P1] LATENT IMPORT BREAK — `MixSourceSelector.vue:4` imports the deleted `BouncyTabs`
```ts
import { BouncyTabs } from "@mkbabb/glass-ui/tabs";   // MixSourceSelector.vue:4
```
glass-ui `dist/components/custom/tabs/index.d.ts` exports **`SegmentedTabs`**, not `BouncyTabs` (renamed). The segmented control is the **first element** of the MIX source selector (template line 100, `<BouncyTabs variant="pill" …>`). This is the demo-side half of the same tabs refactor — it is the `typecheck` RED the brief names, and it would be a runtime `undefined component` even if the CSS resolved. The good news: `variant="pill"` still exists on `SegmentedTabsProps` (`SegmentedTabsVariant = "segmented" | "pill" | "underline"`), and `:options`/`:model-value`/`@update:model-value` are unchanged — so the **migration is a pure rename** (`BouncyTabs`→`SegmentedTabs`, both at import line 4 and template line 100). Gestalt fix: rename at the consumer (no shim — per `feedback_no_backwards_compat`), pinned to glass-ui ≥ 3.13.0 where the CSS also resolves. **Both P1s are one anchor:** they unblock together when the demo re-pins glass-ui.

> Note: these two are the *same* glass-ui tabs refactor — `BouncyTabs`→`SegmentedTabs` renamed the component AND `bouncy-tabs.css`→`segmented-tabs.css`, but the dist barrel's `@import` was left pointing at the new name while the file was not emitted. value.js is the blameless downstream.

## 2. P2 — phantom design classes (intended styling silently absent)

These are genuine *value.js demo* defects, independent of glass-ui, found by tracing every class to a definition.

### [P2] `glass-elevated` is undefined — the Result card has NO elevated glass surface
`MixResultDisplay.vue:31`:
```html
<div class="flex flex-col gap-3 p-4 rounded-xl glass-elevated">
```
`glass-elevated` is defined **nowhere** — not in `demo/@/styles/*`, not in glass-ui `dist` *or* `src`, not in `node_modules`. (`find … -name '*.css' -exec grep -l glass-elevated` → empty across all three trees.) So the **Result card** — the payoff of the whole pane — renders as a flat `rounded-xl` box with `p-4`, **no glass material, no elevation, no border, no backdrop-blur**. Against the house language ("glass surfaces"), the single most important surface in the pane is the one with no glass. This is the only `glass-elevated` callsite in the repo, so it reads as a typo for an intended glass utility (e.g. glass-ui's `glass` / a `Card tier=…`). **Gestalt fix:** make the Result a real glass surface — wrap it in `<Card>` with a glass `tier` (the same primitive the pane shell already uses at `MixPane.vue:61`), not a hand-rolled div with a non-existent class. This is a one-event "the result arrives" moment and deserves the pane's strongest material, elevated above the `tier="wash"` shell.

### [P2] `dashed-well` is undefined — the Selected-colors container has NO well treatment
`MixSourceSelector.vue:111`:
```html
<div class="dashed-well">   <!-- wraps the Selected header + swatch row + add button -->
```
`dashed-well` is also **undefined everywhere** (same exhaustive search). It is used in exactly two places — here and `CurrentPaletteEditor.vue:3` — and defined in neither, so **both** the mix selected-swatches area and the palette editor's add-swatch area render as bare `<div>`s with **no dashed border, no inset padding, no recessed fill**. The intended affordance (a dashed "drop a color here" well that frames the `+` add button) is invisible; the `+` button's own `border-dashed border-primary/30` is the *only* dashed cue, now floating with no container. **Gestalt fix:** define one shared `.dashed-well` utility (dashed `--border` ring, `border-radius: var(--radius-card)`, muted inset bg, `padding`) in `demo/@/styles/utils.css` next to the existing `.section-subtitle` recipe, and let both consumers share it — it is plainly meant to be a system utility (two consumers, zero defs).

### [P2] Fragile cross-component dependency for the swatch animation
`MixSourceSelector.vue` (line 116) and `MixResultDisplay.vue` (line 53) use `<TransitionGroup name="swatch-item">`, but **none of the mix components have a `<style>` block** (verified: 0 across all 5 mix SFCs + the shell). The `.swatch-item-*` transition CSS lives **only** in `CurrentPaletteEditor.vue:250` in an *unscoped* `<style>` block. So the MIX swatch add/remove/move animation **works only by accident** — it silently inherits a global rule authored for, and owned by, an unrelated palette-editor component. If `CurrentPaletteEditor` is ever scoped, deleted, or refactored, the MIX pane's motion dies with no compile error. (The `.pop` reveal transition at `MixResultDisplay.vue:35` / `MixPane.vue:107` is fine — it resolves to glass-ui's `transitions.css:63` `.pop-enter-active`, a `scale(0.9)` `--spring-bouncy` pop — *on-brand purposeful motion*, but **only when the broken `index.css` barrel loads**, since `.pop` ships in the same `dist/styles` tree gated behind the missing import.) **Gestalt fix:** lift `.swatch-item-*` to a shared global stylesheet (`demo/@/styles/animations.css`) so every TransitionGroup consumer owns it equally, rather than one component being the de-facto provider for the tree.

## 3. Design-language scorecard (intended design, from source + resolved tokens)

Where the markup *would* render correctly (glass-ui repaired, classes defined), the MIX pane is **well-composed**. Scored against the house language: glass / grid / math motifs / audacious type / colorful pops / purposeful motion.

### Design hierarchy — **GOOD**
- Clear top-down read: `PaneHeader` ("Mix" `text-heading` 1.618rem/700 + italic caption desc) → segmented mode switch → Selected well → config grid → Mix CTA → animated indicator → Result. (`MixPane.vue` template 59-117.)
- The scroll-linked `PaneHeader` shrink (`pane-header-shrink`/`pane-title-shrink` keyframes, `--pane-scroll` timeline) is a sophisticated, on-brand motif shared across panes — the title collapses `--type-heading`→`--type-prose` on scroll. Strong.
- **Weakness:** the *Result* — the climax — is the **least** materially emphasized surface because of the `glass-elevated` phantom (§2). Hierarchy intent is right; execution inverts it.

### Spacing rhythm — **GOOD**
- Consistent `gap-3`/`gap-4` vertical rhythm (`MixPane.vue:71` `gap-4`; sub-components `gap-3`). Config grid `grid grid-cols-2 gap-2` (`MixConfigBar.vue:49`) is tidy. Swatch row `gap-2.5 flex-wrap` reads. Header pad `px-4 sm:px-6` matches the pane convention.
- **Minor:** swatch sizes step `w-11 h-11 sm:w-12` (selected, MixSourceSelector:129), `w-10 h-10` (result swatches, MixResultDisplay:63), `w-14 h-14` (result single, :41), `w-8 h-8` (palette-dropdown swatches, :185). Four swatch scales is one too many; the result-single 56px vs result-palette 40px vs selected 44/48px lacks a single modular ratio. Not a defect, a tightening opportunity.

### Typography scale coherence — **GOOD (one register collision)**
- Resolved scale is coherent: `text-heading` (1.618rem/700, the φ ratio — a deliberate math motif), `text-caption` (clamp .75→1rem, italic), `section-label` (mono caps, `--type-caption`, `--type-tracking-caps`), `text-mono-small` (Fira Code, `--type-small`), `text-micro` (.6875rem). Display = Fraunces, mono = Fira Code — house pairing intact.
- **Collision:** `MixSourceSelector.vue:113` labels "Selected" with `text-small font-display font-semibold` while the symmetric `section-label` (mono caps) labels "From palettes" (:156) and "Color space"/"Hue method" (`MixConfigBar.vue:51,69`). Two different "section label" registers inside one pane — a Fraunces semibold small vs a Fira mono caps caption — for the same role (a sub-section header). Pick one. **Gestalt fix:** make "Selected" a `section-label` too, so all four sub-section eyebrows share the one mono-caps register the design system defines for exactly this role.

### Color & contrast — **GOOD, but under-audacious**
- The pane is correctly *content-colorful* (the user's swatches are the color), with neutral chrome — the right instinct. `WatercolorDot` (organic blob) for every swatch is the house iconography motif; the gradient-strip result preview (`MixResultDisplay.vue:70`, `linear-gradient` of result colors) is a nice colorful payoff.
- **But** the chrome itself has **no audacious pop** — no gold/violet section-accent eyebrow (glass-ui ships `.section-label--tinted` / `--section-color-7` violet anchor, unused here), no colorful CTA. The Mix button is a default `<Button>` (`MixConfigBar.vue:103`) in neutral `--primary`. The house language wants "colorful audacious pops"; the MIX chrome is monochrome-neutral end to end. **Gestalt fix:** tint the four section eyebrows with the system's `--section-label-accent` (one chroma, page-scoped per glass-ui's one-accent rule), and/or give the Mix CTA a branded fill — the "Mix" action is the pane's one verb and should pop.

### Motion — **GOOD where it lands, gated where it counts**
- `useMixingAnimation.ts` is genuinely ambitious *purposeful* motion: per-color sections grow from random corners (`gathering`), suffuse to the averaged color (`mixing`), shrink to a center swatch (`revealing`) — canvas with `blur(16px) saturate(120%)`, `easeOutCubic`/`easeInQuad`, staggered entry. This is exactly the "purposeful animation" the house wants, and it visualizes the *mathematical* act of mixing. Strong, on-brand.
- The phase indicator (`MixPane.vue:96` spinner + "Gathering…/Mixing…/Revealing…") is a clean status read.
- **Caveats:** (1) the canvas `mixing` overlay averages colors in **sRGB** (`useMixingAnimation.ts:169-171` arithmetic mean of RGB) while the *actual* result mixes in OKLab/OKLCh (`useMixingState.ts:80` `mixColorsN(…, colorSpace …)`) — the animation's preview color can visibly diverge from the revealed result, a small honesty gap in an otherwise lovely effect. (2) Hard-coded `setTimeout` phase schedule (800/2300/2900ms, `useMixingState.ts:84-89`) — no `prefers-reduced-motion` gate on the canvas RAF loop (the MEMORY notes a repo-wide ungated-RAF concern). (3) The `.pop` result reveal is good but rides the broken glass-ui barrel (§2).

## 4. Functional notes (not design, but observed in source)

- **N-color mixing is real and rich:** `useMixingState.ts` supports 2–12 colors (`MAX_COLORS = 12`, MixSourceSelector:39), 9 interpolation spaces (OKLCh/OKLab/Lab/LCh/HSL/HSV/HWB/RGB/XYZ — `useGradientInterpolation.ts:16-25`) and 4 hue arcs (shorter/longer/increasing/decreasing). Palette-mode adds a leftover strategy (discard/repeat/distribute, MixConfigBar:38). The *capability* is the strongest part of the pane.
- Each space/hue Select shows a `section-subtitle` live description (MixConfigBar:52,70) — a thoughtful affordance.
- Result actions reuse `DockIconButton` (copy/save/reset, MixResultDisplay:82-103) — consistent with the dock vocabulary.
- No backend in this env → "From palettes" / palette-mode are empty (expected, not a defect): `savedPalettes` is `[]` so `PaletteCardSkeleton` shows in palette mode (MixSourceSelector:200).
