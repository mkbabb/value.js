# Lane B3 — Release-train quality audit (Tranche-F hand-off commits)

**Scope:** adversarial deep-review of the 11 hand-off commits that compose the
value.js 0.11.0 → 0.11.2 release train. Read-only; every claim verified against
the live tree + executed tests, not the commit prose.

**Date:** 2026-06-11 · branch `tranche-f-handoff` @ HEAD `0cb5dd2` ·
published 0.11.2.

**Bottom line:** the train is **high quality and lands clean** — all 22
tranche-f + computed-cache tests pass, 199 snapshot/color/stylesheet tests pass,
eslint exits 0 on every touched src file, **zero new type escapes** (the 2
`as unknown as` in touched src are the pre-existing CLAUDE.md-documented
irreducibles), and the new public exports are semver-honest (additive, landed in
the 0.11.0 **minor**). The perf claims are plausible and gated by real
measure-first benches. The substantive risks are concentrated in the
**computed-endpoint cache (C1/C2/C7)** — an inherited var()-mutation staleness
gap the C7 epoch model does not close, and an unbounded memo-growth regression
from the C2 key change — plus two narrow correctness edges (B1b percent-alpha,
F7 custom-shadows-builtin) that the test suite does not cover.

## Per-commit verdicts

### e684bce — A2 maximal-munch unit classifier  ✅ SOLID, real latent fix

`any(...UNITS.map(istring))` → `unitParser(units)`: one `[a-zA-Z]+` maximal
munch validated against a `Map<lowercased, canonical>`. The bug it fixes is
genuine: parse-that re-flags `istring`'s RegExp sticky-`y`, anchoring match
START but not END, so `100vhx` tokenized `vh` and dropped `x`, and prefix units
shadowed later-declared ones by declaration-order accident.

- **Correctness:** verified live — `parseResult(CSSValueUnit.Length, "100vhx")`
  now rejects; `vmin`/`vmax`/`vb`/`vi`/`svw`/`dvh`/`lvmax` each classify to their
  full unit. Isomorphism over the full `RELATIVE_LENGTH_UNITS` set holds.
- **Idiomatic fit:** good — `unitParser` lives in `parsing/utils.ts` next to
  `istring`; canonical-spelling return keeps output byte-identical. `%` correctly
  retained on `istring` (single non-identifier glyph, no boundary hazard).
- **Coverage:** strong — A2 block asserts trailing-junk rejection across length
  AND non-length families (time/angle/ms), longest-match, and full-set
  isomorphism.
- **Caveat (P2):** the maximal-munch token is `[a-zA-Z]+` only — a unit family
  with digits (none today) would need the class widened; safe but fragile if a
  digit-bearing unit is ever added. No public-API impact.

### 8383bd8 — C5 resolve the 24 no-op relative length units  ✅ SOLID

Of 38 relative length units, 24 (`vi`/`vb`, the 18 `sv*`/`lv*`/`dv*`, `cap`/
`ic`/`lh`/`rlh`) fell through to `convertAbsoluteUnitToPixels` → silent identity
(`50dvh` → `50px`). New `convertViewportUnitToPixels` +
`convertFontMetricUnitToPixels` + a **fail-loud guard**.

- **Correctness:** verified — the C5 test stubs viewport geometry; `50dvh`/
  `50svh`/`50lvh`/`50vb` = 400px against 800h, width-family = 500px against
  1000w. No double-handling: `vh`/`vw`/`vmin`/`vmax` are caught by their
  explicit `else if` branches **before** the C5 `else` block, so the
  `convertViewportUnitToPixels` regex (which also matches them) never fires for
  them — confirmed by reading the live `convertToPixels` (utils.ts:414-421 vs
  the `else` at :478).
- **Fail-loud guard design (P2, defensible):** a *declared relative* unit with
  no resolution branch now **throws** rather than no-ops. This is a deliberate
  trip-wire for future table additions, but it converts a previously-silent
  degradation into a **runtime throw** — a behavior change for any consumer that
  passed an exotic-but-declared unit with no DOM context. Mitigated: handled
  units lacking context (`em` with no element, viewport unit with no `window`)
  still degrade gracefully; only genuinely-unhandled declared units throw.
- **Approximations (P2, documented):** `cap ≈ 0.7em`, `ic ≈ 1em`,
  `normal` line-height ≈ `1.2 × fontSize`. These are heuristics, not true font
  metrics — acceptable and clearly commented, but `lh`/`cap`/`ic` resolutions
  are approximate by design.
- **Coverage:** good for viewport (the observable family); font-metric units
  (`cap`/`ic`/`lh`/`rlh`) are **not** asserted in the C5 test (they need
  `getComputedStyle` font metrics jsdom doesn't deliver meaningfully) — the
  fail-loud guard is the only thing locking them, untested on value.

### 2b81be1 — B1b formatColor omits `/ alpha` at alpha=1  ⚠️ INCOMPLETE canonicalization

`formatColor` now emits the alpha clause only when `Number(alpha) !== 1`.

- **Correctness:** the common case works — opaque `oklab`/`rgb` drop `/ 1`,
  non-opaque keep, round-trips back to alpha=1. 27 snapshots + 4 color asserts
  updated, each a clean clause removal.
- **FINDING (P2) — percent alpha is not canonicalized:** `Number("100%")` is
  `NaN ≠ 1`, so `parseCSSColor("rgb(255 0 0 / 100%)").toString()` keeps the
  clause → `rgb(255 0 0 / 100%)` (verified live). CSS Color 4 treats `100%`
  alpha as opaque too; the commit's "opaque omits the clause" claim only holds
  for numeric `1`. Whether this leaks depends on whether the pipeline ever
  produces `100%` alpha internally (parsed-then-serialized percent alpha does).
- **Adjacent pre-existing (P2, NOT introduced here):** `rgb(255 0 0 / none)`
  serializes to `rgb(255 0 0 / NaN)` — invalid CSS. B1b is iso for this case
  (NaN ≠ 1, clause kept). Confirmed pre-existing by reading `2b81be1~1`. The
  `none`→`NaN` serialization is a separate alpha-pipeline bug worth a future
  ticket.

### eeff123 — A1 O(1) first-char dispatch  ✅ SOLID, byte-identical

14-way speculative `any()` → `dispatch(table)` (one `Int8Array(128)` charCode
index). Verified the parse-that `dispatch` impl (parse.js:520-558): `"0-9"` is
correctly interpreted as a range (`chars[1]==="-"`), and `.trim(whitespace)`
trims leading WS via `trimStateWhitespace` **before** dispatch reads
`charCodeAt(offset)` (parse.js:1140-1143) — so dispatch sees the first
non-whitespace char, exactly as the old `any().trim()` did.

- **Correctness:** verified iso — uppercase `RGB(...)`, `Red`, `REBECCAPURPLE`,
  `transparent`, `currentcolor` all resolve; every letter bucket retains a
  trailing `nameParser`, non-discriminating letters default to `nameParser`. A
  bare number routed to the kelvin bucket (`#`/digit/`+`/`-`/`.`) fails the
  kelvin parse and throws — iso with the old `any()` (number is not a color).
  128 parser snapshots + color-classes stay green.
- **Perf claim:** 2.42× median (gate ≥2×) over real parse-that combinators in
  `bench/color-dispatch.mjs`. Plausible — eliminating up to 13 failed
  `mergeErrorState` descents per parse is exactly what dispatch saves.
- **P2 — residual diagnostic leak:** a failed color parse still triggers
  parse-that's `console.error` diagnostic spew (observed in the live REPL for
  `parseCSSColor("100")`). A1 doesn't touch this; F7 fixes only the custom-name
  path. The broader diagnostic-sink fix is correctly deferred to parse-that.

### 652aa71 — F7 custom-name map before speculative parse  ⚠️ undocumented semantic change

Reorders `parseCSSColor` to try `customColorNames.get(key)` first when the
registry is non-empty, eliding a per-parse `console.error` leak.

- **Correctness:** the leak elision works — verified zero `console.error` on a
  registered custom name with diagnostics on. When the registry is empty the
  branch is a no-op (built-ins resolve via the rich parser unchanged).
- **FINDING (P2) — custom now shadows built-in:** registering `red → #00ff00`
  then parsing `red` returns **green** (`rgb(0 255 0)`, verified live).
  Pre-F7 the rich parser ran first, so a built-in always won and a colliding
  custom name was a dead entry. F7 inverts this: custom-shadows-builtin. This is
  a defensible design, but it is an **undocumented behavior change** in a `fix`
  commit — the "iso" claim is scoped only to *no custom names registered*, and
  the test (`mybrandcolor`) covers only a **non-colliding** name. The collision
  case is untested.

### c17263d — F4 @property lossless round-trip test  ✅ test-only, correct

Pure test addition (no src). Locks that `buildPropertyDescriptor` stores the raw
quote-stripped author syntax (not a re-serialized AST), so `+`/`#` multipliers,
`|` unions, and `*` survive byte-exact. 23 stylesheet tests pass. Honest
"VERIFIED-NO-CHANGE" framing; the `{n,m}`-trips-the-scanner observation is
correctly scoped out (value-definition syntax, not @property grammar).

### c868f3d — B3+B5 frozen color-channel plan  ✅ SOLID, good SoA discipline

`prepareInterpVar` freezes a `ColorChannelPlan` (SoA `Float64Array`
startN/stopN + hueIndex + per-channel dst `ValueUnit` refs); the per-frame loop
is closure-/`keys()`-/`unwrapDeep`-free. B5 pre-divides hue ÷360 into the plan.

- **Correctness:** the B3 test asserts plan == fallback within 1e-12 across
  oklch/oklab/hsl at 51 t-samples each — passes. Externally-constructed (no-plan)
  ivs keep the byte-identical fallback walk.
- **Cache-hazard (P2):** the plan caches `keys`, `startN`/`stopN`, and `dstVU`
  refs into `value.value`'s channels **once**. If a consumer swaps `iv.value`'s
  Color object or re-points endpoints after `prepareInterpVar`, the plan's
  `dstVU` refs go stale and the hue ÷360 pre-divide is wrong for the new
  endpoints — no invalidation path exists (unlike C1's epoch/target check). Safe
  under the documented contract (endpoints fixed for the iv's life), but the
  contract is enforced by convention, not by the type system. The memory note
  ("`Color.clone()`'s depth-counter is off the per-frame path") is verified — the
  plan mutates in place, never clones.
- **Perf claim:** 3.93× median (3.08–4.49×) over `dist/value.js` in
  `bench/color-interp.mjs` — apples-to-apples (same engine, plan deleted to force
  fallback). Plausible; color interp is the named ~40×-a-numeric-prop hot lane.

### c920c9b — D2 SoA lerpArray primitive  ✅ SOLID, honest measure-first

New `lerpArray(start, stop, t, out)` flat loop over `Float64Array`s.

- **Correctness:** strict-equality to K independent scalar `lerp()` across
  K∈{1,2,5,16,64} × 21 t-samples; returns `out`, inputs untouched. Passes.
- **Perf honesty:** the commit ships it as a primitive and **openly states it is
  SLOWER at K=1 (0.61×)** and only bites from K≥2 — measure-first vindicated, D1
  monomorphization explicitly NOT shipped as a non-win. This is exemplary.
- **Public API:** new export `lerpArray` (additive, minor). The primitive has no
  length-mismatch guard — `start`/`stop`/`out` must share length, documented but
  unchecked; a shorter `out` would silently truncate. P2, by-design.

### 5c947de — C1+C2+C4+C7 computed-endpoint cache  ⚠️ HIGHEST-RISK lane

The largest change. Four sub-levers; the endpoint cache is the focus of this
lane's cache-hazard mandate.

**C1 (endpoint memo on the iv):** caches resolved `(startN, stopN, unit, target,
epoch)` on `iv._computedCache`; steady frame collapses to a bare `lerp`.
- **Key shape / mutation safety:** invalidation keys are **only** `(target,
  epoch)`. Verified the 6 cache-bust gates pass (setTargets bust, resize bust,
  pixel-identity, O(1)-resolve-count, C2-no-toString, C4-no-Date.now).
- **FINDING (P1) — var() mutation staleness the C7 epoch model does NOT close:**
  built a focused test — warm the cache (`--stop=100px` → mid=50), then mutate
  `--stop` to `300px` **without** a resize (no epoch bump), re-lerp → **50, not
  150** (stale). A custom-property reassignment mid-animation (theme switch,
  JS-driven var) is not a "layout" event and does not bump the epoch, so C1
  serves pre-mutation pixels for the rest of the epoch. **Nuance (keeps this
  P1 not P0):** the pre-C1 `getComputedValue` memo *already* served stale var()
  reads (verified: returns 100 after mutation to 300 with no cache clear), so
  this is an **inherited limitation C1 moves up a level, not a fresh
  regression** — but the C7 "epoch = resize" invalidation story is presented as
  the complete answer and silently excludes var-mutation. Consumers driving vars
  imperatively must call `bumpLayoutEpoch()` themselves; this is undocumented in
  the export comment (which says only "resize").

**C2 (stable-id memo key):** keys `getComputedValue` on a per-instance monotonic
`WeakMap` id instead of `value.toString()`.
- **FINDING (P1) — unbounded memo growth regression:** `getComputedValue` is
  `memoize(...)` with **no `maxCacheSize`** (defaults to `Infinity`,
  utils.ts:114) and **no TTL**. Pre-C2, distinct-but-content-equal ValueUnits
  (e.g. many leaves all `var(--x)`) collapsed to ONE cache entry by content.
  Post-C2 each distinct ValueUnit *instance* gets a **unique** key, so N
  short-lived computed ValueUnits leave N permanent entries. The `WeakMap` lets
  the *ValueUnit* be GC'd, but the **string-keyed Map entry persists forever**
  (the Map holds the string id, not the ValueUnit). Apps that re-create
  interpolation vars per animation cycle leak one entry per leaf per cycle, only
  ever cleared by `bumpLayoutEpoch()` (resize). This is a real
  memory-growth-over-time hazard for long-lived SPAs.
- **Stale-doc (P2):** `getComputedValue`'s docstring (normalize.ts:197) still
  says "Memoised by `(value.toString(), element-id)`" — wrong after C2.

**C4 (ttl===Infinity fast path):** hoists `hasTtl` once, skips `Date.now()` on
the hit path. Correct; eviction order preserved (`cache.keys().next()` FIFO
unchanged). No hazard.

**C7 (layout epoch):** process-global counter + `bumpLayoutEpoch`/`getLayoutEpoch`
(new exports), auto-installed on `window.resize`.
- **P2 — global resize listener never removed:** `installResizeEpoch()` binds a
  permanent `window` resize listener at module load (guarded against double-bind
  and SSR). No teardown — acceptable for a singleton library module, but it is a
  permanent global side-effect on import.
- `bumpLayoutEpoch` clears `getComputedValue.cache` wholesale (the only relief
  for the C2 unbounded-growth + the var-staleness) and stamps C1 lazily via the
  epoch compare. Correct ordering (the `const getComputedValue` is referenced
  only inside the runtime callback, after module eval).

**Perf claim:** steady leaf 107.4 → 6.5 ns/frame (−94%); the call-counter test
proves O(frames)→O(1) resolve collapse directly — the strongest gate of the
train.

### fbea3e2 — parseCSSValueUnit typed-empty contract  ✅ SOLID

Empty/whitespace/null input → `new ValueUnit(0)` (no unit, no-op interpolation
contribution) instead of the `tryParse` `'......'` throw. Verified live: `""`,
`"   "`, and `null` all return `{value: 0}`.

- **Correctness:** the guard `input == null || input.trim() === ""` runs
  **inside** the memoized fn, so the empty result is cached under key `""` (the
  `keyFn` is identity) — fine, it's a constant. Heals the unset-`var(--x)`
  read-back seam at one source. The kf I.W0 dependency that drove the 0.11.2
  release.
- **P2 — shared mutable identity:** the empty path returns a **fresh**
  `new ValueUnit(0)` each cache-miss but the memo then caches that instance;
  every later `parseCSSValueUnit("")` returns the **same** ValueUnit object. If
  any consumer mutates the returned empty ValueUnit in place, it poisons all
  later empty reads. Low risk (empty is an identity sentinel) but shared-mutable.

## Cross-cutting assessments

**Public-API / semver honesty — ✅ HONEST.** New runtime exports across the
train: `lerpArray`, `bumpLayoutEpoch`, `getLayoutEpoch` + type exports
`ColorChannelPlan`, `ComputedEndpointCache`. All landed in the **0.11.0 minor**
bump (from 0.10.x) — additive features correctly warrant minor, not patch.
0.11.1 (drop broken `development` export) and 0.11.2 (empty contract) are pure
fixes, no new exports. Semver discipline is correct.

**Type escapes — ✅ CLEAN.** Zero new `as any`/`as unknown as` in touched src.
The 2 `as unknown as` (normalize.ts:137 CSSStyleDeclaration, color.ts:59
clone-reinterpret) are the pre-existing CLAUDE.md-documented irreducibles. The
5 `as never` are confined to the test file (loose `_colorPlan`/options typing).
C1's `newStart.value as number` (interpolate.ts:59) is a narrowing assertion on
an `any`-typed iv — acceptable but unchecked (a non-numeric computed resolve
would cache `NaN`).

**Test coverage — STRONG where it counts, GAPS at the edges.** The bite-tests
(revert → red) are exemplary, esp. the C1 resolve-call-counter. Gaps: C5
font-metric units (`cap`/`ic`/`lh`/`rlh`) value-untested; B1b percent-alpha
uncovered; F7 custom-shadows-builtin collision uncovered; the C1 var-mutation
staleness uncovered (it would have caught the gap).

**Perf-claim plausibility — ✅ ALL PLAUSIBLE + measure-first.** Every perf
commit ships a hermetic bench with an explicit gate; D2 honestly reports a
K=1 slowdown and withholds D1. No inflated numbers detected.

**Lint / typecheck — ✅** eslint exits 0 on all 10 touched src files. (The 3 RED
typecheck errors named in the brief are demo-side glass-ui API skew, unrelated
to this train.)
