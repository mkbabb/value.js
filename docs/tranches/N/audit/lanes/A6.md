### 1.3 Merge strategy recommendation

The canonical close ceremony (per `docs/RELEASE.md §3.2`) uses `git merge --no-ff tranche-X` to preserve tranche history. The current state is unusual: this branch contains **three tranches' worth of docs (K/L/M), a patch release train (0.11.0..0.11.2), and multiple CI fixes** — not a single clean tranche. This warrants a deliberate sequencing decision at merge time:

1. **Pre-merge prerequisite**: the three uncommitted working-tree deletions (CHANGELOG.md, CONTRIBUTING.md, VENDOR-POLICY.md) must be committed and staged before merge. Currently they are **unstaged deletions** — the files exist in HEAD but are absent from the working tree. If the merge fires without committing them, the merge commit will carry the files back in and the deletions will be lost. See §3.
2. **Merge timing**: this branch should merge to master at the **next natural close** (when tranche N closes or when the N pre-merge gate matrix is satisfied). There is no reason to hold it longer — the CI-unblocking fix (`b4defb0`) is urgently needed because **master CI is currently RED** (see §4.1).
3. **No rebase**: the release tags `v0.11.0` and `v0.11.1` are annotated tags on commits in this branch (`e8cc1fb` and `b4defb0` respectively). Rebasing would detach those tags from their commits. Merge with `--no-ff` only.
