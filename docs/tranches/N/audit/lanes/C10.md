## (b) LIFT — demo facilities befitting glass-ui (refine extant components, don't invent)

- **Demo (fork):** `demo/@/components/custom/dock/composables/useLayerTransition.ts` (123) — its own
  header (lines 1-5) states it's a *"Local fork … with the legacy `layerProps()` API the consumer
  (ActionBarLayer.vue) relies on. Upstream glass-ui retains the crossfade primitive but exposes only
  `currentLayer` / `leavingLayer` refs + `onTransitionEnd`; the per-id class + inert helper is a thin
  convenience that lives here."*
- **glass-ui (shipped):** `useLayerTransition` IS re-exported from `/dock`
  (`src/components/custom/dock/index.ts` `export { useLayerTransition } from "./composables"`), but the
  upstream version (276 LoC, AX.W01 single-`SpringProgress` morph,
  `src/components/custom/dock/composables/useLayerTransition.ts`) exposes raw refs, not the
  `layerProps(id){ class, inert }` convenience. **LIFT target:** add the `layerProps()` (per-id class +
  `inert` helper) onto glass-ui's `useLayerTransition` return so the demo can drop its fork and adopt
  the shipped primitive. Reuses the existing composable — no new surface. (The glass-ui barrel comment
  already anticipates this: "fork ONLY because this barrel never re-exported the primitive.")

### L2 [P2] `usePopupMutex` — single-open mutex for dock popups (was retired upstream; reconsider)
- **Demo (fork):** `demo/@/components/custom/dock/composables/usePopupMutex.ts` (85) — header:
  *"`usePopupMutex` was retired upstream from glass-ui at the D-II tranche. Local fork — single-open
  mutex for dock popups with a brief swap delay."* A clean, generic `<K extends string>` mutex
  (`current`, `isAnyOpen`, `popupModel(key)`).
- **glass-ui:** no current equivalent (retired). This is a genuinely reusable dock-chrome primitive
  (single-open popover coordination is a design-system concern). **LIFT** back into `/dock` composables
  IF the dock-rail redesign (AZ, active — see §c) wants single-open semantics; otherwise BOOK as a
  demo-local that proved its keep. Lower confidence — verify against AZ's rail direction before lifting.

## (d) MECHANICS — how the demo resolves glass-ui today + the idiomatic target

### M1 [P1] Resolution today: `file:` symlink + `dist`-resolution (NOT source-resolution)
- **Declaration:** `package.json` `devDependencies["@mkbabb/glass-ui"] = "file:../glass-ui"`. No
  `dependencies` entry, no `overrides`. → `node_modules/@mkbabb/glass-ui` is a symlink
  (`ls -la` → `-> ../../../glass-ui`).
- **value.js self-alias:** `@mkbabb/value.js` bare specifier is aliased to `src/index.ts` in
  `vite.config.ts:37` (a package never installs itself; glass-ui's aurora `/color` path imports
  value.js via the bare specifier, which must point at THIS repo's source). value.js's own `exports`
  map is **precept-clean** — `{types, import, default}`, **0** `development` keys
  (`node -e` on package.json). Matches the brief (clean since 0.11.1).
- **vue/reka dedupe:** `vite.config.ts:49-56` dedupes `vue`, `@vue/runtime-core`,
  `@vue/runtime-dom`, `@vue/reactivity`, `@vue/shared`, `reka-ui` — glass-ui ships nested copies; without
  dedupe the demo loads two Vue instances (broken provide/inject + a Teleport `insertBefore`
  NotFoundError on the dock Select portal).

### M2 [P1] **STALE vite.config narrative** — the `development`-condition story is dead
- `vite.config.ts:60-68` and `:99-109` describe **inv-K-4 / contract-v2** as: *"K.W2 added a
  `development` export condition to glass-ui (→ its live src/); Vite's built-in serve default applies
  `development`, so the demo serve modes source-resolve glass-ui … while build modes (production
  condition) resolve dist/."* **This is now FALSE.** glass-ui's `package.json` has **0** `development`
  keys anywhere (`grep -c '"development"' = 0`); its `.` export is `{types, import, default}` →
  `dist/glass-ui.js` (all subpaths likewise `{types, import}` → `dist/*.js`). The
  cross-repo-dev-resolution precept (`docs/precepts/cross-repo-dev-resolution.md` §2.3, §3, §4)
  **STRUCK** the `development` key (mechanism-C / contract-v2-final: dist-resolution + `build:watch`);
  glass-ui declares `build:watch: vite build --watch` (its `package.json`). So **today the demo
  resolves glass-ui's `dist/` in BOTH serve and build** — the source-resolution mechanism the comments
  describe no longer exists. The comments (and the inv-K-4 references) are documentation drift; the
  freshness guarantee is now `build:watch`, not the `development`→`src/` redirect.
  - Corroboration: glass-ui `dist/glass-ui.js` mtime (Jun 11 15:36) is newer than `src/index.ts`
    (Jun 11 08:55) — a freshly built dist, consistent with dist-resolution.

### M3 [P2] Idiomatic target consumption
- The demo is ALREADY on the precept-conformant mechanism (dist-resolution via the `file:` symlink;
  glass-ui's exports map is 4-key-shape-conformant). The idiomatic target is unchanged from today's
  *runtime* behavior; the only debt is **documentation** (M2 — purge the stale `development`/inv-K-4
  prose from `vite.config.ts` and update any inv-K-4 references to cite contract-v2-final). For local
  cohort dev, the idiom is `npm run build:watch` in the glass-ui checkout while the demo dev server
  resolves the rebuilt `dist/` (HMR on the consumer; rebuild-on-save on the publisher). No
  `resolve.conditions` override is needed and none is present (correct — the precept forbids it).
- **Registry-vs-local skew note:** local glass-ui is `3.10.1`; registry is `3.12.0` (glass-ui judges
  3.11/3.12 stale-lineage, next cut 3.13.0). The demo consumes the LOCAL `file:` checkout (`3.10.1`),
  so the audit's adopt/gap findings are against `3.10.1`. When the demo eventually pins a published
  range, target `3.13.0` (the next clean cut) — the carousel/tabs renames are stable since AX, so the
  migrations (G1/G2) hold across 3.10→3.13.
