### 2.3 Admin mock fixture

**File:** `e2e/smoke/admin/fixtures/admin-auth.ts`  
**Mechanism:** `base.extend` → `page.addInitScript` seeds `localStorage["palette-admin-token"] = "test-admin-token"` before the page's first script runs. `useAdminAuth`'s lazy-init reads this value once, treating the whole session as admin-authenticated.  
**Network mocking:** `page.route("**/sessions", ...)` stubs POST /sessions to return a fake session token; `page.route("**/admin/**", ...)` stubs all admin GETs with shape-correct empty envelopes (`{data: [], total: 0, limit: 50, offset: 0}` or `[]` for tags).  
**Consequence:** Admin tests never hit the live API; they are fully offline-deterministic.

### 2.4 Env-noise fixture

**File:** `e2e/smoke/fixtures/env-noise.ts`  
**Mechanism:** Installs `console` + `pageerror` listeners that filter out HTTP 4xx/5xx rate-limit noise (429/503/504, "Too Many Requests", "Failed to load resource", "blocked by CORS policy") from the env-noise regex. Returns a captured-errors array; callers assert `expect(errors).toEqual([])`.  
**Scope:** Used by the smoke project specs that hit the optional production API surface. Admin specs and mobile spec use their own narrower variants (or route mocks); they do not import this fixture.

## 7. Test Architecture Observations

### 7.1 Vitest include glob is intentionally flat

`test: { include: ["test/*.ts"] }` — the top-level glob only. This means any file placed in a subdirectory of `test/` would be silently ignored. The current tree has no subdirectories, so there is no immediate problem, but it is a latent footgun as the suite grows.

### 7.2 No property-based testing

The `fast-check` library is not present. The random-whitespace loop in `parsing.test.ts` (`Math.random()` with no seed) is not reproducible and cannot shrink on failure. For a CSS parsing library, property-based testing (arbitrary well-formed CSS value strings) would provide much stronger guarantees than the current hand-curated corpus.

### 7.3 BBNF grammars are documentation-only

The `bbnf-equivalence` label on `parser-snapshot.test.ts` is misleading: the `.bbnf` files are never loaded or executed in tests. The grammars remain in `src/parsing/grammars/` as specification documents. The "equivalence" is snapshot-stabilization of the hand-written parsers, not actual grammar execution.

### 7.4 `color-picker-lifecycle.test.ts` is demo-coupled

This test validates lifecycle patterns from `demo/@/components/custom/color-picker/ColorPicker.vue` via a synthetic proxy component. It is the only test touching `@vue/test-utils`. The test couples the library unit suite to a demo UI concern. The patterns tested (keydown listener registration, rAF gating, debounce cancellation) are not part of the public library API and would be better positioned in a `demo/test/` directory.

### 7.5 API suite is serial and slow on cold start

`mongodb-memory-server` downloads the MongoDB binary on first run; with `fileParallelism: false` and `pool: "forks"`, the suite serializes 28 files. CI timeout is 60s per test; the cold-start download can take 2–3 minutes. There is no warm-cache assertion or binary pre-download step in the workflow.

### 7.6 E2E coverage of `smoke` vs `smoke-admin` is asymmetric

The `smoke` project has 21 test functions (20 user-facing flows + WebGL). The `smoke-admin` project has 12. However, admin tests use `page.route` mocks for every XHR — they never exercise actual API behavior. If an admin endpoint changes its request shape, the admin e2e tests will pass while the API breaks in production.
