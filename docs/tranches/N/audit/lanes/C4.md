## Findings

### P1 — Extraction discards population/dominance; the result misrepresents the image (gestalt)

The library's `QuantizedColor` carries `population` (per-cluster pixel count,
`src/quantize/types.ts:41`) and ships a `dominantColor()` export
(`src/quantize/index.ts:176`). **Neither is consumed by the demo.**
`ExtractPane.vue:117-132` maps each color to `{ css, position: i/(n-1) }` and
throws `population` away; `PaletteColorStrip.vue:18-21` then renders every swatch
at `Math.max(100/colors.length, …)%` — **equal width**.

Evidence (`shots/C4-05-result-harness-real.png`): the top strip (current UI) and
the bottom bar (population-weighted, what the lib computes) of the *same* palette
are wildly different. My near-black cluster has population **900**; the dominant
green **8356** — a ~9× spread — yet they get identical 1/6 widths. A user who
photographs a mostly-green scene gets a strip implying black is as present as
green. The feature's core promise ("pull palettes from any image") is undercut by
discarding the perceptual weight the engine already computed.

**Gestalt fix (not a band-aid):** thread `population` through `PaletteColor` (or a
sibling field) end-to-end and (a) width the extract strip proportional to
population, and (b) surface a single **dominant hero swatch** (large, with the
color value typeset on it) above the equal-grid swatches — giving the result the
audacious colorful pop the house language wants. `dominantColor()` already exists;
wire it.

### P1 — Two divergent extractor shells (`ExtractPane` vs `ImagePaletteExtractor`) — one-path violation

`ExtractPane.vue` (the live `#/extract` pane) and
`image-palette-extractor/ImagePaletteExtractor.vue` (tab inside `PaletteDialog.vue:80`)
are ~90% copy-paste: both own `previewDataUrl`, `colorCount`, `chromaWeight`,
identical `extractedPalette` + `kSliderGradient` computeds, identical
`runQuantize`/`debouncedReQuantize`/`onFile`/`onKChange`/`onReset`. They drift:
ExtractPane has the eyedropper + skeleton placeholder + `PaneHeader`;
ImagePaletteExtractor has an inline camera viewfinder + empty-state pipette text
and **no** skeleton. Two surfaces for one concern means design fixes (e.g. the
dominance fix above) must be made twice or silently diverge. **Fix:** collapse to
one shell + one extraction composable; let the host (pane vs dialog tab) pass
layout flags. (This is a precept §5 one-path smell, the same shape `usePaneRouter`
was created to kill.)

### P2 — Result has no color value at rest; values are hover-only (touch-hostile)

`PaletteCardSwatches.vue` renders bare color blocks; the hex/oklch is revealed
only via `SwatchHoverMenu` popover (hover, or tap on touch). The at-rest extracted
palette (`shots/C4-05`, real harness) shows six anonymous squares — no hex, no
name, no copy affordance visible. For an *extraction* tool whose entire output is
color values, the headline swatch (at least the dominant one) should typeset its
value inline. Pairs naturally with the dominance fix above.

### P2 — Empty/idle, skeleton, and result are three near-identical gray rectangles; no audacious moment

From the rendered empty state (`shots/C4-00-ref-DW6-empty-desktop.png`): dropzone
(dashed gray), then a `PaletteCardSkeleton` (gray pulsing bars + gray circles,
`PaletteCardSkeleton.vue`) sitting *below* an already-gray control strip. Idle,
loading, and (pre-upload) result all read as muted gray boxes. The house language
calls for "colorful audacious pops" and "large audacious typography"; this pane's
resting state is the most chromatically timid in the app. The skeleton swatches
are even `rounded-badge` circles while the real result swatches are
`rounded-[10px]` squares (shape mismatch between skeleton and payload → the reveal
"jumps"). **Fix:** make the skeleton swatch shape match the result, and let the
empty dropzone hint at color (e.g. a faint sample gradient or the math-grid motif)
rather than a flat gray panel.

### P2 — `oklch()` is the only output format; raw-number serialization

`oklchToCss` (`src/quantize/cluster.ts:34-38`) emits e.g.
`oklch(58.11% 0.1999 266.41)`. Extracted swatches are therefore oklch-only — no
hex toggle on the result, even though the rest of the app is hex-fluent and
casual users expect `#xxxxxx`. The eyedropper *does* honor a `colorSpace` prop
(`ImageEyedropper.vue:89`, defaulting hex), but the *quantized palette* result is
hardcoded oklch. Minor, but a format toggle on the result would lower the
intimidation floor.

### P2 — Control-row density at narrow widths

`ExtractControls.vue:28-66`: a single row packs upload button · `dock-separator` ·
`kC` label + spectrum slider + numeric · `dock-separator` · reset. At the 375
mobile width (`shots/C4-06-ref-KW2-empty-mobile375.png`) the kC slider track is
already short and flanked by two dividers; the rhythm is busier than the clean
full-width K slider above it. Consider dropping the dividers (the glass dock idiom
elsewhere uses spacing, not rules) or moving kC to its own labeled row to match
the K slider's generous treatment.

## What's good (keep)

- **Header typography** is exactly on-language: Fraunces serif `Extract` + italic
  serif "Pull palettes from any image." (`ExtractPane.vue:4`, `PaneHeader`). The
  single strongest design moment in the pane.
- **Dropzone affordance** is correct: dashed border, `ImagePlus` icon, drag
  states (`border-primary bg-primary/10 scale-[1.01]`), a11y `role=button` +
  keyboard activation + descriptive `aria-label` (`ImageDropZone.vue:14-18`).
- **Motion is tokenized and purposeful** — `var(--duration-normal)`/`--ease-standard`
  on the dropzone, `fade-slide` result transition, the eyedropper's spring
  `swatch-pop` keyframe (`ImageEyedropper.vue:289-295`). No gratuitous motion.
- **Eyedropper** is a genuinely rich, on-language sub-surface (glass `bg-card/75
  backdrop-blur-sm`, loupe magnifier, inertia gesture, pinned-state primary ring).
- Quantization runs **off-thread** via a transferable-buffer Web Worker
  (`useImageQuantize.ts`), with a 300ms debounce on K/kC changes — correct.
