## 2. Contract symmetry census — what HOLDS

These are conformant today (healthy parity, verified):

- **problem+json content-type** — both emit `application/problem+json`. value.js: `errors/index.ts:151,177,187` + the route-level `problemResponse()` wrapper (`index.ts:85-90`, 404 fallback + global `onError`). ✓
- **`/diff` WIRE envelope** — value.js's `{fromHash, toHash, ops, identical}` is byte-isomorphic to J-diff-shape §3.2 under the §4 casing rule. The conformance probe (`api/test/conformance/diff.test.ts:121-152`) asserts EXACTLY four keys, no `fromSetHash`/`toSetHash` redundancy, `fromHash`/`toHash` ARE the set-hashes, closed op vocab `added`/`removed`/`changed`, diff symmetry, dedup, immutable-pair ETag (`"<from>:<to>"`), and the inv-J-1 divergent-branch → 422 guard. **This is the model of correct one-contract-source discipline.** ✓
- **atomdiff op vocabulary** — both use the closed past-tense triple `added|removed|changed`; fourier retired `modified`→`changed` per J-diff-shape §2.1 (`atomdiff.py:56`). value.js `atomdiff.ts:25`. ✓
- **STORED vs WIRE field split** — both keep `atom_diff`/`atomDiff` as the persisted edge field while exposing `ops` on the wire (J-diff-shape §2.2). ✓
- **Session token shape + transport** — UUIDv4 in `X-Session-Token` header, never cookie. value.js `auth.ts:64,130` (`crypto.randomUUID`); fourier `sessions.py` (`str(uuid.uuid4())`). ✓
- **Login timing-safety** — both pad login to a constant **200ms** to mask user-enumeration (contract §6 C6.4). value.js `auth.ts:39,115,122` (`LOGIN_CONSTANT_DELAY_MS = 200`); fourier `sessions.py:67,74` (`asyncio.sleep(0.2)`). ✓
- **Suspension cache 60s TTL** — value.js `resolve-session.ts:28` (`SUSPENDED_CACHE_TTL_MS = 60_000`); contract §6. ✓
- **`$`-key MongoDB-operator injection guard** — both reject JSON bodies with `$`-prefixed keys. value.js `middleware/sanitize-body.ts`; fourier `main.py:70-102` (`reject_dollar_keys`). ✓
- **Soft-delete 410 Gone** — value.js LANDED I.W2 (`crud.ts:55-56`, `visibility.ts:68-69` → `GoneError`); this CLOSES the DEFERRED-TO-VALUE.JS cell the contract §13 noted. ✓
- **ETag/If-Match status discipline** — both: absent on mutation → 428, mismatch → 412, `*`→accept. value.js `middleware/etag.ts:37-51`; fourier `etag.py:34-43`. ✓
- **Slug shape-floor** — value.js `^[a-z0-9][a-z0-9-]*$` ≤120 (contract §2 cites `validation/palette.ts:19-23`); insert-then-catch via `code === 11000` (`crud.ts:133`). ✓
- **`palette.id` top-level field** — the contract §13/§1.3 forbade a `palette.id` field; value.js's `formatPalette` (`format/palette.ts:66-88`) emits `slug` as identity with NO top-level `id`/`_id`. The §13 concern is **stale/already-resolved**. ✓

## 3. Contract symmetry census — GAPS (value.js → contract divergences)

### P1 — Error-type URN namespace divergence (the prime one-contract-source breach)
The CRUD-CONTRACT names the shared problem-type namespace **`urn:contract:<kebab>`** and §2 C2.5 *explicitly binds value.js* to emit **`urn:contract:slug-conflict`** on a user-supplied-slug collision. value.js emits **`urn:palette-api:problem:conflict`** instead (`ConflictError` → `errors/index.ts:72` code `"conflict"` → URN `urn:palette-api:problem:conflict`, `errors/index.ts:155`). Grep confirms **ZERO `urn:contract:` strings anywhere in value.js `api/src` or `api/test`**. Every value.js error type is in its own repo-local namespace. This is the single largest "what SHOULD be shared" gap: the contract's §5 catalog (`slug-conflict`, `not-found`, `etag-mismatch`, `validation-failed`, `rate-limited`, …) is a closed shared vocabulary; value.js re-spells all of them. A generic cross-repo client / agent reading the `type` URN to branch CANNOT treat the two backends uniformly — the very thing problem+json's `type` URI exists to enable.
**Verdict:** real contract divergence. Either the contract namespace should be honored (value.js adopts `urn:contract:`), or the contract should bless per-repo URN namespaces as healthy (it currently does NOT — §2 C2.5 names the literal `urn:contract:slug-conflict` as value.js's binding). **This is the one-contract-source decision tranche-N should make explicit.**

### P1 — Session TTL: 7 days vs the contract's 30
Contract §6 binds `session_ttl_days = 30` ("30-day TTL"). fourier uses `settings.session_ttl_days` (30). **value.js mints sessions at 7 days** — `auth.ts:36`: `const SESSION_TTL_MS = 7 * 24 * 60 * 60 * 1000;` with the comment "7 days — the only place this constant lives", applied at both register (`auth.ts:95`) and login (`auth.ts:147`). This is a concrete, measurable §6 violation.
**Compounding internal inconsistency:** value.js's cron reaper targets `sessions.lastSeenAt < now − 30d` (`cron.ts:12,20` `THIRTY_DAYS_MS`), but the `expiresAt` index reaps at the 7-day mint horizon first — so the 30-day cron sweep is **dead code** for the stale-session class (anything past 7d is already gone via `expiresAt`). The two horizons (7d mint, 30d cron) disagree with each other AND with the contract's single 30d figure.

### P2 — Idempotency: in-process LRU vs the contract's durable, conflict-detecting store
Contract §0 SOTA-4 + `lib/crud/idempotency.py` define a **Mongo-backed** store with a TTL index, a `request_hash` (sha256 of method+path+body), and a **409 `urn:contract:idempotency-replay-conflict`** when the same key arrives with a *different* body. value.js's `middleware/idempotency.ts` is an **in-process `LRU`** (`cache/lru.ts`), per-process, lost on restart, that does **NOT** compute a request-hash and does **NOT** emit the replay-conflict 409 — instead it replays the FIRST result verbatim regardless of body (its own test asserts this: `idempotency.test.ts:113-143` "replay short-circuits even when the second body differs"). So a client that reuses a key with a genuinely different payload gets a silent wrong-body replay where the contract mandates a loud 409. This is a deliberate, documented value.js simplification (the middleware header-comment §"Store" justifies it as a retry-collapser, not exactly-once), but it is a **behavioral divergence from a named contract clause** (the `idempotency-replay-conflict` catalog row, `errors.py:63`).
**Verdict:** healthy-ish KISS at the value.js scale (single-replica, retry-collapse use-case), BUT undocumented at the contract level — the contract has no DEFERRED/relaxation row for it. Tranche-N should either book it as a sanctioned divergence or close it.

### P2 — ETag basis divergence (different strong validators)
fourier's ETag is a canonical digest over a **mutable-field projection** (`etag.py:19`: visibility/title/description/tags/palette_slug/updated_at). value.js's is `currentHash ?? updatedAt.toISOString()` (`etag.ts:23`). Both are strong, both flip per-update, both RFC-quote — the *status discipline* (428/412/`*`) is identical and conformant. But the validators are computed from different material, so they are NOT cross-comparable (a client cannot transplant one repo's ETag to the other). The contract §5 binds the *behavior* (strong validator, If-Match required on mutation) not the *derivation*, so this is **within-contract latitude** — flagged as a divergence to be AWARE of, not a violation.

### P2 — Response field-casing: camelCase vs the contract's snake_case (sessions/palettes)
The contract §6 sessions table is written entirely snake_case: `{token, user_slug}`, `/me` → `{user_slug, created_at}`. value.js returns **camelCase**: `{token, userSlug}` (`auth.ts:101,154`), `/me` → `{userSlug, createdAt}` (`auth.ts:80-82`). The J-diff-shape doc §4 explicitly blessed TS-camel↔Python-snake as the ONLY allowed difference *for the `/diff` envelope* and added a casing-normalizing conformance clause — but **no analogous casing clause exists in CRUD-CONTRACT §6/§3 for the session/palette envelopes**. So the broader API surface has the same camel↔snake split as `/diff`, but only `/diff` documents+normalizes it. **Verdict: healthy divergence (per-language serializer convention), but UNDER-DOCUMENTED** — the contract should generalize J-diff-shape §4's casing rule to the whole surface, or value.js's camelCase is technically off-contract everywhere the contract writes snake_case.

### P2 — `npm run test:conformance` script absent
Contract §10 names value.js's conformance run-command as `npm run test:conformance` (a "to be authored by the value.js alignment-tranche" deliverable). value.js `api/package.json` has only `"test": "vitest run"` — no `test:conformance` script, though the three conformance files DO exist under `api/test/conformance/`. The conformance suite runs but is NOT separately invokable by the contract-named command. Minor, but it means the §10 "aggregate run command per repo" row is unsatisfiable as written for value.js.

## 4. Gaps in the OTHER direction (fourier → value.js)

- **fourier's idempotency is the richer, contract-faithful impl** (durable Mongo store + request-hash conflict). If the one-contract-source spine is to be honored, the SHARED idiom is fourier's, and value.js is the laggard — not vice versa. No value.js-side capability is missing FROM fourier on the CRUD core.
- **fourier carries a `palette_slug` FK to value.js** (`visualization.palette_slug`, contract §13) — a real cross-repo coupling value.js must keep resolvable via `GET /palettes/:slug`. value.js honors it (200/404/410). No gap.
- **fourier's deploy-gate enforces the `/`→404 problem+json vhost contract** (F-Inv 22* enforced arm); value.js's `/`→200 health JSON (`index.ts:78`) does NOT, and `/health`/`/docs`/`/openapi.json`→404. This asymmetry is exactly why INVARIANTS.md §2.7 RETRACTED the "symmetric vhost" claim and booked the color gap as a value.js coordination note (`docs/constellation/ADOPTION-ASKS.md §4`). **This is a known, ledgered divergence, not a fresh finding** — but tranche-N should decide whether value.js adopts the 4-endpoint vhost contract or the contract formally narrows to fourier-only.

## 5. Shared-idiom opportunities (what SHOULD be shared vs healthy divergence)

**SHOULD converge (one-contract-source):**
1. **Error-type URN namespace** — the single highest-value convergence. The `type` URI is problem+json's cross-system contract; two namespaces defeat it. Either value.js adopts `urn:contract:`, or the contract sanctions per-repo namespaces in §2/§5. (P1)
2. **Session TTL = 30d** — a one-constant fix in value.js (`auth.ts:36`) that also repairs the internal 7d-mint/30d-cron inconsistency. (P1)
3. **Casing rule generalization** — lift J-diff-shape §4's TS-camel↔Python-snake clause from `/diff`-only to the whole CRUD-CONTRACT surface, so value.js's camelCase responses are *on-contract by rule* rather than silently off the snake_case tables. (P2, doc-side)

**Healthy divergence (keep, but DOCUMENT the disposition):**
- Idempotency store backing (durable Mongo vs in-process LRU) — the value.js LRU is a defensible KISS at single-replica scale; book it as a sanctioned relaxation with the `idempotency-replay-conflict` 409 either added or explicitly waived.
- ETag derivation (mutable-projection vs currentHash) — within contract §5 latitude; the behavior is symmetric.
- Slug word-lists, framework, route prefix, deploy topology — correctly per-repo (contract §2 already RETIRED-AS-OVER-SPEC the shared word-list; inv-16 admits per-language modules).
- The atomdiff hand-typed twins — the GOLD STANDARD here; identical wire shape, repo-local atom value, each probe binds to the doc not the sibling. No change.

**Correctly NOT shared (no shared package/codegen — inv-16/inv-26):** there is no `BaseCRUDRouter`/`CRUDMixin`/codegen, and that abstinence is *deliberate and correct* (fourier `lib/crud/__init__.py:9-12` documents the "framework-in-disguise probe rejects any helper that owns the request lifecycle"; value.js composes helpers explicitly too). Do not introduce a shared runtime.

## 6. Scale & simplicity verdicts

- **Scale:** Both APIs are right-sized for the product (a palette/visualization gallery with sessions, votes, forks, versions, admin). No god modules — value.js holds the L-invariant ≤350 LoC cap (largest crud-list.ts = 226). fourier's `lib/crud` modules are all ≤145 LoC. No over-abstraction; no premature sharding/queue/event-bus.
- **Simplicity (KISS) — value.js side:** the in-process LRU idempotency is the *simpler* choice and arguably the right one at single-replica scale (matches the rate-limit + suspended-user-cache precedent). The 7d-mint/30d-cron mismatch is an *accidental* complexity (two horizons that should be one). No contrivance found.
- **Simplicity — fourier side:** the Mongo-backed idempotency is heavier but contract-faithful; the `problem()` + `functools.partial` catalog is exemplary KISS (one builder, 21 closed rows, `errors.py` is 80 lines total). No KISS violations.
- **No KISS violation in EITHER repo's CRUD core.** The only "complexity" findings are *divergences* (two URN namespaces, two TTLs, two casing conventions) that cost cross-repo legibility, not local simplicity.

---

## 7. Bottom line for tranche-N

The one-contract-source spine is genuinely built and the `/diff` boundary is the model of how to do it (one doc, two hand-typed probes, neither binds the other). The remaining divergences are concentrated and cheap to either close or sanction:
- **Close (cheap, P1):** session TTL 7→30 (one line + cron reconcile); decide+book the URN namespace.
- **Document (P2):** generalize the casing rule; book the idempotency-store relaxation (or add the 409); add `test:conformance` script; resolve the `/`-vhost asymmetry.

None of these are correctness bugs in value.js's own surface (161/161 green, soft-delete landed, atomdiff conformant). They are *cross-repo contract-fidelity* items — the precise subject of the one-contract-source invariant.
