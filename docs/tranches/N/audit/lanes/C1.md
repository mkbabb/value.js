# Lane C1 — Color PICKER pane visual + design audit

**Scope:** `demo/@/components/custom/color-picker/` (controls/, display/, editing/, visual/) at 1440×900.
**Date:** 2026-06-11 · **Branch:** tranche-f-handoff @ 0cb5dd2 · **glass-ui:** 3.10.1 (local node_modules) · **value.js:** 0.11.2

## Evidence basis (read this first)

The live demo **does not mount** on this branch — two independent glass-ui skews hard-fail the build:

1. **JS layer** — `ComponentSliders.vue:117` imports `{ GlassCarousel, GlassCarouselItem } from "@mkbabb/glass-ui/glass-carousel"`. Neither the subpath nor the symbols exist in glass-ui 3.10.1: the export key is `./carousel` (not `glass-carousel`), and the carousel module exports `Carousel`/`CarouselItem` (verified: `node_modules/@mkbabb/glass-ui/dist/carousel.d.ts` exports `Carousel, CarouselContent, CarouselDots, CarouselItem, CarouselNext, CarouselPager, CarouselPrevious, GlassCarouselPager, useCarousel`). `require.resolve('@mkbabb/glass-ui/glass-carousel')` throws.
2. **CSS layer** — `@mkbabb/glass-ui/styles` (imported at `demo/@/styles/style.css:23`) pulls `dist/styles/index.css:148` → `@import "../components/custom/tabs/segmented-tabs.css"`, but that file is **not in the published dist** (`dist/components/custom/tabs/` holds only `SegmentedTabs.vue.d.ts`, `composables/`, `constants.d.ts`, `index.d.ts`). PostCSS throws `ENOENT … segmented-tabs.css`, which is the actual Vite overlay that paints over the whole viewport.

My headless capture (`/tmp/c1-shot.mjs`, 1440×900 dsf2, warm `.vite` cache) confirms the dead mount: `.pane-shell` absent, 0 slider tracks, 0 thumbs, 6× HTTP 500, and the only screenshot (`shots/C1-01-picker-full.png`) is the red Vite error overlay. So this lane's pixel judgments are grounded in the **committed K-era evidence** `docs/tranches/K/audit/visual-evidence-2026-06-04/picker-{1280,1440,375}-{light,dark}.png` — which faithfully represent today's picker because **no picker source has changed since 2026-06-04** (last `ComponentSliders.vue` edit: 2026-05-20, D.W4 `5674d1f`). The K captures were taken when glass-ui still resolved `GlassCarousel`; today's tree cannot render the label rail at all.

## P0 findings

### P0-1 — Picker pane is dead-on-arrival (two glass-ui skews)
Both blockers above. The brief framed this as "3 demo typecheck errors"; in fact it is a **runtime mount failure** — worse than typecheck-red. The CSS `segmented-tabs.css` ENOENT (a glass-ui *packaging* bug: the SegmentedTabs CSS isn't shipped) blocks every pane, not just the picker. Gestalt fix: (a) value.js side — `ComponentSliders.vue:117` must import from `@mkbabb/glass-ui/carousel` with the real symbol names, but see P1-1: the *intended* widget (a vertical letter rail) is **not** what `Carousel`/`CarouselItem` is, so this is a design re-think, not a rename; (b) glass-ui side — publish `segmented-tabs.css` in dist or drop the dangling `@import` at `index.css:148`. This is the value.js→glass-ui→consume publish-spine breaking in the consumer; it belongs in tranche N's cohort-gate.

### P0-2 — The channel label rail (the picker's primary navigational affordance) cannot render
`ComponentSliders.vue:7-31` wraps the per-channel letter labels (L/A/B…) in `<GlassCarousel orientation="vertical">` + `<GlassCarouselItem :active=…>`. With the import dead, the entire `grid-cols-[auto_1fr]` left column (the letter rail visible in `picker-375-{light,dark}.png`) is gone. Even after the import is fixed to `Carousel`/`CarouselItem`, glass-ui's `Carousel` is an **embla horizontal swipe carousel** — it has no `orientation="vertical"` letter-rail-with-`:active`-item semantics. The design intent (a vertical, click-to-scroll, active-highlighted channel index) maps to a **`Tabs`/`SegmentedTabs` vertical variant or a plain styled list**, not a carousel. Gestalt fix: replace the GlassCarousel with the glass-ui vertical-Tabs primitive (reuse an existing component-type name per `feedback_glass_ui_first_class.md`), or a semantic `<nav>` list; the carousel was always a category error.

## P1 findings

### P1-1 — Channel-letter rail has a label collision (a vs alpha, c vs …)
`componentLabel()` (`ComponentSliders.vue:155-157`) returns `component.charAt(0).toUpperCase()`. Lab's component keys are `l, a, b, alpha` (verified `src/units/color/constants.ts`), so the `a` axis **and** `alpha` both render as **"A"** in the rail — two identical, indistinguishable labels stacked vertically. `componentDescription()` (line 167) compounds it: `c.startsWith(upper) || c.startsWith(component)` matches `a` before `alpha`, so the alpha row's tooltip can show the a-axis description. Same single-letter collapse loses information across spaces (e.g. `lch` `l/c/h/alpha`). Fix: a per-space label map (or 2-char disambiguation: "A" vs "α"/"Op") rather than blind first-letter truncation.

### P1-2 — Color-space-switch entrance animation is wired but never fires (dead CSS class)
`ComponentSliders.vue:4` sets `class="… stagger-children"` and maintains a whole re-trigger apparatus (`animationKey` ref + `:key="animationKey"` + `watch(currentColorSpace, () => animationKey.value++)`, lines 150-152). But **`.stagger-children` is defined nowhere** — not in `demo/@/styles/*.css`, not in glass-ui's published CSS (glass-ui ships `.char-stagger > .char` and scroll-driven stagger-reveal, but no `.stagger-children`). The `:key` forces a full slider-list remount on every space change (a real cost) for an animation that produces zero visual effect. This is the same "silent no-op utility in a consumer" pattern as the constellation grand-audit's P9 rounded-panel finding. Fix: either define the stagger utility (per-child `animation-delay` via `nth-child`/`--index`) or remove the dead `animationKey`/`:key` machinery.

### P1-3 — `ColorComponentDisplay` (the hero readout) bypasses the φ type scale
The single most prominent element — the giant component readout ("92%, 88.8, 20") — uses raw `text-4xl` (`ColorComponentDisplay.vue:3`) and `text-lg` for the unit (line 28), while **every sibling in the pane honors the semantic ladder** (`ColorSpaceSelector` → `text-title sm:text-display`; sliders → `text-subheading`/`text-caption`/`text-mono-caption`; nutrition label → `text-subheading`/`text-body`/`text-small`). DESIGN.md §17/§21 makes the φ-ratio named utilities canonical and calls raw `text-2xl etc.` the anti-pattern. The hero number is exactly where scale coherence matters most. Fix: promote to `text-display`/`text-title` (the readout reads as a display element) so it sits on the same ladder as the space selector beside it.

### P1-4 — `EditDrawer` renders into the DOM but is permanently `display:none`
`EditDrawer.vue:115-117` ends the scoped style with an **unconditional** `.edit-drawer { display: none; }` (comment: "fully handled by TopDock on mobile and not needed on desktop"). Yet the component is still mounted in `ColorPicker.vue:33`, teleports to `<body>`, runs a `<Transition>`, renders two `WatercolorDot`s + the →-arrow diff + Check/Undo buttons + two TooltipProviders — all invisibly, on every edit. The `.edit-drawer-{enter,leave}` desktop/mobile transitions (lines 91-104) are likewise dead. This is mounted-but-hidden dead UI carrying real render/tooltip cost. Fix: if the drawer is truly superseded by the dock, delete the component and its mount; if it should show on desktop, the blanket `display:none` is a bug.

## P2 findings

### P2-1 — HeroBlob footprint degenerates to an orphan speck at desktop
`HeroBlob.vue:8` sizes `GooBlob` at `w-[7rem]` (~112px) and `ColorPicker.vue:22` places it in `col-start-2 col-span-2 row-span-2 justify-self-end`. K-era `picker-1280-light.png` + DELTA C3 show it collapsing to a ~24px speck floating top-center over empty space at desktop (ambiguous grid intrinsic width + absolute-overflow canvas), while mobile reads correctly (`picker-375-dark.png`). The blob is the pane's signature "audacious colorful pop" and at the audited 1440 viewport it is nearly absent. (Pre-existing, already routed to a blob-extirpation/footprint mandate; re-confirmed here for the picker pane specifically.)

### P2-2 — Aurora backdrop is palette-blind behind the picker
DELTA C2: the full-viewport WebGL aurora paints an identical static blue/teal "Sky" preset across all captures while the picked color is magenta — so the picker card floats on a backdrop that ignores the very color it is editing. The house language wants the surface to respond; here the most color-aware pane sits on a color-deaf field. (Chronic aurora-from-color mandate; re-confirmed for picker.)

### P2-3 — Slider value badge (`SliderThumb` tooltip) competes with the always-on range hint
Each slider row stacks: a muted italic range hint above the track (`ComponentSliders.vue:40-42`, `text-caption italic opacity-50`) **and** a `fira-code` value tooltip on the thumb (lines 88-94). In `picker-375-*` the per-row range-hint line ("(0% - 100%)", "(-125 - 125)") adds a loose vertical gap and a second numeric system competing with the thumb badge and the big readout above — three numeric representations of the same channel. Consider folding the range into the thumb tooltip or the channel-rail tooltip (which already shows `currentColorRanges` at line 27) to tighten the rhythm and reduce numeric redundancy.

### P2-4 — Spectrum hover shadow is a fixed offset, ignores the dot
`SpectrumCanvas.vue:264` applies a hard `8px 8px 0 0` offset shadow in `color-mix(spectrum-color, black)` on hover (consistent with the house hard-shadow idiom — a strength). Minor: the offset is constant regardless of where the watercolor dot sits, so at extreme dot positions the hard shadow and the dot can read as disconnected. Low priority; the idiom itself is correct.

## What's working (preserve)

- **Distinctive display typography** — genuine Fraunces italic ("Lab" space label in magenta, the large numeric readout); not generic-AI sans (DELTA C-OK1, `picker-375-light`).
- **Per-channel gradient slider tracks** — each track paints its own channel gradient (`componentsSlidersStyle`), L=black→pink, A=green→magenta; banding-free (inv-K-2, DELTA C-OK4).
- **Color-reactive ActionButton motion** — `ActionButton.vue:124-126` flashes the icon stroke to the *live* `cssColorOpaque` on click (`--flash-color`), with a rotate variant on reset; purposeful, color-tied animation (the correct "colorful pop" idiom).
- **Dark-mode chrome parity** — panel → charcoal glass, heading stays magenta, readout flips white (DELTA C-OK2, `picker-375-dark`).
- **Watercolor dot on the spectrum** — organic SVG-filtered crosshair with contrast-safe border logic (`SpectrumCanvas.vue:226-243`); the filter def exists (`svg-filters/SvgFilters.vue:5`).
