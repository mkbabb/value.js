# Lane E3 — Performance Audit

**Scope**: Build outputs (lib + gh-pages), hot-path verification (computed-endpoint cache,
frozen channel plan, lerpArray, color dispatch, quantize, parsing throughput), demo runtime
(initial load, blob shader, RAF discipline). Read-only; benches under `/tmp` + the repo's own
`bench/` suite (re-run, not authored).

**Substrate**: value.js `tranche-f-handoff` @ `0cb5dd2` (v0.11.2), node v26.0.0, darwin arm64.
glass-ui local `tranche/AY` @ 3.10.1. `@mkbabb/parse-that` 0.8.2 installed.

**Method note**: every size/number below is measured live, not read from a doc. The doc-cited
numbers I checked (F.W3 bundle gate "124,936"; bench "≥2×" gates) are stale-but-honest — the
core grew to 128,029 B; the DIRECT_PATHS gate's oklab/oklch arms are flat (see F-3).

---

## Headline

The library's *hot paths are genuinely fast and the optimizations land* (computed-endpoint
cache −93.6%, frozen color plan 3.0–4.4×, SoA lerpArray bites ≥K=2, parse memo-HIT ~9 ns).
The **structural perf debt is all in the SHIP SURFACE, not the compute**: (1) the published npm
tarball is **560.7 KB**, of which **303.6 KB (54%) is bundled Prettier** that a non-`formatCSS`
consumer never needs and that the bundle-size CI gate does not even see; (2) the **gh-pages
deploy build is fully BROKEN at HEAD** (glass-ui 3.10.1 CSS skew), so the demo deploy artifact —
and its vendor-katex/vendor-highlight chunk map — cannot be produced.

## §1 — Build outputs

### 1.1 Library build (`npm run build` — mode production) — PASS

`dist/value.js` (ESM core):
- raw **128,029 B (125.0 KB)**, gzip **38,884 B (38.0 KB)**.
- `vue` + `@mkbabb/parse-that` correctly `external` (vite.config.ts rolldownOptions.external).
- `attachDebugInfo: "none"` + `drop:["console","debugger"]` applied — clean.
- Core grew ~3 KB since F.W3 (124,936 → 128,029); still under the 145 KB raw gate (~20 KB headroom).

**But the published package is 560.7 KB unpacked, 46 files** (`npm pack --dry-run`). Top entries:

| File | Size | Note |
|---|---:|---|
| `dist/postcss-CWmVrHvs.js` | 192.7 KB | **bundled Prettier postcss plugin (lazy)** |
| `dist/value.js` | 125.0 KB | the core |
| `dist/standalone-DlyrSOeX.js` | 110.9 KB | **bundled Prettier standalone (lazy)** |
| `dist/units/color/constants.d.ts` | 27.3 KB | dts |
| `dist/units/constants.d.ts` | 13.6 KB | dts |

The two Prettier chunks (**303.6 KB raw / 82.5 KB gzip combined**) are emitted by the *single*
dynamic `import("prettier")` + `import("prettier/plugins/postcss")` at
`src/parsing/serialize.ts:135-137` (the `formatCSS` lazy wrapper). They are tree-shaken out of
`value.js`'s static graph (dynamic-only) but **still shipped in `dist/` and published**
(`files: ["dist"]`). See F-1 — this is the lane's top finding.

### 1.2 gh-pages build (`npm run gh-pages`) — **FAILS, 0 chunks emitted**

`npm run gh-pages` aborts with a `CssSyntaxError` (Rolldown aggregated build error). **76**
occurrences of the same failing CSS `@import`; `dist/gh-pages/` is never created.

Root cause (cross-repo, confirmed): glass-ui 3.10.1's published
`dist/styles/index.css:148` contains
```
@import "../components/custom/tabs/segmented-tabs.css";
```
but **that file does not exist in glass-ui 3.10.1's dist tree** (the SegmentedTabs CSS was
renamed/relocated; only `dist/styles/dock-controls/tab-button.css` survives). The demo's
Tailwind/PostCSS compiler resolves glass-ui's `./styles` Tailwind-source surface
(`@import "@mkbabb/glass-ui/styles"`), walks that dangling `@import`, and throws. It is the same
glass-ui API-skew root cause as the RED typecheck — but it escalates the impact from
"typecheck red" to **"the deploy artifact cannot be built."**

Consequence for this lane: **the vendor-katex / vendor-highlight chunk map at HEAD is
UNMEASURABLE** — there is no demo build to inspect. The chunk-split config is present and
correct (vite.config.ts `codeSplitting.groups` for `vendor-katex` /node_modules[\/]katex/ and
`vendor-highlight` /node_modules[\/]highlight/), and the heavy markdown/katex/highlight code is
deferred behind 11 `defineAsyncComponent(() => import(...))` pane boundaries
(`usePaneRouter.ts:68-77`), so the *architecture* is sound — but I cannot certify the byte map
until the deploy build is unblocked. See F-2.

## §2 — Hot paths at HEAD (all verified against the real engine / dist)

### 2.1 Computed-endpoint cache (C1) — SOLID, lands

`src/units/interpolate.ts:26-72` (`lerpComputedValue`) + `src/units/normalize.ts:154-189`
(`layoutEpoch` / `bumpLayoutEpoch`). Steady-state collapses to one int-compare + bare `lerp`;
cold path resolves both endpoints once per `(target, epoch)`. Invalidation is O(1) (epoch++ +
`getComputedValue.cache.clear()`), auto-wired to `window.resize` with an SSR guard.

`bench/computed-endpoint.mjs`: baseline 128.3 ns/leaf/frame → C1 8.2 ns/leaf/frame =
**−93.6%** (BITES). Design is clean; no transposition.

### 2.2 Frozen color-channel plan (B3) — SOLID, lands

`src/units/interpolate.ts:104-135` + `buildColorChannelPlan` (236-267). Closure-free flat loop
over parallel `Float64Array`s (light SoA), hue ÷360 folded into the plan (B5), `_lerp`
predispatch. `bench/color-interp.mjs` median **3.65×** (hex/rgb 4.44×, rgb 3.65×, oklch-hue
3.64×, hsl-hue 3.03×). No transposition.

### 2.3 `lerpArray` (SoA, D2) — SOLID, correctly gated by K

`src/math.ts:48-60`. `bench/numeric-soa.mjs`: K=1 0.57× (SLOWER), K=2 1.82×, K=4 2.41×,
K=8 3.13×, K=64 5.14×. The MEASURE-FIRST guidance in the docstring ("use only for K≥2") is
honest and matches measurement. No issue.

### 2.4 Color-conversion dispatch (`color2` + DIRECT_PATHS) — partly flat (F-3)

`src/units/color/dispatch.ts:159-185` consults `getDirectPath` then falls to the XYZ hub.
`bench/color-dispatch.mjs` median speedup-over-XYZ-hub:
- **hsl→rgb 3.08×** (real win — `hsl2rgb` is a cheap closed-form, no cube-roots, no XYZ).
- **oklab→rgb 1.03×, oklch→rgb 1.08×** — essentially FLAT.

Structural reason (read `conversions/direct.ts:79,157`): `directOklabToRgb` /
`directOklchToRgb` call `gamutMap()` unconditionally to match `xyz2rgb`'s default
`correctGamut=true`. `gamutMap` (`dispatch.ts:189-215`) allocates an `RGBColor`, runs the
in-gamut analysis, and on the in-gamut fast path round-trips `color2(clamped, colorSpace)`. The
gamut-map cost dominates and is paid by BOTH the direct and the XYZ-hub path, so the "saved
XYZColor allocation + one matmul" is in the noise. The 4 oklab/oklch DIRECT_PATHS entries add
code + a typed mapped-type for ~3–8% — they earn their keep far less than the hsl pair. See F-3
(P2).

### 2.5 Quantize — well-structured, off main thread

`src/quantize/cluster.ts` is tight typed-array code (Float64Array SoA pixels, Uint32Array
index sort, pre-allocated accumulators in `kmeansIterate`). Critically, the demo runs it on a
**Web Worker** (`useImageQuantize.ts:11` `@lib/quantize-worker?worker`), so image extraction
never blocks the main thread. No transposition found. (One micro-note, not load-bearing:
`splitBucket` allocates `indices`+`sorted` Float64Arrays per split and `Array.from(populations)`
allocates on every k-means return; both are one-shot per extraction, not per-frame — ignore.)

### 2.6 Parsing throughput — healthy

Real-engine probe (`/tmp/e3-realbench.mjs` against `dist/value.js`):
- `parseCSSColor` memo HIT (rotating 9-input pool): **9.1 ns/call**.
- `parseCSSColor` memo MISS (unique `rgb(...)`): 121.3 ns/call.
- `parseCSSValue` memo MISS (unique dim): 74.2 ns/call.

`parseCSSValue` / `parseCSSPercent` / `parseCSSTime` are memoized (`parsing/index.ts:262-275`);
the F-era name-lookup dispatch (`bench/parser-namelookup.mjs` 40.8× over the 155-branch any())
and the color first-char dispatch land. No transposition.

## §3 — Demo runtime

### 3.1 Initial load weight — unmeasurable at HEAD (build broken)

Blocked by §1.2. Architecture is sound: App.vue has 22 eager imports but all panes (and thus
markdown/katex/highlight) are `defineAsyncComponent` route-split (`usePaneRouter.ts:68-77`,
`AboutPane.vue` 11 sites), and the vendor-katex/vendor-highlight chunk groups are configured.
The initial chunk should be lean — but **cannot be certified** until the deploy build is fixed.

### 3.2 Blob shader (GooBlob / `useMetaballRenderer.ts`) — EXEMPLARY RAF discipline

`demo/@/components/custom/goo-blob/composables/useMetaballRenderer.ts`. This is the model the
rest of the fleet should copy:
- DPR capped at 2 (line 149) — no 3×/4× retina overdraw.
- `tabHidden` gate via `visibilitychange` (256-258, 164-167) — RAF idles when tab hidden.
- `prefers-reduced-motion` honored — single render, no loop (84-86, 278-282, 251-253).
- `cssColorToRgb` memoized, 256-entry cap (56-70) — the 2D-canvas `getImageData` resolve runs
  once per unique color, not per frame.
- Full teardown: `cancelAnimationFrame`, `ResizeObserver.disconnect`, listener removal, GL
  resource deletion (304-323); mid-frame-after-destroy guard (189-194).
- Context-loss / restore handlers wired (270-271, 285-302).

No perf concern in the blob. (Scope note: the blob's *existence* is mandated for extirpation by
a successor tranche per memory, but that is an architecture call, not a perf one.)

### 3.3 RAF discipline across the demo — clean within value.js

12 RAF call sites (`grep -rln requestAnimationFrame demo/` → 12 files; 5 are hero-lab, a
separate experimental entry). The 5 main-app loops I spot-checked
(watercolor-dot, mix-animation, inertia-gesture, SpectrumCanvas, dock layer-transition) each
carry teardown + (where it's a sustained loop) visibility/reduced-motion gating. The
"~40-ungated-RAF-loop epidemic" the constellation grand-audit flagged is a *cross-repo* count;
**value.js's own demo is not a contributor** — its RAF hygiene is good.

## §4 — Findings ledger

### F-1 (P1) — 304 KB of Prettier is bundled into the published tarball; gate is blind to it
`src/parsing/serialize.ts:135-137` dynamic-imports Prettier for `formatCSS`. Prettier is in
`devDependencies` only (not `dependencies`, not in rolldown `external`), so Rolldown **bundles**
Prettier's standalone+postcss into `dist/postcss-*.js` (192.7 KB) + `dist/standalone-*.js`
(110.9 KB). Published tarball = **560.7 KB unpacked, 303.6 KB (54%) Prettier.** Every consumer
downloads it; `formatCSS` consumers get a bundled copy that cannot dedup against their own
Prettier; non-`formatCSS` consumers ship pure dead weight. The F.W3 bundle-size CI gate
(`.github/workflows/node.js.yml`, `stat dist/value.js`) measures *only* `value.js` and reports
PASS — it is structurally blind to the 304 KB. **Structural fix**: add `prettier` to rolldown
`external` (alongside `vue`/`parse-that`) and declare it an *optional peerDependency*; or move
`formatCSS` to a separate `/format` subpath export so the core never pulls it. Either way the
core ships at ~38 KB gzip and `formatCSS` users supply their own Prettier.

### F-2 (P0) — gh-pages deploy build is fully broken (glass-ui 3.10.1 CSS skew); chunk map unmeasurable
`npm run gh-pages` aborts: glass-ui `dist/styles/index.css:148` `@import`s a
`segmented-tabs.css` absent from 3.10.1's dist. 76 failing imports, `dist/gh-pages/` never
emitted. The demo deploy artifact cannot be built at HEAD; vendor-katex/vendor-highlight chunk
sizes cannot be certified. Same root cause as the RED typecheck (SegmentedTabs/BouncyTabs/
glass-carousel skew), but the *deploy* impact is more severe than the typecheck framing
suggests. Resolved by the cohort cut to a glass-ui that ships `segmented-tabs.css` (or the demo
dropping the dangling Tailwind-source `./styles` import). Tranche-N must treat the deploy build
as a gated cohort dependency, not a value.js-internal fix.

### F-3 (P2) — oklab/oklch DIRECT_PATHS entries are ~flat (1.03–1.08×); only hsl earns its keep
`conversions/direct.ts`: the oklab/oklch direct paths call `gamutMap()` unconditionally
(lines 79, 157), and gamut-map cost dominates equally on both the direct and XYZ-hub routes, so
the saved XYZColor alloc + matmul nets ~3–8% (`bench/color-dispatch.mjs`). Only hsl→rgb is a
true win (3.08×). Not a regression — but the 4 oklab/oklch entries + the `DirectPathsTable`
mapped-type carry maintenance weight for a near-noise gain. Options for tranche-N: (a) keep but
stop documenting them as "hot wins"; (b) drop the oklab/oklch direct entries and let them route
the hub (the table then carries only the hsl pair); (c) the real win — hoist the in-gamut
fast-path check so an already-in-gamut conversion skips the `gamutMap` round-trip entirely (this
would help BOTH paths). No action needed for correctness.

### F-4 (P2) — bundle-size CI gate scope is too narrow
Follows from F-1: the gate (`stat dist/value.js ≤ 148480`) should measure the *published
tarball* (`npm pack --dry-run --json` unpackedSize) or at least sum every emitted `dist/*.js`,
so a re-bundling regression (like the Prettier chunks) surfaces at PR time. Today a 304 KB
ship-weight regression passes the gate green.

## §5 — Top structural perf transpositions for tranche-N (priority order)

1. **[P1] Externalize Prettier** — `external: [..., "prettier"]` + optional peerDep, or a
   `/format` subpath. Drops the published tarball from 560 KB → ~250 KB and removes 304 KB of
   non-dedup-able bundled vendor. (F-1)
2. **[P0, cohort] Unblock the gh-pages deploy build** — gate on a glass-ui cut that ships
   `segmented-tabs.css`; then certify the vendor-katex/vendor-highlight chunk map (currently
   unmeasurable). (F-2)
3. **[P2] Widen the bundle gate to the published tarball** so ship-weight regressions are
   caught. (F-4)
4. **[P2] Rationalize DIRECT_PATHS** — either drop the flat oklab/oklch entries or hoist the
   in-gamut fast-path out of `gamutMap` to benefit both routes. (F-3)

Everything else in the compute hot paths (computed cache, frozen plan, SoA lerp, quantize
worker, parse memo) is in good shape and needs no perf work.
