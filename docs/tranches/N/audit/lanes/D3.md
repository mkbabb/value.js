## §1 — What the design standard says

`fourier-analysis/docs/constellation/DEPLOY-STANDARDIZATION-DESIGN.md` (authored 2026-05-28) defines the constellation deploy standard. The five normalization targets that matter for this lane:

1. **Frontend hosting**: converge to CF Pages (wrangler recipe) — retire the `GH Pages → CF CNAME` pattern.
2. **Backend deploy**: ONE pattern — webhook → per-repo `scripts/deploy-hook.sh` (flock + health-gate + rollback-on-rollback + auto-migration). Retire `dispatch.sh` only once all four non-fourier repos migrate.
3. **CI**: ONE template — `lint → typecheck → build → test` on push/PR to master; backends add live-service tests; libraries add a release job. Universal guards: `concurrency` cancel-in-progress, `permissions: contents: read`, `timeout-minutes`.
4. **Docker hardening**: `read_only: true` + `tmpfs /tmp` + `cap_drop: ALL` + `security_opt: no-new-privileges` + resource limits + json-file log rotation.
5. **Secret discipline**: NO secret in tracked files; host-only `.env` at `0600`; GH Actions secrets for CI.

`ADOPTION-ASKS.md §3` tracks per-repo asks. The key value.js items (last re-affirmed G.W8, 2026-05-30):
- **Ask 3 (P1)** — rsync deploy-dir → git checkout, then adopt `deploy-hook.sh`.
- **Ask 5 (P3)** — GH-Pages → CF Pages.
- **inv-22-color (P3)** — bring `api.color.babb.dev` to the 4-endpoint vhost contract (`/health`, `/docs`, `/openapi.json` currently 404).

### Gaps specific to value.js CI

**G1 — No automated CF Pages deploy workflow (missing `deploy-pages.yml` analog).**  
value.js has exactly two workflows: `ci.yml` and `release.yml`. There is no `deploy-pages.yml` equivalent. The CI _builds_ the demo SPA (`npm run gh-pages` at ci.yml:186) but does NOT deploy it. The CF Pages deploy for `color.babb.dev` is purely manual via `scripts/deploy.sh frontend`. Fourier has `deploy-pages.yml` which gates the SPA deploy on a green CI run for the same SHA (the inv-28 `verified-deploy-of-record` pattern). value.js has no equivalent verified-deploy-of-record, so a red CI run does not block a manual SPA ship, and the deployed SPA is not provably tied to a CI-green SHA. This is the largest CI structural gap.

**G2 — Lighthouse gate is still soft (`continue-on-error: true`).** ci.yml:190. The lighthouserc.json budget gates are configured (CLS ≤ 0.1, LCP ≤ 2.5s, INP ≤ 200ms, TBT ≤ 300ms, a11y ≥ 0.9), but the CI step is soft-launched (ci.yml comment: "baseline-capture wave, then flip to hard-fail"). The flip has not happened. M.W3 was supposed to address this (M5.2 gap, "zero instrumented runs"), but M was never ratified. No Lighthouse/axe runs have been instrumented.

**G3 — glass-ui private-repo checkout in CI creates a fragile dependency.** ci.yml:46-51 and ci.yml:175-179 both execute `actions/checkout@v4` for `mkbabb/glass-ui` at `path: ../glass-ui`. This requires the runner to have access to a private repo without an explicit `token:` key specified. If the repository becomes truly private or access tokens change, this step will fail silently (the checkout action returns exit 0 but the path is empty). The lack of an explicit `token: ${{ secrets.GITHUB_TOKEN }}` or a PAT is a fragility even if it currently works with the GITHUB_TOKEN default scope.

**G4 — `npm run bench` bench gate in CI has fragile regex-based extraction.** ci.yml:110-124. The bench gate asserts L8 ≥ 5x, HSL→RGB ≥ 2x, and nameParser ≥ 5x speedup. The extraction uses `grep -E` + `head` piped into awk. A change in benchmark output format would silently yield empty variables, causing the awk condition `!(m < 5)` to evaluate against `0`, meaning the gate would pass with zero speedup. The `${L8_MEDIAN:-0}` fallback ensures a missing value reads as 0x speedup, but the failure mode is silent (no "regex didn't match" warning).

**G5 — Playwright smoke suite is still soft (`continue-on-error: true`) for the full project.** ci.yml:153-154. The inv-K-5 `page-load` spec is hard-gated (ci.yml:144-145), but the broader smoke project remains soft due to the pre-existing dock view-select blocker (the collapsed summary-layer overlays the combobox in headless). The comment says this is "booked at K.W2b, orthogonal to the color topology" but it has not been fixed.

### Gaps specific to value.js API deploy

**G6 — `scripts/deploy-hook.sh` is ABSENT from the repo tree (Ask 3 OPEN).**  
The fourier reference architecture has `scripts/deploy-hook.sh` tracked in the repo (flock + health-gate + rollback-on-rollback + auto-migration). value.js `scripts/` contains only `check-types.mjs`, `deploy.sh`, `dev.sh`, and `generate-favicon.mjs`. There is no `deploy-hook.sh`. Per `ADOPTION-ASKS Ask 3`, this is because the deploy directory on the host (`~/Programming/palette-api`) is an rsync target, not a git checkout — so the standard hook's `git fetch / git reset --hard origin/master` would fail outright. The `api/CLAUDE.md` claims "No rsync, no SSH (the old api/deploy.sh rsync outlier + the mbabb.fi.ncsu.edu/colors/ :3100 NCSU topology were retired at K.W2 / DEC-9)" but the sibling lane V3 (`docs/tranches/N/audit/lanes/n-verify-V3.md`) found the wire still serving pre-K.W2 code (the `id` and `status` fields from before K.W2 are still in the live API response; `published` and `atomSetHash` are absent). The CLAUDE.md claim is therefore a documentation-vs-reality gap. **Ask 3 is the critical-path blocker for deleting `dispatch.sh` on the host** — it must precede all three other repos (words, speedtest, csp-solver) to unblock that deletion.

**G7 — External health gate probes `/palettes` instead of `/health`.**  
`scripts/deploy.sh` defaults `API_HEALTH_URL` to `https://api.color.babb.dev/palettes` (deploy.sh:31) and gates on HTTP 200. The deploy precept (`docs/precepts/infra/deploy.md:20,47-48`) requires the gate to poll `/api/health` for `{"status":"ok"}`. The value.js API has NO `/health` route — the server's closest equivalent is `GET /` which returns `{"status":"ok", "service":"palette-api"}` (api/src/index.ts:78). The `compose.yaml` healthcheck probes `http://localhost:3000/` (the root), not `/health` (compose.yaml:36). These three health-check endpoints (`/palettes`, `/`, `compose healthcheck /`) are all functionally adequate but diverge from the constellation standard and from each other. The `inv-22-color` ask (ADOPTION-ASKS:116) requires adding `/health` to reach the 4-endpoint vhost contract.

**G8 — `api/apache-vhost.conf` is stale (port 3100 vs compose binding 8130).**  
`api/apache-vhost.conf` proxies `http://localhost:3100/` (lines 8-9). The `api/compose.yaml` binds the container at `127.0.0.1:8130:3000` (line 6). The vhost conf reflects the pre-K.W2 topology (`mbabb.fi.ncsu.edu/colors/` at port 3100, the NCSU-era config). If the vhost conf is in use on the live host, it is pointing at the wrong port. The `api/CLAUDE.md` §Deployment says the prod URL is `api.color.babb.dev` (the babb.dev spine), not `mbabb.fi.ncsu.edu`. This is dead configuration that should have been updated or retired at K.W2/DEC-9.

**G9 — No `/health` route in the API (inv-22-color).** Confirmed by sibling V3 audit: `GET /health` returns 404 problem+json (the correct not-found handler), and there is no `app.get("/health", ...)` in `api/src/index.ts`. The constellation inv-22 gate (DEPLOY-STANDARDIZATION-DESIGN §2.2 + the fourier deploy-hook's dual-probe) requires `/api/health` → `{"status":"ok"}`. The ADOPTION-ASKS `inv-22-color` row has this as P3 open, targeted at M.W8.C. Without it, adopting the fourier reference `deploy-hook.sh` verbatim would fail the health gate on every deploy.

### Gaps in fourier-analysis CI

**G10 — `ci.yml` has NO `concurrency` group.**  
fourier `ci.yml` starts at `name: CI` on line 27 and jumps directly to `on:` and then `jobs:` (lines 29-36). There is no `concurrency:` block. Back-to-back pushes will queue (or run in parallel) rather than cancel the superseded run. value.js ci.yml:25-27 has this correctly: `concurrency: group: ci-${{ github.workflow }}-${{ github.ref }} / cancel-in-progress: true`.

**G11 — `ci.yml` has NO `permissions:` block (implicit write-all).**  
fourier `ci.yml` carries no workflow-level `permissions:` block (confirmed: grep returns nothing). GitHub's default is `permissions: write-all` when no explicit floor is set. value.js ci.yml:29-30 sets `permissions: contents: read` as the minimal floor. The DEPLOY-STANDARDIZATION-DESIGN §2.3 template includes this guard; fourier's CI does not conform.

**G12 — `ci.yml` uses `setup-node@v4` (value.js uses `@v5`).**  
fourier ci.yml:85, 122 and deploy-pages.yml:100 all use `actions/setup-node@v4`. value.js ci.yml:53, 180 use `setup-node@v5`. The `@v5` version uses the OIDC-based token for npm provenance; `@v4` is the prior stable. For fourier (no npm publish step in CI), this is a minor hygiene gap, not a functional break — but the template specifies `@v5`.

**G13 — `ci.yml` has NO `timeout-minutes` on any job.**  
fourier ci.yml jobs (`api-tests`, `web-build`, `e2e-tests`) have no `timeout-minutes`. A hung job (e.g. a Playwright deadlock, a stalled Python server) will run for GitHub's 6-hour default before being killed. value.js ci.yml:33 sets `timeout-minutes: 20`. The UNIFIED-CICD template specifies this guard.

**G14 — `ci.yml` has NO lighthouse/axe budget gate.**  
fourier has no Lighthouse CI step in either `ci.yml` or `deploy-pages.yml`. value.js has a `lighthouse` job (soft) in `ci.yml`. The DEPLOY-STANDARDIZATION-DESIGN §2.3 template includes Lighthouse+axe. fourier's gap here is noted in the UNIFIED-CICD doc (`docs/tranches/K/audit/visual-evidence-2026-06-02/grand-audit/UNIFIED-CICD.md:91`).

**Ask 3 is the gating fourth migration for deleting `dispatch.sh` from the host.** The dispatcher remains active, routing value.js pushes through a latent-broken arm. Until a true git checkout is in place AND `scripts/deploy-hook.sh` is tracked in the value.js tree, every push to `master` silently fails to deploy the backend (or re-deploys the old rsync snapshot if the dispatcher falls through to an rsync-style operation).

The accept shape (ADOPTION-ASKS:64):
```
git -C <canonical palette-api dir> rev-parse HEAD # must succeed
push to master → deploy.babb.dev/hooks/value-js → deploy-hook.sh → git reset --hard + rebuild + health-gate GREEN
rsync deploy.sh step RETIRED
value.js's arm OUT of dispatch.sh
```
None of these are currently satisfied.

**Gaps vs the fourier reference:**

The remaining Ask 5 gap is: (a) no CI-automated deploy workflow, and (b) no inv-25 deployment ID capture.

## §10 — Summary verdict

**value.js CI is structurally sound (ci.yml is the reference template shape)** but missing two critical automation links: (1) no CI-gated automated CF Pages deploy (`deploy-pages.yml` analog) and (2) the on-host `scripts/deploy-hook.sh` is absent from the repo tree. The second is the root cause of Ask 3 still being open — the palette-api deploy directory is a non-git rsync target, so the canonical hook cannot run. The combination means value.js backend pushes route through the multiplex `dispatch.sh` to a latent-broken arm, and the SPA deploys are unverified manual operations.

**fourier-analysis CI is functionally sound** (the three-job structure, live-Mongo service, Playwright suite, verified CF Pages deploy) but misses four template-compliance guards in `ci.yml`: `concurrency`, `permissions`, `timeout-minutes`, and `setup-node@v4 → @v5`. These are hygiene items compared to value.js's structural gaps.

**The DEPLOY-STANDARDIZATION-DESIGN standard is partially implemented in both repos.** Fourier is the reference for the backend deploy pattern; value.js is the reference for the Docker hardening floor. Neither is fully compliant: fourier lacks lighthouse CI; value.js lacks the deploy-hook and the CI-gated SPA deploy.

The three most actionable items for tranche N's M.W8 wave are G6 (Ask 3 + deploy-hook.sh), G1 (deploy-pages.yml CI gate), and G9 (add `/health` route to enable hook adoption).
