## 1. The WithId census — EXACT count NOW

**Pattern `as <Model> & { _id: unknown }` (the value cast form): exactly 25.**
Verified `grep -rn "as [A-Za-z]* & { _id" src/ | wc -l` → 25. This matches the tranche-M audit's "25 casts / 4 models → `WithId<T>`" finding (M.W2) precisely — it has NOT drifted since M was authored (2026-06-04).

Broader `& { _id` occurrences = 38: the other 13 are the *type-annotation* form on function params/fields (not casts) — `format/palette.ts:57` `doc: Palette & { _id: unknown }`, `forks.ts:30/167/191`, `versions.ts:128`, `color/queries.ts:48/61/102`, `admin/{colors,audit,tags}.ts`, `paletteVersion.ts:40` (`& { _id: string }`).

Census by model:
- `Palette & { _id: unknown }` — **18** casts (crud.ts x5, crud-list.ts x3, forks.ts x2 incl. an array cast `data as (Palette & {_id})[]`, versions.ts x1, visibility.ts x2, routes/forks.ts x3, routes/versions.ts x1, admin/users.ts x1).
- `ProposedName & { _id: unknown }` — **5** (color/queries.ts x3, color/proposals.ts x1, admin/colors.ts x1).
- `Tag & { _id: unknown }` — **2** (color/queries.ts x1, admin/tags.ts x1).
- `AdminAuditEvent & { _id: unknown }` — **1** (admin/audit.ts x1).

**Root cause (verified):** `models.ts` deliberately *omits* `_id` from `Palette`, `ProposedName`, `Tag`, `AdminAuditEvent` (comment at `models.ts:18` — "`_id` field is omitted from most interfaces"). But every formatter needs `_id` (to project `id: String(doc._id)`), and `Collection<Palette>.findOne` returns `WithId<Palette>` whose `_id` is `ObjectId` (for the ObjectId-keyed collections) — yet the model type says `_id` is absent. So callers re-attach it via the intersection cast at every formatter call site. This is the SAME escape-hatch class L retired for `Session`/`User` (those two carry branded `_id` IN the model, so no cast is needed — `auth.ts:175` reads `user._id` directly with zero cast). The fix is the mirror image of L.W2: stop omitting `_id` from the model and instead type it. `mongodb` ships `WithId<T>` (`T & { _id: InferIdType<T> }`) — replacing the 25 `as Model & { _id: unknown }` with formatters that accept `WithId<Model>` and repository reads that already return `WithId<Model>` collapses all 25 to zero. **This is the single highest win-per-LoC simplification in the api** (see §5).

## 2. Scale — hot endpoints, indexes vs query shapes

**Index count = 26** (verified `grep -c createIndex src/db.ts`), matches the doc. But three filter fields that the HOT public-list path uses on EVERY query have **no index**:

### P1 — `listPalettes` public filter has no covering index (the hottest read path)
`crud-list.ts:85` unconditionally sets `f.deletedAt = null`, and `crud-list.ts:113` sets `f.visibility = "public"` for every anonymous/foreign view (the overwhelmingly common case). Neither `deletedAt` nor `visibility` is indexed (`grep -n visibility src/db.ts` → none; `deletedAt` → none). The three sort indexes are `{createdAt:-1}`, `{voteCount:-1,createdAt:-1}`, `{forkCount:-1,createdAt:-1}` — none carries `deletedAt`/`visibility` as a prefix. So the public browse (the single most-hit endpoint) walks the sort index in order and **filters `deletedAt`/`visibility` in memory** for every candidate doc until it fills a page. With a healthy ratio of live-public docs this is tolerable, but it degrades linearly as soft-deleted + private docs accumulate (they are exactly the docs the reaper leaves around for 30 days). The right shape is a compound index lead by the equality predicates: `{ visibility: 1, deletedAt: 1, createdAt: -1 }` (+ the voteCount/forkCount variants), so the common "public, live, newest" query is a pure index range scan. **Recommend N-wave: add the three `(visibility, deletedAt, <sortkey>)` compound indexes; they make the busiest endpoint index-covered.**

### P2 — `tier` filter unindexed
`crud-list.ts:90` `if (query.tier) f.tier = query.tier` — `tier` has no index. Lower traffic than visibility but same class; folds into the compound above if `tier` is appended.

### P2 — `findEmptyUserSlugs` + `aggregateUsersWithPaletteCount` `$lookup` on `palettes.userSlug`
`user.ts:99` and `user.ts:130` `$lookup` from `palettes` on `foreignField: "userSlug"`. There IS a `{userSlug:1,createdAt:-1}` index on palettes (`db.ts:44`) whose `userSlug` prefix the `$lookup` can use — so this is **adequately indexed** (no action). Noting it because the prune-empty + admin-users-list aggregations are the heaviest admin reads; they are fine.

### P2 — `migrations/check.ts` full-collection scan at every startup
`check.ts:83` `db.collection("palettes").find({}, {projection})` iterates **the entire palettes collection** on every process boot to assert schema invariants (capped at 100 surfaced violations but the cursor still walks all docs). On a large collection this adds unbounded startup latency before the server accepts connections (`index.ts:147` awaits it pre-`serve`). For a smoke probe, sampling (e.g. `.limit(1000)` or a `$sample`) gives the same regression-catch with bounded cost. **Low urgency but it is a genuine O(collection) op on the critical-boot path.**

### P2 — color-distance filter (`colorL/A/B`) is a post-fetch in-memory page filter → silent under-return
`crud-list.ts:166-185`: after fetching ONE page (limit+1) the handler filters by OKLab distance in JS (`data.filter(...)`). This means a `?colorL=..&colorRadius=..` query returns "however many of THIS page's ≤20 docs happen to match" — not the top-N matching docs. `nextCursor`/`total` reflect the pre-filter page, so a client paging through gets sparse, inconsistent pages and may see an empty `data` with `hasMore:true`. This is a correctness-of-results bug masquerading as a filter. The honest fix is a server-side `$geoWithin`-style bound on the denormalized `oklabColors` (Mongo can't index arbitrary 3-vectors natively, but a bounding-box `$elemMatch` on `oklabColors.{L,a,b}` ranges narrows the candidate set in-index before the exact-distance refine). **P2 because the feature is niche, but flag it: the current shape silently lies about result completeness.**

### No N+1 patterns found (good)
The list path batches the voted-set with ONE `findByUserAndPaletteSlugs($in: slugs)` (`crud-list.ts:149`) rather than per-row vote lookups — correct. `listMine`/`listForks`/`listVersions` use `Promise.all([data, count])` (two queries, not N). The provenance walk (`forks.ts:228`) and the diff chain walk (`diff.ts:33`) are bounded sequential reads (≤50) — these ARE per-step `findByHash`/`findBySlug` loops, i.e. up to 50 round-trips each, but bounded and on cold paths. Acceptable.

### Payload sizes — bounded (good)
`colorsArraySchema = .min(1).max(50)`, `css ≤ 200`, name `≤ 100`, tags `≤ 10 × 30`, body limit 64 KB (`index.ts:49`). `listUserPalettes` fetches a hard-coded `findByUserSlug(slug, 0, 1000)` window (`admin/users.ts:88`) with NO pagination — an admin viewing a 1000+ palette user truncates silently. P2.

## 3. Correctness — TOCTOU + transaction gaps

### P1 — orphaned-vote sweep TOCTOU (the L-close flag, STILL LIVE)
`cron.ts:50-51`:
```
const paletteSlugs = await palettes.listAllSlugs();
const orphanedVotes = await votes.deleteOrphaned(paletteSlugs);   // deleteMany({paletteSlug:{$nin: validSlugs}})
```
`deleteOrphaned` (`vote.ts:98`) deletes every vote whose `paletteSlug ∉ validSlugs`. The snapshot `listAllSlugs()` is taken, THEN the `$nin` delete runs — un-sessioned, NOT in a transaction. A palette created *between* the `distinct("slug")` read and the `deleteMany` has its slug absent from `paletteSlugs`, so any vote cast on that brand-new palette in that window is **deleted as "orphaned"** even though the palette exists. This is the exact orphaned-vote-sweep TOCTOU flagged at L-close. Worse, this sweep runs AFTER the reaper loop (`cron.ts:39-48`) which itself takes time proportional to the grace-expired set — widening the window. Two viable fixes: (a) scope the orphan-delete to votes older than the sweep-start timestamp (`createdAt < sweepStart`), so a vote on a just-created palette is never in range; or (b) delete by the inverse — enumerate vote `paletteSlug`s via `distinct`, diff against live slugs, delete only the confirmed-dead set. (a) is the smaller change. **P1: this can silently destroy legitimate votes; it is data loss, not just a stale read.**

### P1 — `decrementForkCount`/`incrementForkCount` asymmetry on delete/restore (latent double-count)
`crud.ts:253` soft-delete decrements the parent's forkCount; `crud.ts:286` restore increments it back. `decrementForkCount` (`palette.ts:169`) is GATED `{slug, forkCount:{$gt:0}}` (won't go negative — good), but `incrementForkCount` (`palette.ts:179`) is UNGATED. Sequence: soft-delete a fork when parent forkCount is already 0 (decrement no-ops, count stays 0) → restore it → increment fires → parent forkCount = 1 with the same single live fork → the count is now permanently inflated by 1. The decrement's `$gt:0` guard, intended to prevent underflow, breaks the delete/restore round-trip's symmetry. **P1 (silent counter drift on a public-facing number).** Fix: make the pair symmetric — either both gated by a recomputed truth, or recompute `forkCount` from `countForksOf` on restore rather than blind-increment.

### P2 — PATCH / publish / vote ETag TOCTOU (accepted, documented, ledger #16)
`crud.ts` PATCH (`routes/palettes/crud.ts:118`) reads the ETag pre-write but `patchPalette` does NO in-txn If-Match re-validation; same for `setVisibility` (`publish.ts:33`). A concurrent write between the route's ETag read and the service write is not fenced. This is explicitly documented as accepted (ledger #16, comments at `ownership.ts:42` and both call sites). Noting for completeness — it is a known, bounded window, not a regression. If N wants to close it, the move is a conditional update: `updateOne({slug, currentHash: <expected>}, ...)` and 412 on `matchedCount===0`. P2 only because it's already a conscious decision.

### P2 — `restorePalette` reads `palette.forkOf` from the PRE-transaction snapshot
`crud.ts:268-292`: the `forkOf` used to decide the increment is read OUTSIDE the txn (`crud.ts:273`), then the increment fires inside. If the palette's `forkOf` changed (it can't today — `forkOf` is immutable post-create), this would be stale. Currently safe because `forkOf` is write-once. Noting as a latent coupling, not a live bug.

### Transactions — coverage is genuinely complete
Every cross-collection write I traced is wrapped: create/patch/delete/restore palette, remix/fork, revert, toggleVote, register/login, setUserStatus(suspend), deleteUser, deleteUserPalettes, pruneEmptyUsers, batchPalettes(delete), batchUsers(suspend), the reaper. `flagPalette` (`flags.ts`) and `setVisibility`/feature-toggle are correctly single-collection and deliberately un-wrapped. The remix in-txn source re-read (`forks.ts:111`) genuinely closes the fork-of-deleted-source race. This part of the H1/L discipline is solid.

## 4. atomdiff / remix / publish subsystem — quality

**High quality, the cleanest part of the api.** `lib/crud/atomdiff.ts` `diffAtoms` (96 LoC) is pure, deterministic, keyed by `position`, with a closed 3-op vocabulary (`added`/`removed`/`changed`, no `moved`), sorted output, and a documented symmetry property (`diffAtoms(B,A)` inverts `diffAtoms(A,B)`). `remixPalette` is the single code path; `forkPalette` is a genuine semantic alias (`colors: undefined` → empty diff), NOT a back-compat shim — exactly the "one code path, no legacy shim" discipline the memory feedback demands. `/diff` (`services/palette/diff.ts`) is a derived projection (recomputed on demand, never stored — inv-J-2), correctly chain-only (422 on divergent branches via the bounded `isAncestor` walk), and the route ETags the immutable `(fromHash,toHash)` pair with `If-None-Match` 304 short-circuit (`diff.ts:31`). publish/unpublish (`visibility.ts`) is a correct idempotent in-place `$set` with the already-at-target no-op short-circuit. **One nit:** `diff.ts:77-78` does TWO bounded chain walks (`isAncestor(to,from) || isAncestor(from,to)`), each up to 50 `findByHash` round-trips — worst case 100 sequential reads for a divergent pair. Cold path, acceptable, but a single walk that records the visited set and checks membership both directions would halve it. P2.

**One real bug in the diff/remix `_id` handling:** `diff.ts:77` calls `isAncestor(services, toVersion._id, fromVersion._id)` — `PaletteVersion._id` IS typed (string content-hash, declared at `models.ts:155`), so this is correct and cast-free. Good — versions, like sessions/users, carry a typed `_id` and need no WithId cast. This is the proof that the §1 fix works: the collections that type `_id` have zero casts; the four that omit it carry all 25.

## 5. Simplification transpositions, ranked by win-per-LoC

1. **`WithId<Model>` — retire all 25 `_id` intersection casts (HIGHEST win/LoC).** Stop omitting `_id` from `Palette`/`ProposedName`/`Tag`/`AdminAuditEvent` in `models.ts`; type formatters + the ~13 annotation sites against `mongodb`'s `WithId<T>`. Repository reads already return `WithId<T>`, so the casts become identity and delete themselves. ~25 cast deletions + 4 model-comment edits + ~13 signature swaps, net negative LoC, zero behaviour change. Mirrors L.W2 exactly (the branded-`_id` move that retired the Session/User casts). This is the obvious tranche-N/M.W2 lane.

2. **Three `(visibility, deletedAt, <sortkey>)` compound indexes (highest win/LoC on SCALE).** ~6 lines in `db.ts` makes the single busiest endpoint (public browse) index-covered instead of in-memory-filtered. Pure additive, no code change.

3. **Orphaned-vote sweep: bound by `createdAt < sweepStart`.** ~2 lines (`cron.ts` + `vote.ts:deleteOrphaned` filter) closes a P1 data-loss TOCTOU.

4. **Symmetric fork-count on restore: recompute from `countForksOf` instead of blind `incrementForkCount`.** ~3 lines, closes the P1 counter-drift.

5. **`migrations/check.ts`: `$sample`/`.limit(1000)` the probe.** ~1 line, removes an O(collection) op from the boot path.

6. **`diff.ts` single bidirectional chain walk.** ~10 lines, halves worst-case diff latency. Lowest priority (cold path).

**Nothing is contrived or dead.** The LRU is genuinely shared (rate-limit + suspended-cache + idempotency, 3 consumers). The DI seam earns its keep (cron + request both use `getServices`). The error hierarchy is RFC-7807-correct and not over-built. The `assertVisibilityTransition` "dead guard made live" (`visibility.ts:35`) is honest about being a near-no-op composition point. No god modules (largest 296 < 350). The api is in good shape; the work is the WithId census retirement (cohesion) + three indexes + two TOCTOU closures (correctness/scale).
