#### A2 — Maximal-munch unit classifier (latent boundary correctness)
**Asked:** Wave A2 in `valuejs-sota-handoff-v2.md`. Replace per-dimension `any(...UNITS.map(istring))`
with one maximal-munch `[a-zA-Z]+` token validated against a `Set`. Both a correctness fix
(non-anchored end on the `y`-flagged regex let `100vhx` tokenize as `vh`) and a perf win.  
**Shipped:** `e684bce` (`src/parsing/units.ts`, `src/parsing/utils.ts`). Verified live:
`src/parsing/units.ts:20-25` uses `utils.unitParser(LENGTH_UNITS)` (not `any(...istring)`).
The `unitParser` helper is in `src/parsing/utils.ts`.  
**Verdict: SHIPPED-AS-ASKED.**

#### C5 — Fix 24 no-op relative length units (wrong-pixel bug)
**Asked:** Wave C5 — 24 of 45 declared relative length units (the `sv*`/`lv*`/`dv*` 18-member
family, `vi`, `vb`, `cap`, `ic`, `lh`, `rlh`) fell through to `convertAbsoluteUnitToPixels`
returning the raw number unchanged. `50dvh` → `50px`, silent wrong pixels.  
**Shipped:** `8383bd8` (`src/units/utils.ts`, `test/tranche-f.test.ts`). Verified live:
`src/units/utils.ts:258-354` has `convertViewportUnitToPixels` (covering `sv*`/`lv*`/`dv*`/
`vi`/`vb`) and `convertFontMetricUnitToPixels` (covering `cap`, `ic`, `lh`, `rlh`) plus the
C5 fail-loud guard at the declared-unit-with-no-branch path.  
**Verdict: SHIPPED-AS-ASKED.** The count correction (`24 of 45`) is confirmed by the charter
and validated in the commit message.

#### B1b — formatColor omits `/ alpha` at alpha=1
**Asked:** Wave B1b in handoff-v2 §B. `formatColor` unconditionally emitted `/ ${alpha}` even at
alpha=1. Emit only when `Number(alpha) !== 1`.

`color/index.ts` now have `if (Number(alpha) === 1) { return ... }` before the `/ ${alpha}`
form. The commit updated 27 parser snapshots + 4 color-class test assertions.  
**Verdict: SHIPPED-AS-ASKED.**

was authored as the gate substrate, measuring 2.41× median (the charter asked ≥2×). The commit
message states: "MEASURED: median 2.42× on a representative pool."  
**Verdict: SHIPPED-AS-ASKED.** Measured at 2.42× vs the asked 2.41× (lane re-measured slightly
differently).

### 3.2 Does value.js 0.11.2 satisfy this constraint?

**Yes.** `^0.11.2` requires `>= 0.11.2 < 0.12.0`. The current published version on the branch
is `0.11.2` (verified: `package.json:version`). Constraint satisfied exactly at the floor.

### 3.3 value.js parse-that sub-dependency

value.js at `0.11.2` declares `"@mkbabb/parse-that": "^0.8.2"` (verified in `package.json`).
The installed version is `0.8.2` (verified from `package-lock.json`). This creates a
**realm-convergence gap**: keyframes uses parse-that `^0.9.0` (per `package.json`) while
value.js's runtime `require` resolves to `0.8.2` from within value.js's own `node_modules`.
This was surfaced non-gating in kf G.W2 (`proof:deps-current`) and documented as a
value.js-owned fix (value.js should re-pin its own parse-that to `^0.9.x`). As of 0.11.2, the
realm divergence persists — value.js has NOT been updated to `^0.9.0`.

**This is a KNOWN OPEN ITEM** documented in kf G.W2 and carried as a non-blocking hygiene gap.
It does not prevent kf from consuming value.js correctly (the API surface is shared through the
`lerpValue → iv._lerp` seam, not through parse-that's internals).
