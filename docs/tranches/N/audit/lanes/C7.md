## P1 â€” dock-convergence divergence (the core C7 cross-repo assessment)

The DOCK-ANIMATION-CONVERGENCE.md verdict: *"the constellation does not need a new
animation system; it needs to USE the one it ships"* â€” `SpringProgress` /
`--spring-*` from keyframes.js, consumed by the glass-ui dock. value.js was explicitly
**BOOKED** in that spec (Â§7: dirty tree â†’ owner executes the dock-consumer alignment
on a clean checkout). It has **not** been executed. Concrete divergences:

### F-C7-2 [P1] value.js forks the OLD CSS-FLIP `useLayerTransition`; glass-ui now ships the spring one

- value.js: `dock/composables/useLayerTransition.ts` is a **local fork** that drives
  the layer-width morph with `el.style.transition` + `el.style.width` +
  `setTimeout(â€¦, 400)` â€” a plain CSS transition, **no spring, no velocity continuity,
  no interruptible re-base**. Its header comment claims glass-ui "exposes only
  `currentLayer`/`leavingLayer` refs" â€” that justification is now **STALE**.
- glass-ui 3.10.1 `dock/composables/useLayerTransition.ts` is the **single-scalar
  `SpringProgress` morph** (`DOCK_SPRING = {response 0.32, dampingFraction 0.7}`,
  writes `--dock-morph-t`, velocity-continuous interruptible retarget â€” the exact iOS
  spring the convergence headline prescribes). It **is re-exported on the `/dock`
  subpath** (`node_modules/.../dock/index.d.ts` exports `useLayerTransition` +
  `UseLayerTransitionOptions`/`Return`), and the glass-ui barrel comment names
  value.js's fork as the open debt: *"value.js maintains a local FLIP-width fork ONLY
  because this barrel never re-exported the primitive; with the export landed the
  consumer-adoption leg (value.js deletes its fork + re-points ActionBarLayer.vue)
  routes to AX.W34."*
- Dependency is already in place: `package.json:69` pins
  `@mkbabb/keyframes.js: file:../keyframes.js`, and glass-ui's dock spring consumes it
  â€” so adoption is **not** dep-blocked.

> Gestalt fix: delete `dock/composables/useLayerTransition.ts`; import
> `useLayerTransition` from `@mkbabb/glass-ui/dock`; re-point `ActionBarLayer.vue`
> (the sole consumer, line 9/57) to the published primitive. The sub-layer
> actionsâ†”input swap then rides the same `SpringProgress` clock as the dock's own
> collapse morph (one engine, one clock) instead of a 400ms CSS timeout.

### F-C7-3 [P1] dock chrome animations use plain `--ease-standard` CSS transitions, not the spring

The dock's own micro-motion is hand-rolled bezier, diverging from the spring motif:
- `Dock.vue:218-221` â€” the action-bar-toggle reveal slot:
  `transition: grid-template-columns var(--duration-normal) var(--ease-standard), opacity â€¦`.
- `ActionBarLayer.vue:118-121` â€” the toggle-icon crossfade:
  `transition: opacity var(--duration-fast) var(--ease-standard), transform â€¦`.

These are the "+27.5% overshoot / no-settle bezier" class the convergence audit
flagged on the dock's *most-seen* morph. value.js DOES use `var(--ease-spring)` in
4 NON-dock sites (`GradientStopEditor.vue`, `ImageEyedropper.vue`, `PaletteSlugBar.vue`,
`PaletteCard.vue:361`) â€” so the spring ease token is already in the contract; the dock
is the inconsistent surface.

> Gestalt fix: route the action-bar slot reveal + the toggle-icon crossfade through
> `var(--spring-*)` (or the `SpringProgress` scalar once F-C7-2 lands), so the dock's
> chrome motion matches both the rest of the app and the upstream dock spring.

### F-C7-4 [P1] value.js uses raw `.dock-separator` divs, not the `<DockSeparator>` primitive

glass-ui 3.10.1 ships `<DockSeparator>` â€” an orientation-aware divider with
`role="separator"` + `aria-orientation` + axis-perpendicular paint (the bare class is
"axis-blind" per the primitive's own doc). value.js hand-writes
`<div class="dock-separator"></div>` at **4 dock sites** (`Dock.vue:100,113,134,161,173`,
`ActionBarLayer.vue:88`, `SlugEditLayer.vue:98`, `ProfileSection.vue:35,91`) + 3 more
outside the dock. The `.dock-separator` class is still token-backed (glass-ui
`dist/styles/dock/shell.css` defines it), so it **renders** â€” but it carries no
`role="separator"`/`aria-orientation` semantics, so the dock's group structure is
invisible to assistive tech. This is the glass-ui-first-class violation
(`feedback_glass_ui_first_class.md`): consume the primitive, don't hand-roll.

> Gestalt fix: import `DockSeparator` from `@mkbabb/glass-ui/dock`; replace the raw
> `<div class="dock-separator">` sites with `<DockSeparator />`.

### F-C7-5 [P2] the `useId` route-VT seam + the persistent-rail idiom are unused leverage

glass-ui's `GlassDock` now exposes a `#persistent` slot (the iOS Now-Playing
always-present rail), a `#rail` chrome slot (`<DockRail>`, survives collapse), and a
per-instance `view-transition-name` route-morph seam. value.js's dock uses none of
them; its admin/view context is carried entirely inside the collapsible `--full` pane,
so on collapse the whole control set vanishes to the pill. Not a bug, but the
convergence's `<Role>Dock` slot-filler convention (Â§4) is the booked direction and the
persistent-rail would let the view indicator survive collapse.
