## 1. Session / anon-user model

### Current shape
- `register` (`session/auth.ts:56`) generates a unique slug (`generateUniqueSlug`), then **in a transaction** inserts a `users` row (`_id = slug`) AND a `sessions` row (`_id = uuid token`).
- `login` inserts a session + `users.touchLastSeen` (transaction).
- Every "anonymous" visitor who saves a palette becomes a **real `users` document** with a slug. There is no separate anon vs registered distinction ‚Äî a user is a slug, full stop. Sessions name a `userSlug`. (`models.ts:189-197`, `259-265`.)
- `resolve-session.ts` reads `X-Session-Token` ‚Üí `findAndTouch` (atomic find + `lastSeenAt` bump, gated `expiresAt > now`) ‚Üí if `userSlug`, a 60s LRU-cached suspended check.

### Assessment
This is **already simple and correct**. The slug-as-_id branding (L.W2) removed the ObjectId casts. The two-collection split (users / sessions) is justified: a user outlives its sessions; sessions carry IP + expiry. Do not collapse them.

### Proposed simpler shape (one real win)
- **`register` does not need a transaction.** The two writes are `users.insert` then `sessions.insert`. A partial failure leaves an orphan `users` row with no session ‚Äî which is *exactly what `login` already tolerates* (login re-uses an existing user with a fresh session, and `prune-empty` reaps zero-palette users). The orphan-user failure mode is benign and already swept. The transaction here buys atomicity for a pair where the non-atomic outcome is a no-op-recoverable orphan. **Contrast** `createPalette`/`fork`/`revert` where the orphan is a *version row pointing at a palette that never landed* (a real invariant break) ‚Äî those must stay wrapped.
  - **DELETES:** the `withTransaction` wrapper in `registerSession` (auth.ts:75-99) ‚Äî 16 ‚Üí 15 sites. The H1-coverage doc row for `registerSession` moves to DEFERRED-WITH-RATIONALE.
  - **Risk:** LOW. The orphan user is indistinguishable from a registered-then-never-saved user; `prune-empty` already handles it. Flag, do not force.
  - **Honest caveat:** `loginSession`'s txn (session insert + `touchLastSeen`) has the *same* benign-orphan property and is also a candidate. Keeping both wrapped is defensible "uniform call site" hygiene; this is a judgment call, not a defect. Recommend collapsing both OR neither, and documenting the rule ("wrap iff the non-atomic outcome is a cross-collection *referential* break, not a recoverable orphan").

### Session expiry ‚Äî see ¬ß10 (the TTL-index win belongs there).

## 2. Idempotency storage

### Current shape
`middleware/idempotency.ts` ‚Äî opt-in `Idempotency-Key` replay store backed by the in-process `LRU` (50k cap, 24h TTL). Captures only 2xx (non-204) responses; scopes key by `identity:method:path:key`. **Per-process, does not survive restart, does not span replicas** (documented, deliberate ‚Äî mirrors rate-limit + suspended-cache).

### Assessment
This is **honest and correctly bounded** for its stated job ("collapse client retries within a request burst"). It is NOT a Mongo collection ‚Äî good. The K.W2 adversarial review already caught the cache-errors bug. No change needed to the *mechanism*.

### Proposed simpler shape (de-risk, not re-architect)
- The replay store buffers full response **bodies** in memory (up to 50k entries). For a 64KB body-limit that is a 3.2GB worst-case ceiling ‚Äî unrealistic but unbounded-by-body-size. **Flag:** cap stored body size (e.g. skip-capture > 16KB; a large successful create that won't be retried with the same key is the rare case). This is a hardening note, not a simplification.
- **No DELETE here.** The idempotency middleware is the right size. Do not Mongo-back it (that would be *more* code for cross-replica durability the use-case does not need).
- **Cohesion observation:** idempotency + rate-limit + suspended-cache are **three consumers of one `LRU` primitive with three independent eviction timers** (rate-limit runs its own `setInterval` sweep; idempotency + suspended rely on lazy eviction). This is already consolidated (D.W2 Lane D folded 3 impls ‚Üí 1 `LRU<K,V>`). No further collapse warranted ‚Äî they have genuinely different TTLs/caps.

## 3. ETag / ownership flow

### Current shape
- `etag.ts`: `paletteETag` = `"currentHash"` (or `updatedAt.toISOString()` fallback); `assertIfMatch` enforces 428-absent / 412-stale on PATCH/publish/unpublish.
- Ownership: route-level `requireOwnership(paletteOwnerExtractor)` middleware ‚Üí `getOwnerSlug` service helper ‚Üí `repositories.palettes.findBySlug`.
- The PATCH/publish flow does **two reads** of the same palette: one in the route for the ETag pre-check (`getPaletteETagData`), one in the service (`patchPalette`/`setVisibility` re-read). Plus `requireOwnership` did a *third* read via `getOwnerSlug`. So a single PATCH reads the palette **3√ó** before writing.

### Assessment
The 3-read fan-out is the cost of the layered `validate ‚Üí authz ‚Üí service` pipeline + the ledger-#16 TOCTOU acknowledgement (no in-txn ETag re-validation). It is correct but wasteful.

### Proposed simpler shape
- **Collapse the ownership-read + ETag-read into one route-level palette fetch**, passed down. `requireOwnership`'s extractor and the ETag pre-check both call `findBySlug(slug)` back-to-back in the same request. Have the ownership middleware stash the fetched palette on `c.var` (e.g. `c.set("palette", doc)`) and have the ETag pre-check + service read it from there. 3 reads ‚Üí 1.
  - **DELETES:** `getOwnerSlug` and `getPaletteETagData` as separate round-trips; the service-layer re-read in `patchPalette`/`setVisibility`/`revert`/`delete` (they re-read "for the patch payload" but the middleware already has the doc).
  - **Less code:** removes 2 of the 3 `findBySlug` call chains per owner-gated write; `ownership.ts` (47 LoC) shrinks toward a single extractor.
  - **Better scale:** one indexed `{slug}` lookup per PATCH instead of three.
  - **Risk:** MEDIUM. The narrow TOCTOU window (ledger #16) is *unchanged* ‚Äî still no in-txn re-validation ‚Äî but the read is now even earlier, so the window widens slightly. This is acceptable given the existing accepted-TOCTOU stance, but must be documented. The bigger risk is plumbing the palette through `c.var` typed-cleanly (AppEnv addition). Worth it; this is the single highest-leverage CRUD simplification.
- **Keep** the 428/412 contract ‚Äî it is RFC-correct and conformance-tested. Do not weaken it.

## 4. Votes

### Current shape
`votes.ts` toggle: outside-txn existence check ‚Üí `withTransaction(delete-or-upsert + gated $inc + in-txn re-read)`. Vote rows are `(userSlug, paletteSlug, createdAt)` with a unique compound index. `voteCount` is a **denormalized counter** on the palette, kept in sync by gated `$inc`.

### Assessment
The denormalized `voteCount` + unique-index-gated upsert is the **right** design for read-heavy "popular" sorting (sort on `voteCount` index, no per-list aggregation). The transaction wrapping (E.W2 Lane B) is *belt-and-suspenders*: the unique `(userSlug,paletteSlug)` index already makes the upsert idempotent; the txn adds snapshot-isolation on the re-read. The comment itself admits the upsert pattern "doesn't depend on transactional isolation."

### Proposed simpler shape
- **Drop the transaction from `toggleVote`; keep the gated `$inc`.** The unique index is the real correctness anchor. The only thing the txn buys is that the post-update `voteCount` re-read reflects *this* increment under concurrency ‚Äî but the returned count is advisory UI state, not an invariant; a concurrent vote making it off-by-one for one response is harmless and self-heals on the next read. The `$inc` itself is atomic at the document level without a session.
  - **DELETES:** `withTransaction` in `votes.ts:45-86` ‚Üí 15 ‚Üí 14 sites; the two in-txn re-reads collapse to one post-write read (or use `findOneAndUpdate` returnDocument:after on the `$inc` to fold the re-read into the increment ‚Äî **2 round-trips ‚Üí 1**).
  - **Risk:** LOW. Worst case: a returned `voteCount` transiently off-by-one under a same-user double-click race (which the idempotency middleware + unique index already collapse anyway).
- **Sharper:** replace `incrementVoteCount` + separate `findBySlug` re-read with a single `findOneAndUpdate({slug},{$inc},{returnDocument:"after"})`. Deletes a repository method's caller round-trip.

## 5. The 9 collections + 26 indexes (collapsibility)

### Current shape
| Collection | Indexes | Notes |
|---|---|---|
| `palettes` | 8 | slug(uniq), createdAt, voteCount+createdAt, userSlug+createdAt, tags, forkOf, forkCount+createdAt, name(text) |
| `palette_versions` | 4 | paletteSlug+createdAt, forkedFromHash, rootHash, authorSlug+createdAt |
| `votes` | 2 | (userSlug,paletteSlug)uniq, paletteSlug |
| `sessions` | 2 | lastSeenAt, expiresAt |
| `proposed_names` | 3 | name(uniq), status, (name,css)text |
| `users` | 1 | createdAt |
| `tags` | 1 | name(uniq) |
| `flags` | 3 | (paletteSlug,reporterSlug)uniq, paletteSlug, createdAt |
| `admin_audit` | 2 | timestamp, action+timestamp |

### Collapse candidates (indexes)
- **`palettes.createdAt` (db.ts:42) is redundant.** Every list sort is `{createdAt:-1, _id:-1}` or `{voteCount/forkCount, createdAt, _id}`. A lone `{createdAt:-1}` index is a prefix of nothing the "newest" sort needs that `{_id:-1}` + collection scan can't serve cheaply at small N ‚Äî but more concretely, the "newest" path filters `{deletedAt:null, visibility:"public"}` first, so `createdAt` alone is rarely the selective index. **Flag for measurement, not blind deletion** ‚Äî verify with `explain()` against prod data shape; if the newest-sort always also constrains visibility/deletedAt, a `{deletedAt:1, visibility:1, createdAt:-1}` compound would serve list+count far better than the 3 separate single-field indexes currently used (`createdAt`, plus filter scans on `deletedAt`/`visibility` which have **no index at all**).
- **`palettes` has NO index on `deletedAt` or `visibility`** despite *every public list* filtering `{deletedAt:null, visibility:"public"}` (`crud-list.ts:85,114`). This is the real scale gap: the list query filters on two unindexed fields then sorts. **Proposed: replace `createdAt` (-1) with a compound `{deletedAt:1, visibility:1, createdAt:-1}`** ‚Äî net **same index count**, strictly better selectivity for the hot path. Net-zero LoC, better scale. This is the strongest index finding.
- **`palette_versions.authorSlug+createdAt` (db.ts:57)** ‚Äî confirm a consumer exists. The version routes query by `paletteSlug` (list) and `_id`/hash (get). `authorSlug` is written but I find no service read filtering versions by author. **Flag: likely droppable** (verify against admin routes). If dead ‚Üí 26 ‚Üí 25.

### Collapse candidates (collections)
- **`tags` collection is near-vestigial.** It holds `{name, category, createdAt}` and exists only for the admin tag CRUD + the public `/colors/tags` list. Palettes store tags as an inline `string[]` (`palettes.tags`, indexed). The `tags` collection is a *controlled-vocabulary registry* that the palette tag array does not actually reference (no FK; `pullTagFromAll` cascades on delete but palettes can carry tags absent from the registry). If product does not need a curated tag vocabulary, **`tags` + its 1 index + `TagRepository` + admin tags service/route could DELETE** (palette tags become free-form, already are). **Flag ‚Äî product call, not a pure refactor.** Evidence is suggestive (no referential integrity enforced) but not overwhelming.
- **`flags` cannot collapse** ‚Äî distinct lifecycle (moderation), unique reporter constraint.
- **`admin_audit` cannot collapse** ‚Äî append-only log, distinct retention.

### `forkOfHash` ‚Äî dead field (clear win)
`forkOfHash` is **written** at 3 sites (`crud.ts:103`, `import.ts:60`, `forks.ts:95`) and **passed through** `format/palette.ts` but **never read for any logic** ‚Äî no query filters or branches on it (`grep` confirms only write + passthrough). The provenance walk uses `forkOf` (the slug), not `forkOfHash`. **Proposed: DELETE `forkOfHash`** from `Palette`, `FormattedPalette`, the 3 write sites, and the formatter. Net LoC reduction, zero behavior change, no consumer. **Risk: LOW** (verify no demo type depends on it ‚Äî the formatted field is in the wire envelope, so this is a wire-contract change; gate behind a minor version). This is the cleanest pure-deletion in the CRUD surface.

## 6. Pagination

### Current shape
Two modes coexist in `listPalettes` (`crud-list.ts`): **cursor** (base64url-encoded `{_id, createdAt, voteCount?, forkCount?}`, keyset `$or` predicate) and **offset** (`skip`/`limit` + a separate `countDocuments`). The mode is selected by presence of `?cursor`. The offset path does a second `countByFilter` query (with `$or` stripped). `findManyByFilter` uses `.skip(offset).limit(limit+1)`.

### Assessment
Carrying **both** keyset and offset pagination is the contrivance. Keyset (cursor) is strictly superior for an infinite-scroll color browser: O(1) deep pages, no `skip` penalty, no count round-trip. Offset is only needed for "page N of M" UIs with a total count ‚Äî which the demo (infinite scroll) does not use for the main browse. `/mine` and `/forks` use offset but at small N.

### Proposed simpler shape
- **Make cursor the sole list pagination; drop offset from the public `/palettes` list.** Keep a lightweight `total` only where a UI genuinely shows it (admin tables). The keyset `$or` ladder (3 branches √ó 3 sort modes) is the only retained complexity and it is irreducible for correct keyset over a non-unique sort key.
  - **DELETES:** the offset branch in `listPalettes` (~30 LoC: the `offset` clamp, `findManyByFilter` offset call, the `countFilter`/`countByFilter` round-trip), and the `Math.min(...,10_000)` offset cap (an admission that deep offset is a footgun).
  - **Better scale:** removes the `skip(10000)` worst case AND the per-page count query on the hot browse path.
  - **Risk:** MEDIUM ‚Äî wire contract change (offset consumers must move to cursor). Demo already supports cursor (`listPalettes` client). Gate behind minor version. `/mine` + `/forks` (small-N, owner-scoped) can keep offset ‚Äî no scale pressure there, and forcing them to cursor would be churn for churn's sake.
- **Sharper:** the cursor encodes `voteCount`/`forkCount` unconditionally even in "newest" mode (`crud-list.ts:156-161`) ‚Äî harmless but the decode side already guards. Leave it.

## 7. Diff / remix subsystem ‚Äî the largest gestalt finding

### Current shape
- `lib/crud/atomdiff.ts` (96 LoC): pure `diffAtoms(before, after)` ‚Üí `AtomDiffOp[]`.
- `services/palette/diff.ts` (92 LoC): `computePaletteDiff` ‚Äî chain-walk ancestry (‚â§50), reject divergent (422), recompute diff on demand.
- `routes/palettes/diff.ts`: `GET /:slug/diff?from&to`, ETag-cached.
- `remixPalette` (`forks.ts:49`): fork + recorded atom-diff persisted on the version edge (`PaletteVersion.atomDiff?`).
- `forkPalette` is `remixPalette` with empty colors (ONE code path ‚Äî genuinely clean, J.W2).
- `routes/palettes/forks.ts` mounts `POST /:slug/remix`.
- **Three conformance test files** (`conformance/diff.test.ts` 245 LoC, `idempotency.test.ts` 274, `crud.test.ts` 240 = 759 LoC) keep this green.

### Assessment ‚Äî overwhelming evidence of substrate-without-consumer
- **The demo never calls `/diff` or `/remix`** (verified 0 hits). The frontend uses `fork` (no recorded diff) + `revert` + `provenance`. The atom-diff was authored for the constellation cross-repo "WAVE-D atom-diff pattern" cohesion with `fourier-analysis` (J-diff-shape), **not** for a value.js product feature.
- `PaletteVersion.atomDiff` is **persisted on every remix** but **never read back** by any value.js route (the `/diff` endpoint *recomputes* from colors and does not consult the stored `atomDiff` ‚Äî confirmed in `diff.ts`, which calls `diffAtoms(fromVersion.colors, toVersion.colors)`, NOT `version.atomDiff`). So the stored `atomDiff` field is **write-only dead storage** in this repo.

### Proposed simpler shape
Two honest options, in order of aggressiveness:

**(A) Minimal ‚Äî delete the dead storage, keep the recompute endpoint:**
- DELETE `PaletteVersion.atomDiff` (write-only, never read) + its plumbing in `versions.ts`/`forks.ts`/`models.ts`. The `/diff` endpoint is unaffected (it recomputes).
- Collapse `remixPalette` back into `forkPalette` IF the recorded-diff is the only thing `remix` adds (it is) ‚Äî `remix` and `fork` become **one route + one service fn** again. The `colors` override on fork is a 1-line addition to `forkPalette`.
- **DELETES:** `atomDiff` field + `/remix` route + `RemixInput.colors` branch's persistence; ~40 LoC + 1 route.

**(B) Aggressive ‚Äî if the constellation cohort no longer pins it (per brief, fourier-J cohort is "CORE GREEN"/closing): delete the whole diff subsystem.**
- DELETE `lib/crud/atomdiff.ts`, `services/palette/diff.ts`, `routes/palettes/diff.ts`, `PaletteVersion.atomDiff`, the `/remix` route + service branch, and `conformance/diff.test.ts`. Fold `remix`‚Üí`fork`.
- **DELETES:** ~330 LoC of source + ~245 LoC of conformance test + 1 collection field + 2 routes (`/diff`, `/remix`).
- **Risk:** HIGH on cohort-coupling ‚Äî the atom-diff is a *constellation contract artifact* (it twins fourier's `atomdiff.py`, binds to `J-diff-shape`). Deleting it unilaterally breaks the cross-repo pattern parity even though value.js has no local consumer. **This is a constellation-level decision, not a value.js-local one.** Flag it loudly: value.js is carrying ~575 LoC of source+test for a pattern with zero in-repo consumers, justified only by cross-repo symmetry. If tranche N confirms the cohort no longer needs the value.js twin, (B) is the single biggest code deletion available.

**Recommendation:** (A) unconditionally (the stored `atomDiff` is indefensible dead storage in *any* reading). (B) only after a constellation ratification that the value.js diff twin is no longer a cohort obligation.

### `provenance` walk ‚Äî minor
`getProvenance` (forks.ts:220) walks `forkOf` slug-by-slug, one `findBySlug` per ancestor (‚â§50). At depth-50 that is 50 sequential round-trips. **Flag:** a `$graphLookup` aggregation would do it in one query. Low priority (depth is small, forks are shallow in practice), but it is the one N+1 in the read path. The version-chain `isAncestor` walk in `diff.ts` has the same shape (dies with the diff subsystem under option B).

## 8. Versions subsystem ‚Äî denormalized counters

### Current shape
`versionCount` + `forkCount` are denormalized counters on the palette, `$inc`'d on create/patch/revert/fork/restore. `versionCount` is computed as `(palette.versionCount ?? 1) + 1` in patch (`crud.ts:185`) but `$inc:1` in revert (`versions.ts:182`) ‚Äî **two different update idioms for the same counter** (one `$set` a computed value, one `$inc`). This is a latent inconsistency: the patch path can drift if `versionCount` is ever null.

### Proposed simpler shape
- **Unify to `$inc: {versionCount: 1}` everywhere** (delete the `$set`-computed branch in patch). One idiom, atomic, no read-then-write race. Trivial LoC reduction + removes the `?? 1` defensive default.
- **Risk:** LOW. The migration probe already asserts `versionCount` presence... actually it does NOT (check.ts invariants don't include `versionCount`/`forkCount`+versionCount ‚Äî only `forkCount`). Minor: add `versionCount` to the probe OR rely on `$inc` (which upserts the field to the delta from absent). `$inc` is robust here.

## 9. Cron + soft-delete reaper

### Current shape
`cron.ts` (daily 3AM): deletes expired sessions, stale sessions, reaps grace-expired soft-deleted palettes (txn cascade votes+flags), deletes orphaned votes via `listAllSlugs()` (a **full-collection `distinct("slug")` scan** at `palette.ts:91`) + `deleteOrphaned($nin: allSlugs)`.

### Assessment
- The **orphaned-vote sweep is an O(palettes) scan + an O(votes) `$nin` against a potentially huge array** every night. This does not scale: `$nin: [50000 slugs]` is pathological. Orphaned votes only arise from the *hard-delete* path; the reaper already cascades votes inside its txn (`cron.ts:44`). So **the standalone orphan-vote sweep is largely redundant** with the reaper's own cascade.

### Proposed simpler shape
- **DELETE the `listAllSlugs` + `deleteOrphaned` orphan-vote sweep.** Orphaned votes are produced only when a palette is hard-deleted without cascading ‚Äî but the reaper cascades, and the admin hard-delete (`admin/palettes`) cascades too (verify). If every hard-delete cascades its votes (it should, post-H1), there are **no orphans to sweep**. The sweep is defending against a class the transactional cascades already close.
  - **DELETES:** `deleteOrphaned` (vote repo), `listAllSlugs` (palette repo), the sweep block in cron ‚Äî and removes the nightly full-collection scan.
  - **Risk:** MEDIUM ‚Äî requires auditing that ALL hard-delete paths cascade votes (reaper ‚úì, admin delete-palette ‚úì per H1 coverage, admin delete-user-palettes ‚úì). If one path does not, orphans accrue silently. Verify the H1 coverage doc enumerates them, then delete the sweep. This is "less code AND better scale" if the cascade invariant holds.

## 10. Mongo-with-transactions ‚Äî does it pull its weight? (FLAG-ONLY per brief)

### Honest assessment against actual access patterns
**Verdict: Mongo earns its place; the transactions are ~half theater but the half that's real is load-bearing. Do NOT migrate.**

Evidence FOR keeping Mongo + transactions:
- Genuine cross-collection referential invariants exist: palette+version must land together (an orphan version pointing at a missing palette is a real corruption); fork must insert child + version + bump parent atomically. These (`createPalette`, `remix/fork`, `revert`, admin cascades) **need** the transaction. ~9 of the 16 sites are defensible.
- The document model fits: a palette + its colors + oklab + denormalized counters is a natural aggregate. Color-distance filtering uses the embedded `oklabColors[]`. Text search uses Mongo `$text`. These are Mongo-idiomatic and would be *more* code on a relational store.
- The slug-as-_id, content-hash-as-_id design exploits Mongo's flexible `_id` cleanly.

Evidence that the transactions are over-applied (the "half theater"):
- `registerSession`, `loginSession`, `toggleVote` wrap pairs where the non-atomic outcome is benign/self-healing (orphan user; off-by-one advisory count) ‚Äî see ¬ß1, ¬ß4. These add session-start cost + require a replica set for *no real invariant*. ~3-4 of 16 sites are removable (drops to ~12).
- The whole thing **requires a replica set** (`inject-services.ts:56` ‚Äî single standalone Mongo throws on first txn op). That is real operational weight (`compose.yaml` runs `rs0`). The benefit must justify the rs0 requirement ‚Äî and it does, but only because of the ~9 genuine cross-collection sites.

Native Mongo primitives that would DELETE hand-rolled code (the real "Mongo pulling its weight" wins):
- **TTL index on `sessions.expiresAt` (`{expireAfterSeconds:0}`)** ‚Äî Mongo would auto-expire sessions. This **DELETES** `deleteExpired` + the cron expired-session sweep entirely. (`db.ts:65` currently indexes `expiresAt` as a plain index *for the cron sweep* ‚Äî converting it to a TTL index is a 1-line change that deletes a cron branch + a repo method.) **Strong flag ‚Äî native primitive, less code, better scale.** Same for stale sessions if `lastSeenAt` gets a TTL (30d).
- `findOneAndUpdate` returnDocument:after to fold re-reads (votes ¬ß4, visibility ¬ß3).

### Conclusion (flag-only)
Mongo stays. The actionable wins are *within* Mongo: (1) TTL index ‚Üí delete session-cleanup cron+code; (2) drop ~3-4 benign-orphan transactions ‚Üí drop the rs0-dependency surface (though rs0 stays for the genuine sites); (3) the missing `{deletedAt,visibility,createdAt}` compound index is the real scale fix. **No relational migration ‚Äî the evidence is the opposite of overwhelming; the document model fits the aggregate.**
