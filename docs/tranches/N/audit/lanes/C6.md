## P1 findings

### P1-TYPE-LADDER — The docs ignore the glass-ui φ type-scale **and** the entire audacious-display ladder
**Evidence:** `Markdown.vue:105-130` styles headings with raw Tailwind classes —
`> h1 {text-4xl font-extrabold}`, `> h2 {text-3xl}`, `> h3 {text-2xl}`, `> h4 {text-xl}`,
`> h5 {text-lg}`, `> h6 {text-base}`. `demo/DESIGN.md:21` **explicitly mandates** "Use
glass-ui's named utilities — `.text-display`, `.text-title`, `.text-heading`, `.text-prose`,
`.text-body` … instead of raw `text-2xl` etc. The φ-ratio scale … is the canonical step
ladder." The markdown renderer is the single largest body of prose in the app and it violates
this house rule wholesale. Worse: glass-ui 3.10.1 ships a full poster ladder —
`--type-display-1` (→2.618rem) through `--type-display-audacious`
(`clamp(8.728rem … 22rem)`, "Fast.com-scale single number") in `dist/styles/typography.css` +
`DESIGN.md:594-601`. **None** is used. The most "audacious typography" the docs reach is
`text-4xl` = Tailwind's flat **2.25rem**, which isn't even mapped onto the display tokens.
**Pixel proof:** `shots/C6-H1-harness-oklch-light.png` — "Detailed Guide" (the hardcoded
`text-title` h2 in `AboutPane.vue:27`, 2.058rem) is the single largest glyph on the page; the
in-markdown headings are all smaller. The math motif's home page has no poster type.
**Gestalt fix:** re-author `Markdown.vue` heading CSS onto the φ ladder — give the page-title
(or a synthesized doc title, see P1-H1-DEAD) a `--type-display-1/2` poster treatment in
Fraunces' high optical-size + 900 weight axis (the variable font is already loaded with
`opsz 9..144; wght 100..900`, index.html:14), and route h2/h3/body through
`.text-heading`/`.text-subheading`/`.text-prose`. One audacious display headline per page
anchors the math motif instead of letting it open flat.

### P1-PROSE-TOKEN — Long-form prose ignores `--type-prose`, the token built for exactly this
**Evidence:** `Markdown.vue:71-72` sets `.markdown-body {@apply text-base leading-7}` (16px),
and `> p {@apply mb-4 leading-7}` — i.e. 16px body. glass-ui ships `--type-prose:
clamp(1.125rem, 1.04rem + 0.28vw, 1.5rem)` (18px floor) with the comment "long-form prose"
(`typography.css`) and `.text-prose` exposes it (`components.css`). The one surface that *is*
long-form prose uses the 16px body cascade instead of the 18px prose token. Reading the
historical-context paragraphs (`oklch.md:23`, ~3 sentences of dense color science) at 16px in
a 30rem column is tighter than the design system intends.
**Gestalt fix:** `.markdown-body` → `font-size: var(--type-prose); line-height:
var(--type-leading-prose)`. Free, token-correct, and visibly improves the read.

### P1-INLINE-CODE-EATEN — Inline `code` is force-rendered as KaTeX math; component labels and code calls become italic math glyphs
**Evidence:** `useMarkdownHighlighting.ts:74-93` (`renderKatex`) walks **every** `<code>` with
no `className` and replaces it with KaTeX (`displayMode:false`). But the docs use bare
backtick code for two unrelated purposes: (a) *real* inline math — `oklch.md` doesn't, but
`lab.md:60` `` `L^* = 0 \text{ (black) to } 100` `` is genuine LaTeX — and (b) *plain code
labels* — `oklch.md:17-19` `` `L` `` `` `C` `` `` `H` ``, `oklch.md:32` `` `oklch()` ``,
`oklch.md:46` `` `0`–`1` ``, `oklch.md:62` `` `C = sqrt(a^2 + b^2)` ``. Because both share the
backtick syntax and `renderKatex` can't tell them apart, the *labels* get rendered as math:
`L`/`C`/`H` become italic math variables and `oklch()` becomes math `oklch()`.
**Pixel proof:** `shots/C6-H1-harness-oklch-light.png` — the Attributes list shows *L, C, H*
as serif-italic KaTeX glyphs with **no code background, no mono font**; "CSS-native"'s
`oklch()` is italic math, not the styled inline code DESIGN.md intends. The careful
`p > code {text-primary bg-muted rounded px-1}` rule (`Markdown.vue:185-188`) **never fires** —
its targets are deleted before paint. The code-vs-math distinction (a core part of the
math/code motif) collapses into one ambiguous visual register.
**Gestalt fix:** stop overloading backticks. Adopt an explicit inline-math delimiter (the docs
are Vue SFCs — use `<Katex :display-mode="false">` for inline math as `lab.md:84` already
does, or a `$…$` convention the loader transforms) and let plain backtick code render as code.
This is the single highest-leverage fix for the "is this math or code?" incoherence.

### P1-MATHML — KaTeX is forced to `output: "mathml"`, bypassing its own CSS metrics and `katex.min.css`
**Evidence:** `Katex.vue:21` and `useMarkdownHighlighting.ts:89` both pass
`output: "mathml"`. MathML delegates glyph layout to the **browser's native math renderer**,
which (i) varies across Chrome/Safari/Firefox, (ii) typically uses the system math/serif font,
and (iii) does **not** consume KaTeX's hand-tuned HTML/CSS box metrics. Yet `Katex.vue:33`
still `@import "katex/dist/katex.min.css"` — that stylesheet is almost entirely
HTML-renderer-specific and is largely inert under MathML output. So the docs pay the CSS
weight without the typographic payoff, and the rendered math looks like *system default* math,
not the polished KaTeX house look. (My harness used KaTeX HTML output and therefore renders
*better* than the real app — treat `shots/C6-H1` as the optimistic ceiling.)
**Gestalt fix:** pick one. Either switch to `output: "html"` (or `"htmlAndMathml"`) so the
imported `katex.min.css` actually governs the look and the math is consistent + branded across
browsers; or commit to MathML and drop the dead CSS import + add explicit `math {}` styling
(font, color-accent) so the motif is intentional. Today it is neither.

### P1-MATH-UNACCENTED — Headings get a derived OKLCh accent; the math gets nothing, so the motif reads generic
**Evidence:** `useMarkdownColors.ts:44-48` derives `--md-color-h2/h3/accent` from the live
color and applies them to h2/h3 + `mark.cs-name` (`Markdown.vue:111,116,133-137`). The display
math (`<Katex>` blocks) inherits plain `foreground` (black in light / white in dark) — see
`shots/C6-H1` (light) and `C6-H2` (dark): every matrix, fraction, and equation is flat
black/white while the headings glow teal/blue. In the app whose *entire pitch* is color, the
mathematics — the thing the page exists to show — is the one un-colored element.
**Gestalt fix:** thread the same derived accent into the display-math tier (e.g. operators,
matrix brackets, or a left rule on `<Katex>` blocks tinted with `--md-color-accent`), or give
math blocks a tinted glass mini-surface. Make the math a colorful pop, not monochrome filler.

## P2 findings

### P2-H1-DEAD — The most audacious heading style (`> h1 {text-4xl font-extrabold}`) is dead code
**Evidence:** all 11 docs start at `### Attributes` (h3); `grep -c '^# '` = 0 across
`assets/docs/*.md`. So `Markdown.vue:105-107` (`> h1 {text-4xl font-extrabold}`) **never
renders**. The visual top of each doc is the hardcoded "Detailed Guide" h2 + an h2/h3 jump,
with the h1 tier — the page's would-be hero — permanently unused. The hierarchy opens on a
mid-tier heading.
**Gestalt fix:** synthesize a real per-space page hero (the space's display name, e.g.
"OKLCh", from `colorSpaceInfo`/`DISPLAY_COLOR_SPACE_NAMES`) as an h1 in the audacious display
tier (ties to P1-TYPE-LADDER), so the dead style becomes the anchor it was written to be —
and the doc reads as a *titled reference page*, not an untitled fragment.

### P2-MEASURE-VS-MATH — The 30rem reading column is good for prose but crowds wide display math
**Evidence:** `AboutPane.vue:6` caps the card at `lg:max-w-desktop-pane` =
`--desktop-pane-max-w: 30rem` (`style.css:96`). Prose at 30rem ≈ 60-65ch is ideal. But the
widest display math — `oklch.md:88` (the `M_1^{-1}`/`M_2^{-1}` double-matrix) and
`oklch.md:96` (the XYZ→OKLab double-matrix) — barely fit and trigger
`overflow-x:auto` (`Markdown.vue:222-225`). Pixel: `shots/C6-H1` "XYZ to OKLab" — the two
bracketed matrices nearly touch both card edges. On a real MathML render (wider glyphs) they
will horizontal-scroll, which is a poor reading experience for the centerpiece equations.
**Gestalt fix:** let display-math blocks break the prose measure — a `.markdown-body >
div:has(.katex)` rule that bleeds to a wider `max-w` (or full card width with internal
padding) while the prose stays at the 60ch optimum. Math wants width; prose wants narrowness;
one column can't serve both.

### P2-NO-NAV — Zero navigation among the 11 reference pages; no index, no cross-links, no prev/next
**Evidence:** `grep` for intra-doc links across `assets/docs/*.md` returns only external links
(Ottosson's blog, etc.) — no link points to another color-space page. The only way to move
between the 11 docs is the `ColorSpaceSelector` dropdown in the pane header, which **also
mutates the active picker color** (`AboutPane.vue:10-16`, shared `selectedColorSpace`). The
conversion sections constantly name sibling spaces ("OKLCh → OKLab → XYZ", `oklch.md:44`) but
never link to those pages. For a 10/11-page reference set this is a discoverability and
gestalt gap — it reads as one rotating page, not a navigable reference.
**Gestalt fix:** add a lightweight "related spaces" chip row at the foot of each doc
(linking the conversion-chain neighbors), and/or a compact doc index. Decouple "I'm reading
about OKLCh" from "my working color is OKLCh" so browsing the reference doesn't hijack the
picker.

### P2-CONTENT-VIS-MATH — `content-visibility:auto` on every non-first child risks math reflow jank
**Evidence:** `Markdown.vue:76-80` applies `content-visibility:auto; contain-intrinsic-size:
auto 200px` to `> *:not(:first-child)`. KaTeX/MathML blocks and code `<pre>` are far taller
than the 200px estimate; as the user scrolls, off-screen math/code blocks render and the
placeholder→actual swap can shift layout (and the `applyHighlighting()` in `onUpdated`,
`Markdown.vue:63-65`, re-renders KaTeX each update). Combined with `output:"mathml"` (slow
first layout in some engines) this is a measurable scroll-jank risk on the math-heavy docs
(`xyz.md`, `lab.md`, `oklch.md`).
**Gestalt fix:** give math/code blocks a realistic `contain-intrinsic-size` (or exempt them),
and render KaTeX once on mount rather than on every `onUpdated`.

### P2-CODE-FORMATTER-UNUSED — `useCodeFormatting` (Prettier + live hljs) exists but `Markdown.vue` never calls it
**Evidence:** `composables/useCodeFormatting.ts` is a full Prettier-reflow + highlight.js
pipeline (responsive `printWidth`, resize re-format), but `Markdown.vue` imports only
`useMarkdownColors` + `useMarkdownHighlighting` (lines 31-32, 42-43) — `useCodeFormatting` is
never wired in. Code blocks are instead pre-formatted/highlighted at *build* time by the
`?source` plugin (`vite-source-export.ts`) at a **fixed 80-col width** (`:126-128`). So the
"responsive code width" capability is dead, and code blocks won't reflow to the 30rem column —
in the narrow About pane an 80-col TypeScript line will horizontal-scroll (the `oklab2oklch`
source in `shots/C6-H1` already shows long lines pushing the `<pre>` width).
**Gestalt fix:** either delete the unused composable (it's ~120 LoC of dead weight) or, better,
re-wire it so the displayed code reflows to the actual narrow measure — long
`Math.atan2(...)/(2*Math.PI)` lines wrapping instead of scrolling reads far better at 30rem.
Decide one source of truth for code rendering (build-time `?source` vs runtime
`useCodeFormatting`), not two with one silently dead.

## What's genuinely good (keep)

- **The `?source` plugin is elegant** (`vite-source-export.ts`): real conversion source
  (`oklch2oklab`, etc.) is pulled from `src/units/color/conversions/*` at build time, Prettier-
  formatted, hljs-highlighted, zero runtime cost — and it stays in sync with the library because
  it *is* the library source. This is the strongest expression of the "math/code is the truth"
  motif. The fix in P2-CODE-FORMATTER-UNUSED should preserve this, just make it reflow.
- **Derived heading accents** (`useMarkdownColors`) tie the docs to the live color in both
  light/dark (`shots/C6-H1`/`H2`) — a smart, on-brand touch. P1-MATH-UNACCENTED just asks to
  extend it to the math.
- **Pane-header scroll-shrink motion** (`PaneHeader.vue` + glass-ui `--card-scroll`) is
  purposeful and reduced-motion-aware — the right kind of motion.
- **Content prose is well-written** — the color science in `oklch.md`/`lab.md`/`xyz.md` is
  accurate, well-sectioned, and genuinely educational. The problem is presentation, not substance.
