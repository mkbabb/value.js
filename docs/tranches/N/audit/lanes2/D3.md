## 1 — U24: PaletteCard → the first-class component (the work-order)

**Today** (`PaletteCard.vue:5-13`): one hardcoded class string —
`rounded-card border border-border bg-card overflow-hidden transition-shadow hover:shadow-card-hover`
— no variant axis, no material axis. The skeleton is a **separate file**
(`PaletteCardSkeleton.vue`, 36 LoC) that hand-mirrors the card's anatomy (strip / meta row /
swatches) with raw `animate-pulse` divs and drifts independently (it already has: `rounded-badge`
swatches vs the card's `rounded-badge`-via-`swatchClass`, different paddings). 7 consumers:
`PalettesPane.vue:51`, `BrowsePane.vue:52`, `ExtractPane.vue:44`, `ImagePaletteExtractor.vue:60`,
`GenerateControls.vue` (grep), `PaletteSavedTab.vue`, `PaletteBrowseTab.vue`.

**The spec — demo-owned** (the user said "FIRST-CLASS **in-repo**"; glass-ui gets NO PaletteCard —
KISS, `feedback_kiss_no_contrivance`; it gets consumed primitives only):

```
<PaletteCard
  :palette="p"
  material="glass" | "cartoon" | "flat"     ← the material/depth axis (default "glass")
  :skeleton="true|false"                    ← loading is a STATE of the card, not a sibling file
  layout="default" | "aside"                ← existing, keep
  …existing props/emits unchanged
/>
```

- **`glass`** (the default; browse/saved/dialog lists): translucent plate — `bg-card/55` +
  `backdrop-blur` (the picker pane's resting recipe U-cards probed: soft white inset ring), hairline
  = the new **`--card-edge`** token (U-cards §3 order: `color-mix(in oklab, var(--foreground) 12%,
  transparent)`) — **never `--border` navy**. Rest shadow: `shadow-cartoon-sm` (the chip rung — a
  small resting base). Hover: grow to `shadow-card` (8px) **with** `translate(-2px,-2px)` so the
  plate lifts toward the light and the shadow grows from a base instead of materializing
  (kills the none→10px lurch; `transition: box-shadow, transform` on the existing
  `transition-shadow` site `PaletteCard.vue:7`).
- **`cartoon`** (the hero card: extract result, generate result — ONE per pane): same glass plate +
  full `shadow-card` 8px AT REST (the pop-art statement the user wants — "glass cards WITH cartoon
  shadows"), hover = `shadow-card-hover` 10px + lift. The 85%-opacity slab is correct ONLY here,
  where the card is the pane's single protagonist — never on every list row (the U17 fighting was
  every row going 0→10px over a wash pane).
- **`flat`** (dense admin lists `AdminListItem`/dialog rows): current `bg-card` + `--card-edge`,
  no shadow. Honest density register.
- **`skeleton`**: renders the SAME shell (same material prop, same radii, same strip/meta/swatch
  geometry — zero drift possible) with glass-ui **`<Skeleton variant="shimmer">`** bones
  (exported from the root barrel — `glass-ui/src/index.ts:105`; 3 variants pulse/shimmer/breath,
  PRM-gated, compositor-friendly). Strip bones: 5 segments tinted
  `color-mix(in oklab, <cssColorOpaque> 14%, transparent)` stepped by index — the skeleton
  **previews color in the current color's key**, the antidote to u16's black void (§3).
  `PaletteCardSkeleton.vue` is then DELETED; `ExtractPane.vue:57,90` re-points.

**Acceptance:** zero `border-border` / `hover:shadow-card-hover` inside PaletteCard; the 7 consumers
pass a material; `PaletteCardSkeleton.vue` deleted; dark-mode element shot vs
`shots/D3-palettecard-rest-light.png` + u21 as the π before/after pair.

## 4 — Extract: the dominant-color hero + population threading (chartered) — and a live functional defect

### 4a — **The kC slider is a placebo** (functional, highest severity in this lane)

`ExtractControls` renders a chroma-weight slider (`ExtractControls.vue:42-53`); both shells hold the
ref and re-quantize on change (`ExtractPane.vue:110,178-181`, `ImagePaletteExtractor.vue:116,183-186`)
— but `runQuantize()` calls `quantizeFromFile(lastFile, colorCount)`
(`ExtractPane.vue:143-147`) and **`useImageQuantize.quantizeFromFile` forwards `{ k }` only**
(`useImageQuantize.ts:91-94`). `chromaWeight` never reaches the worker; the library option exists
and is documented (`src/quantize/types.ts:23`, default 0.5). Net behavior: moving kC debounces
300ms, shows the "Extracting..." spinner, and returns the **identical** palette — the UI lies.
**Order:** widen the composable signature to `quantizeFromFile(file, options: Partial<QuantizeOptions>)`
and thread `{ k, chromaWeight }` from both shells (one site after the §5 collapse). A vitest
asserting two kC values yield different centroids on a synthetic two-hue image pins it.

### 4b — Population + dominance discarded at the consumer boundary

`QuantizedColor.population` is computed and returned (`src/quantize/types.ts:41`); `dominantColor()`
is public API (`src/quantize/index.ts:176`). **Zero demo consumers of either** (grep: 0 hits outside
`src/`). Both shells map the worker result down to `{css, position}` (`ExtractPane.vue:119-121`,
`ImagePaletteExtractor.vue:127-129`) — the perceptual story the quantizer tells is thrown away.

**The hero spec (chartered "dominant-color hero + population threading end-to-end"):**
1. `PaletteColor` gains optional `population?: number` (`@lib/palette/types`) — demo-owned type,
   additive; the extract mapping carries it through.
2. **The dominant hero**: above the result card, the dominant color (derive from the already-returned
   `palette.value` — max population, chroma-tiebreak; NO second worker call, the library's
   `dominantColor` re-quantizes and is the wrong tool here) renders as the pane's protagonist — a
   large watercolor swatch (glass-ui `WatercolorDot`, already consumed) + its `oklch()` numerals on
   the display ramp (the U31 hero-numeral idiom, `tabular-nums` + ch-reservation so the card never
   resizes) + the population share as an AUDACIOUS stat: "**41%** of the image". One colorful pop,
   exactly the house language.
3. **Population-proportional strip**: `PaletteColorStrip` gains optional `:weights` (population-
   normalized, floor 8% so no segment vanishes — today hardcoded equal at
   `PaletteColorStrip.vue:21-22`). The extract card's strip becomes a truthful dominance bar; saved
   cards (no population) keep equal widths. The k-slider gradient (`ExtractPane.vue:134-141`) reuses
   the same weights so the slider previews dominance too.
4. Swatch hover popover (`PaletteCardSwatches` → `SwatchHoverMenu`) surfaces per-color population
   when present ("~12% · oklch(…)").

### 4c — Empty + processing registers
