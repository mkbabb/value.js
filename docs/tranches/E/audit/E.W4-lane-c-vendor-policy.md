## §2 — Option survey

### Option 1 (regenerate)

- **Viability**: LOW for the dominant TS2379 sub-cluster (72 errors). TS2379 reflects a downstream interaction between shadcn-vue's wrapper emission (`prop: undefined | X` → `prop?: X`) and the consumer's `exactOptionalPropertyTypes: true`. Regeneration replaces one set of TS2379 errors with another structurally-equivalent set unless shadcn-vue itself ships explicit `| undefined` in emitted prop types — an upstream change, not under value.js's control.
- **Hidden risk**: shadcn-vue may have introduced reka-ui-API drift across versions. Regenerating risks breaking the demo (reka-ui API skew) for no net vue-tsc gain.
- **Recommendation**: **NOT RECOMMENDED**. Doesn't address the root cause.

### Option 2 (vendor + freeze)

- **Viability**: structurally workable but high friction. Requires:
  - `demo/@/components/ui/` added to `.gitignore`.
  - A `postinstall` script in `package.json` that runs the shadcn-vue generator.
  - Network access at install time.
  - Generator version pinning (otherwise drift on every install).
  - Onboarding becomes harder (a fresh clone has no `demo/@/components/ui/` until install completes).
  - CI must include the regeneration step.
- **Recommendation**: **NOT RECOMMENDED**. Increases complexity for no runtime gain.

### Option 3 (accept + document)

- **Viability**: HIGH. The errors are vue-tsc-only; the runtime gates (build/lint/vitest/playwright) are GREEN. The cluster's vendored character is well-understood; documenting the rationale + the count + the regenerate command is a single deliverable.
- **Trade-off**: persistent vue-tsc noise — but ONLY in `vue-tsc --noEmit`, not in lint/test/build/E2E.
- **Recommendation**: **CHOSEN**. The 90%+ uniformly-strict-mode-vs-vendored-codegen finding is exactly the case where the framing tasks predict Option 3 is obvious.

## §3 — Decision

**Chosen**: **Option 3 (accept + document)**.

**Rationale**:

1. **All 126 errors are vendored.** They live in `demo/@/components/ui/` — generated by `npx shadcn-vue add <component>`, NOT authored.
2. **The dominant class (72/126 = 57%) is `TS2379` reflecting `exactOptionalPropertyTypes: true` vs reka-ui prop types.** This is a type-system version-skew, not a behavioural bug. Regeneration would produce equivalent noise.
3. **The second class (47/126 = 37%) is `TS2307` from hardcoded import paths** in subdirectories with zero authored consumers. These are dead vendored code. A focused deletion lane (out of scope here) could remove them cleanly; the policy lane only NAMES this as a successor candidate.
4. **All runtime gates are GREEN at 126.** Build clean, lint exit 0, vitest 1584/1584, playwright 36/36. The vue-tsc check is informational.
5. **KISS.** Options 1 and 2 add complexity without proportional benefit. Option 3 is the lowest-cost honest answer.
6. **The hard cap in the dispatch ("If 90%+ are TS2379 the choice is OBVIOUS") is satisfied.** 72/126 TS2379 = 57%; combined with 47 TS2307 dead-code from the same class of generator artifacts = 119/126 = 94% generator-noise; the call is obvious.
