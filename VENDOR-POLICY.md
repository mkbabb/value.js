### Historical context (E.W4 Lane C — superseded by deletion)

The 126 vue-tsc errors in `demo/@/components/ui/` (post-E.W3) decomposed as:

| TS code  | Count | Class                                                    |
|----------|-------|----------------------------------------------------------|
| TS2379   |   72  | `exactOptionalPropertyTypes: true` vs reka-ui prop types |
| TS2307   |   47  | Hardcoded `demo/@/components/ui/*` import paths          |
| TS2375   |    2  | `exactOptionalPropertyTypes: true` Zod `Shape` assignment|
| TS2532   |    2  | "Object is possibly 'undefined'" (DonutChart, ChartLegend) |
| TS2722   |    1  | "Cannot invoke an object which is possibly 'undefined'"  |
| TS2538   |    1  | `undefined` index type (AutoFormField)                   |
| TS2345   |    1  | Element assignability (ChartSingleTooltip)               |

**All 126 errors live exclusively inside `demo/@/components/ui/`.** No authored code outside the vendored cluster produces vue-tsc errors at HEAD.

### Why these errors exist

1. **TS2379 (72)** — shadcn-vue's generated `<script setup>` wrappers pass `prop: X | undefined` to a reka-ui target typed `prop?: X` (no explicit `| undefined`). The consumer's `exactOptionalPropertyTypes: true` strict-mode flag flags the mismatch. This is a downstream type-system version-skew, not a behavioural bug.
2. **TS2307 (47)** — the shadcn-vue generator emits import paths like `from 'demo/@/components/ui/form'` (literal repo-rooted path) instead of the consumer's alias (`@components/ui/form`). The consumer's vite config has no matching alias, so vue-tsc cannot resolve the module. **These 47 errors are concentrated in `auto-form/`, `chart/*`, `calendar/`, `carousel/`, `pagination/`, `range-calendar/`, `v-calendar/`, `form/` (one self-reference)** — none of which have authored consumers in the demo (verified via corpus-grep at Lane C open).
3. **TS2375/TS2532/TS2722/TS2538/TS2345 (7)** — small bundle of secondary strict-mode and Zod-typing nits within the same cluster.

### Why we accept

1. **The errors are in generated code, not authored code.** The wrappers were generated by `shadcn-vue add <component>` from official shadcn-vue templates. We don't author them; they're vendored.
2. **The errors don't affect runtime.** They surface only in `vue-tsc --noEmit`. The demo renders; the build is clean; the smoke specs pass.
3. **Build + lint + vitest + playwright are all GREEN.** Only vue-tsc surfaces these errors; the vue-tsc check is INFORMATIONAL, not blocking.
4. **Option 1 (regenerate) does not address the root cause.** The TS2379 cluster reflects shadcn-vue's generator vs `exactOptionalPropertyTypes: true` — that interaction persists across generator versions until shadcn-vue ships explicit `| undefined` in its emitted wrappers. A regeneration cycle replaces one set of equivalent-shape TS2379 noise with another.
5. **Option 2 (vendor + freeze in `.gitignore` + postinstall regen) shifts the burden to install-time** and introduces fragility (network access at install, generator version pinning, drift on local modifications). Onboarding becomes harder; CI gains a generation step. The accepted-noise posture is structurally simpler.
6. **Dead-code cluster is BOUNDED.** The 47 TS2307 errors are all in subdirectories with zero authored consumers (`auto-form/`, `chart/*`, `v-calendar/`, `range-calendar/`, `pagination/`, `calendar/`, `carousel/`). They can be deleted in a focused follow-up lane WITHOUT affecting the demo. That deletion is out-of-scope for the policy lane, but is named as a candidate successor.

### How we manage

- **vue-tsc CI gate baseline: 0** (strict zero-error, hardened at F.W3 Lane C). The CI step in `.github/workflows/node.js.yml` fails the build if `COUNT > 0`. Pre-F.W1 the baseline was 126 (E.W4 — accept + document); F.W1 Lane C deleted 29 zero-consumer subdirs and dropped the count to 0; F.W3 Lane C tightened the gate to match.
- If the cluster ever regenerates (manual `npx shadcn-vue add ...`), the error count may shift; update this doc + the CI gate together.
- Lint and the runtime gates remain authoritative for blocking decisions; vue-tsc is now blocking too at the zero-error baseline.
